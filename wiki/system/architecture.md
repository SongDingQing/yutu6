# 玉兔6 系统架构(提炼)

> 提炼自 `shared/reference/多智能体架构设计.md`。

## 一句话
每个 agent 是一段独立上下文,彼此不共享记忆,甚至来自不同厂商;它们唯一共享的是**文件系统**。**文件 = 共享大脑**。

## 四层组织
- **老板(你)**:只读 `board/`,下方向、看趋势、拍板。
- **总管 Orchestrator**:把方向拆成项目派给主管,把进展摘成趋势回报老板。强模型。
- **主管(一项目一个)**:只看自己项目,把项目拆任务、审产物、上报进展。强模型。
- **员工**:只看自己任务文件 + 指定输入;缺跨域信息写进 `questions.md` 问主管。快/省模型。

关键:**没有谁看到全局**。信息向上走逐层压缩(只传摘要+路径),向下/横向按需拉取。

## 三条上下文规则
1. 读取范围按层封顶(强制 need-to-know,白名单路径)。
2. 过界就压缩(只传摘要和指针,不传正文)。
3. 无状态 agent + 有状态文件(醒来→读文件→干活→写结果→退出);配 git,一任务一提交;单写主原则。

## 信封协议
`task.md`(派单:目标/边界/输入/验收/截止)· `result.md`(汇报:做了什么/产物路径/待解/风险)· `questions.md`(按需提问)。

## 生态(v2)
本机工作区 = 执行核心;可选远端服务 = 中继;网页或移动客户端 = 瘦控制面。同步三原则:命令式、append-only 事件日志 + cursor 增量、单调 version 排序。

## 三大省力/治理机制
- **硬化(§16)**:把"一直绷着的肌肉"变成骨头/软骨/肌肉记忆 —— 越用越硬、越省、越快。质量运营师执行。
- **监管人员(§17)**:反复失败/重大漏洞时做 meta 复盘,定误区、固化防复发规则。见 `shared/reference/`。
- **声明式控制平面(§18)**:结构已知的流程用 YAML+Jinja2 写死,零 token;只有结构未知才交回 LLM。见 `shared/routing/`。
