# 玉兔6 · 人力资源部(HR部门)设计方案

> 老板拍板:① 创建把关用**分级审批**;② **先出方案再建**。本方案供过目,认可后落地。
> 秘书拟 · 2026-06-21

---

## 一、为什么要这个部门

现在"加一个智能体"是**手工活**:手写 `agent.json` + `prompt.md`、改 `config.json` 的 `roleRouting`、建工位——没人把关归属/权限/成本,也没有查重和验证,容易乱建、权限给太大、或重复造轮子。

HR 部门把这件事**标准化、可审批、权限可控**:从"随手散建"收口成"一条有把关的流水线"。

## 二、定位

**横向职能部门**,不归任何单一项目,服务全系统。职责:智能体的**招聘(创建)、入职(配置+验证)、归档(停用)**,以及花名册维护。

## 三、编制(先两人)

| 角色 | 干什么 | 模型 | 理由 |
|---|---|---|---|
| **HR 主管** | 把关人:审需求、定四要素规格、做分级判断、跑入职校验、低风险拍板 / 高风险转老板 | `claude` | 要判断、要严谨 |
| **HR 专员** | 执行人:按规格填 `agent.json`+`prompt.md`、注册 `roleRouting`、建工位、跑通验证 | `zhipu-glm`(GLM) | 执行类,用免费/便宜额度省成本 |

## 四、四要素(老板定的把关维度,缺一不可)

每个新智能体创建前必须明确这四项,缺任一项**拒绝创建**:

1. **归属** — 属于哪个项目/部门(控制台 / Simulaid / HR / 横向)。决定它进哪个队列、归谁管。
2. **能力** — 一句话职责 + 用什么工具。决定 `role`、`prompt`、`tools`。
3. **额度(模型)** — 用哪个 runner。按能力选档:写码/重活→`codex`;判断/严谨→`claude`;普通/批量→`GLM`(免费优先);截图→`peekaboo`;高危特权→`codex-privileged`(需审批)。**默认优先免费/便宜模型**(老板成本观)。
4. **文件权限** — `read_paths`(能读) + `writes`(能写)。**最小化**:只给必要路径,默认不给跨项目/核心文件写权。

## 五、创建流程(分级审批)

1. **提需求** — 老板或项目说"要个能做 X 的智能体"(经秘书转 HR)。
2. **拟规格** — HR 主管把需求拆成四要素。
3. **查重** — 先查现有 agent,能复用/扩展的**不新建**(防泛滥)。
4. **分级判断**:
   - 🟢 **低风险**(只读,或只写自己目录 + 用免费/便宜模型)→ **HR 直接建**,事后告知老板。
   - 🔴 **高风险**(要写跨项目/核心文件、或高权限、或用付费模型 codex/claude、或特权 runner)→ HR 出**一页规格卡** → 老板**一键审批** → 才建。
5. **落地**(HR 专员)— 填 `agent.json`(标准模板)+ `prompt.md` + 注册 `config.json` 的 `roleRouting` + 建工位 + 更新花名册。
6. **入职校验** — 建完跑一个 smoke 任务(能 claim → 能跑 → 能 finish),**通过才算入职**;不通过回炉,不假完成。
7. **沉淀** — 归属/能力/技术映射交记忆官写入 `memory/entities.md`。

## 六、agent 标准模板

新 agent 统一按现有 schema 填(字段都在用):

```
id, name, schema_version, role, runner, tier, context_mode,
[queueAgent / persistent_worker], read_paths, writes, [tools],
triggers, io, decoupling, validation, prompt
```

\+ 配套 `prompt.md`(职责 / 边界 / 红线)。HR 维护一份模板,填空即可,几分钟建一个。

## 七、严谨怎么保证(不出乱子)

- **四要素必填**:缺任一项拒绝创建。
- **权限最小化**:白名单路径,不给通配 / 全盘写;高权限必审批。
- **查重防泛滥**:先复用,后新建。
- **分级审批**:高风险老板把关,低风险不打扰。
- **入职 smoke 校验**:建完即验,杜绝"建了跑不起来"。
- **可回滚**:每次创建有记录,可一键停用 / 归档(不硬删)。
- **花名册单一真相**:所有 agent 在 INDEX 花名册登记,杜绝影子 agent。
- **红线**:密钥 / token 绝不写入 agent 配置;Starlaid 排除;特权 runner / 全盘写权一律老板审批。

## 八、高效怎么保证

- 标准模板 + 自动填充,低风险几分钟建好、不打断老板。
- 复用优先,不重复造轮子。
- HR 专员用免费 GLM 执行,省额度。

## 九、接口变更

"新增智能体"入口从**手工散建**收口到 **HR 部门**:老板 / 项目提需求 → 秘书转 HR 主管 → 走上面流程。其它人不再直接手搓 agent。

## 十、HR 的知识职责(建文件夹 + 知识共享 + 知识定位)

HR 创建新 agent 时,除了配置本身,还负责把它**接入知识网络**:

1. **建归属文件夹** — 按部门给新 agent 配工作目录,并接入**部门共享知识区**(同部门 agent 共用一块知识)。
2. **配读权** — `read_paths` 至少含:自己的目录 + 部门共享知识区 + 全局知识入口(`memory/`、`knowledge/`)。
3. **写知识定位** — 在新 agent 的 `prompt.md` 里明确告诉它:你的核心知识在哪、部门共享在哪、深度检索用 `knowledge/query.py`、长期记忆由记忆官注入。让它一上岗就知道去哪找数据。
4. **登记数据地图** — 把新 agent 的归属和知识区更新到全局 `DATA-MAP`(见下节)。

## 十一、文件架构 + 知识库体系优化(让所有 agent 都找得到数据)

现状:数据散在 `memory/`、`knowledge/`、`board/`、`wiki/`、`shared/reference/`、`projects/*/` 多处,agent 靠各自的 `read_paths` 白名单手动找——新 agent 很容易"不知道某块知识在哪"。优化分三步:

**① 明确四层知识分工(各放什么、怎么取)**

| 层 | 放什么 | 怎么取 |
|---|---|---|
| `memory/` | 提炼后的长期记忆(偏好/决策/经验/实体) | 干活前**记忆官主动注入** |
| `knowledge/` | 原文 RAG 库(corpus + kb.sqlite) | 深度问题用 `query.py` **按需检索** |
| `board/` | 运营状态(方向/进展/工单) | **实时直读** |
| 项目/部门区 | 项目数据 + **部门共享知识** | 归属内 agent **直读** |

**② 按归属分区(解决"同部门共享")**

新增**部门/项目共享知识区**(如 `projects/<项目>/knowledge/` 或 `shared/knowledge/<部门>/`):同部门 agent 都挂这块,沉淀和复用本部门知识,不用各记各的。

**③ 一张数据地图(解决"都找得到")**

建 `shared/DATA-MAP.md` 作为**单一索引**:一眼看清"什么数据在哪、用什么方式取(注入/检索/直读)、谁该读什么"。所有 agent 的 prompt 引用它;拿不准去哪找,先查这张图。HR 每建一个 agent 就更新它。

**闭环**:干活前记忆官注入 `memory/` → 要深挖调 `knowledge/query.py` → 不知去哪找查 `DATA-MAP` → 新知识沉淀回部门区 + memory。这样 agent 越用越懂、且都找得到。

## 十二、落地清单(认可后执行)

1. 建 `shared/agents/hr-manager/` 和 `shared/agents/hr-specialist/`(各含 `agent.json` + `prompt.md`)。
2. `config.json` 加两个 role(`hr_manager`→claude、`hr_specialist`→zhipu-glm)+ 队列 + 工位。
3. 写 HR 的创建工具/脚本:四要素校验 → 查重 → 分级 → 填模板 → 注册 → smoke 校验 → 花名册。
4. **知识体系**:建部门共享知识区 + 写 `shared/DATA-MAP.md` 数据地图;HR 创建 agent 时配齐归属文件夹、读权、prompt 知识定位、更新数据地图。
5. 办公室视图加"人力资源部"片区 + 两个工位。
6. smoke 验证整条链路跑通(含新 agent 能按数据地图找到数据),记忆官沉淀。
