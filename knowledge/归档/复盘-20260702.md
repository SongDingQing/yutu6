## 2. 根因分析

### P-1 · 通道失败与评审意见语义混淆

现象上,董事会链路看到多个“董事失败”,老板会理解为董事否决;但原始原因是 runner 没有产出有效意见,例如鉴权失败、上游繁忙或额度不足。维修票确认这类情况应显示为“缺席/通道失败”,不能参与否决、风险汇总或修订轮次判断。

根因:评审系统把 runner 调用失败的传输层/供应商层状态直接混入业务评审语义,缺少 `absent` 这一中间状态。避免方法是把“无有效意见”独立建模,风险统计只消费真实 opinion。维修已在 board-review/server/workspace/test 层做兼容,本复盘不再改代码。

### P-2 · 证据快照把完整相对路径无界编码为 basename

现象上,实现结果已经产出 evidence-only 收口,却在归档 changed_files 时崩溃。维修票指出 `snapshotFiles()` 将完整相对路径替换为单个文件名,没有 byte 上限,也没有跳过已生成的 loop-engineering/review-loop-fixture 快照目录。下一轮把已编码路径再次编码,形成路径膨胀回路。

根因:证据系统把“可追溯”误实现为“完整 rel 全量折进文件名”,而不是 hash + 短名 + receipt 映射;同时没有 generated artifact 二次快照排除。避免方法:所有快照/备份/证据系统必须有 basename byte 上限、hash 命名和 generated artifact skip 回归。

### P-3 · 自由文本 hard regression gate 把政策/元数据当执行任务

现象上,两张 `queue_merge_integrity` 工单都不是队列合并实现失败:第一张是角色边界/空转队列归档方案,第二张是 UI 自省 current 收口里要求按 `rootTaskId/queueId` 去重核对。二者都被“队列/queue + 合并/去重”等自然语言窗口命中,从而错误要求 `queue-organizer` 与 `ceo-queue-control` 硬回归。

根因:done-gate 的 hard regression 触发源仍依赖混有 brief、acceptance、链路元数据、证据路径的自由文本。局部正则补丁能止血单个 false-positive,但同族语料会换形态复发。避免方法:hard regression 触发应优先使用结构化 domain/explicit hardening label/changed_files owner;若必须保留文本扫描,要剥离 `queueId/rootQueueId/queueAgent` 等链路元数据,并维护真实 false-positive 语料库。

### P-4 · 独立时序红灯未被当前维修闭环覆盖

现象上,done-gate 相关聚焦测试已过,但全量测试仍暴露 `ceo-serial-lock.test.js` project-route downstream wake 失败。工单明确它与 done-gate/queue_merge_integrity 修复无调用路径交集,已另列 todo。

根因目前未闭环:可能是测试时序 flake、事件唤醒机制回归或队列状态污染。监管不能替维修员修复,只把它保留在未闭环清单,要求后续专门核查。

### P-5 · 软信号缺少持锁者/心跳归因,容易被 terminal failure 掩盖

现象上,本窗口终态失败只有 3 个,但 all_blocked 与 watchdog restart 都很高。`resource.scheduler.all_blocked` 事件样本缺少可直接复盘的持锁者持续时长/top-N;watchdog restart 只有 `worker-heartbeat-stale <agent>:?` 粗原因,不足以判断是 daily 惊群、长任务无心跳、旧 worker 还是队列阻塞。

根因是可观测性仍不够结构化:软事故发生了,但事件没有足够的锁持有者/冲突者/心跳来源字段。避免方法:daily 复盘必须把 all_blocked 与 watchdog restart 分账列出;超过阈值时,缺少 top holder/top blocked queue 本身就算未闭环。

## 4. 防复发规则(可执行、可验证)

### NR22 · hard regression gate 不能只靠自由文本命中

规则:公共 done-gate 的硬回归触发,若影响 queue/engine/routing 等共享域,必须优先绑定结构化字段:task domain、explicit hardening label、changed_files owner 或 action type。自由文本扫描只能作兜底,且必须剥离 queueId/rootQueueId/queueAgent/taskId 等链路元数据。验证:同一 hard regression false-positive 在 24h 内出现两种语料形态时,必须新增“false-positive corpus + 正例保留”成对测试,不能只改一个正则。

### NR23 · 证据/快照系统必须有 filename byte cap + generated artifact skip

规则:任何把路径映射到文件名的证据、快照、备份、fixture 工具,都必须使用 hash + 短 basename,按 UTF-8 bytes 限长,并默认跳过自身生成的 artifact 目录。验证:测试覆盖超长相对路径、非 ASCII 路径、已生成 fixture 的二次快照;目标 basename 不超过平台安全阈值。

### NR24 · runner 缺席不是评审意见

规则:评审/董事/复审类 runner 的 auth、quota、timeout、429、channel unavailable、provider busy 只能产生 `absent/channel_failed` 语义,不得计入否决、风险、偏差或修订轮次。验证:董事 runner 返回通道失败时,board summary 单列缺席数,hard block/risk count 不增加。

### NR25 · all_blocked/watchdog restart 是软事故,不被 task.failed 数量抵消

规则:每日复盘固定统计 `resource.scheduler.all_blocked`、`watchdog.restart.done`、`runner.failover`、`node.absent`。`all_blocked > 100/日` 或 `watchdog.restart.done >= 3/日` 必须进问题清单;若事件缺持锁者/冲突者/心跳来源字段,该观测缺口本身列未闭环。验证:复盘归档必须有 counts、样本行号、未闭环判断。
