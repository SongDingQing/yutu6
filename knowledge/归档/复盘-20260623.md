## 2. 根因分析

**主根因:重启韧性两端都不安全 + 失败可观测性缺口 + 工单闭环缺口。**

1. **重启端口未释放(P1,新根因)**:watchdog 重启逻辑只"杀旧 pid + 起新进程",**不保证旧进程已优雅退出并释放监听端口**。旧进程残留持有 41218 端口 → 新实例 `bind` EADDRINUSE → 重启失败 → cooldown 节流 → 自我级联近 3 小时。这是 entities「[[重启即清扫,清扫即杀活]]」的**反面同族**:不是"杀活",而是"杀不净/重启不上"。节流(throttle)在这里是**反作用**——它把"重启失败"当成"重启太频繁"压住,延长了宕机。

2. **engine-runner exit-1 零可观测性(P2)**:非 0 退出路径**不 flush 任何 `node.fail` reason、不落 run 目录、不写 worker log**,于是 `oldjump` 这种残留队列的真实根因从工单完全不可定位,维修员无从下手,工单只能悬挂。

3. **工单闭环缺口 + 压制≠解决(P2/P3)**:今日 15 张工单仅 **6 张 done**,**9 张悬挂 todo**(6× oldjump + 3× supervisor)。`dedupe-cooldown` 命中 **25** 次说明同签名高频复发,但冷却去重只是**静默压制**,没有"复发到阈值就升级为根因/人工"的出口——高频残留故障被悄悄吃掉,看板上看不出严重性。

4. **资源调度(P4)**:`all_blocked` 风暴反映故障窗口里 slot/资源锁被大面积、长时间持有(与 lock-leak、心跳超时同时段),具体哪些锁未深挖。

## 3. 防复发规则(可执行、可验证)

- **R1 · 重启端口安全门**:watchdog/console 重启**起新进程前**必须确认旧进程端口已释放——`kill` 后轮询端口直到 free(超时则 SIGKILL 升级),再 `bind`;连续失败 N 次应**升级告警主人**而非无脑 cooldown 空转。验证:重启集成测试模拟"旧进程占端口",断言新实例不抛 EADDRINUSE、且 N 次失败后触发升级通知。
- **R2 · engine-runner exit-1 必落根因**:engine-runner 任何非 0 退出前**必须**向事件流 flush 一条带 `exit_code` + stderr 尾部的 `node.fail`,并在 run 目录落至少一行;禁止"退出码 1;未写明原因"的盲洞。验证:制造 exit-1,断言 events 有非空 reason 且 run 目录非空。
- **R3 · 残留队列与冷却升级出口**:固定名 queueId 反复 exit-1 且**零产物** N 次后,自动 `queue.paused` 并升级告警,停止无脑重排刷工单;`dedupe-cooldown` 命中数达阈值时,把该签名**升级为"需人工/根因"**而非继续静默压制。验证:同签名失败计数达阈值 → 触发 `queue.paused` + 升级通知。
- **R4 · 工单闭环看板 + infra 分流**:每日复盘统计 `status: todo` 悬挂工单,超 X 小时未结案者升级通知主人;基础设施类(重启/SIGTERM/端口占用)按昨日建议的 `infra_restart` 分类**只重入队、不开重复工单**,与业务 `node_failed` 分流。验证:复盘脚本输出"悬挂工单数 + 最老悬挂时长",infra 类不再开自动工单。
- **R5 · 延续观察**:no-progress 判死与心跳联合门、running 恢复字段白名单(2026-06-22 已修部分)本轮窗口仍有 480s/508s 判死 2 例,继续观察是否回归。
