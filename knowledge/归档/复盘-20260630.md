## 2. 根因分析

### P-1 · all_blocked 风暴复燃:不是 05:00 惊群,而是批量任务 + 锁面过宽

现象上,2040 次 all_blocked 并不集中在 05:00 daily 点火,而是集中在北京时间 09:45-14:18 的 CEO/project-route 批量任务窗口。样本事件显示单个 all_blocked 往往包含 1-8 个被阻塞候选,多数都要求写 `queue-state` 与 `brief-status`,不少还额外写 `config`、`agents`、`memory`、`board`、`frontend-public`、`engine`。这使多个本应可以串行推进或局部并发的任务,在调度层被判「全部候选都被资源锁冲突阻塞」。

根因:锁域声明过粗,`queue-state/brief-status` 成为几乎所有任务的公共写锁;再叠加 CEO 在 09:45 附近密集入队多张 project-route 任务,调度器看到的是大量候选都要写同一组全局域,于是频繁 all_blocked。事件里 `blocked.reason=resource-lock-conflict`,但缺当前持锁者 top-N/持续时长汇总,导致只能定位「被阻塞的候选和写域」,不能稳定定位「谁长期持锁」。这是 06-24 NR5「all_blocked 锁持有者审计」未彻底闭合的复发版本。

避免方法:daily 复盘必须把 all_blocked 当一等故障指标,不能只看 `task.failed`;当 all_blocked 超阈值(例如 >100/日)时,输出 top queueId/write-domain/持锁者持续时长。技术落地方向交主管/维修员:把 `brief-status` 从大多数写任务的全局写锁里拆细或降为读/追加语义;对 project-route 批量入队加节流;风暴达阈值时生成 top-N 持锁者快照。监管不在本任务中改锁逻辑。

### P-2/P-5 · done-gate 正确拦截,但上游合同与错误分类仍不够稳

两个 task.failed 分别来自「结构化输出被 JSON.parse 判废后误报缺逻辑链」与「验收表证据未指回对应 decisions.md 行」。维修票已确认:worker 实际内容可能完整,但结构化 JSON 尾部多余括号导致 extractJson 严格解析失败;修复已加配平恢复首段 JSON。另一个失败则说明设计对照行/模板种子行仍会诱导 agent 用不可核证据或未指回 decisions 锚点。

根因不是 done-gate 过严;门禁在正确拒绝不可核/不可解析的交付。真正问题在上游输出合同与错误分类:模型输出轻微尾噪时不能整段判废;而即便完全不可恢复,失败原因也应显示「JSON 解析失败」,不能下游改写成「缺少逻辑链」。同理,验收表模板若把会议纪要/超长设计对照行当可核规格,执行 agent 很容易原样抄写而被门禁正确打回。

避免方法:已修的配平恢复要保持测试;残余规则是「结构化输出解析失败必须保留 parse_error 类原因」。验收模板侧继续执行 NR9:设计对照只能引用可核规格/决策结论行,且证据必须指回同一 `decisions.md:行号` 与真实文件/测试证据。

### P-3/P-6 · runner failover 从单点修复进入全角色审计阶段

本窗口 runner.failover 41 次,其中 http_429 10 次、runner_error 31 次;此外 Claude 401 影响 ui_optimizer、memory_officer、repair-lead 等链路。维修已把 ui_optimizer rollbackRunner 改为 codex,并给 memory_officer 增加 subscription.codex 末级兜底。zhipu「访问量过大」类错误则显示中文 throttle 识别与退避仍不足,低优先级定时任务会在短时间内空耗重试。

根因:声明式路由的 fallback 过去按「已暴露角色」单点补洞,没有对所有自动后处理/写盘角色做系统审计;部分角色的主路和 fallback 仍在同一认证/供应通道。另一个根因是限流/拥塞类错误未统一进入 throttle/failover/退避策略。

避免方法:对承担 done 后处理、记忆沉淀、UI 自动检查、维修后续处理的 queueAgent 做 Claude-only / same-provider-only 审计;每个允许自动写盘的角色至少有一个已验证可用的跨 runner 兜底,或显式记录为何保持单通道。中文 throttle 词典与指数退避属于后续维修/主管工作,本复盘不改。

### P-4 · 维修票 status 不能替代未闭环清单

7 张当日维修票中 6 张写回 done,但仍有 `auto-20260629044723-003d767d65211226.md` 保持 todo。即使 done 票也有残余建议:结构化解析错误原因仍需区分,Claude-only 角色还要全量审计,zhipu throttle 需要退避策略。

根因:当前维修闭环仍偏「单票处置」,每日复盘如果只看 done 数会漏掉「todo 悬挂」和「done 票里的残余建议」。这与既有「假闭环」家族同源:状态字段不能取代复跑证据、残余建议和未闭环清单。

避免方法:每日复盘固定列出当日 repair-tickets 的 todo 数、最老 todo、done 票残余建议;todo 票不由 governance 代修,但必须进入未闭环清单并在次日复核是否仍悬挂。
