## 2. 根因分析(误区诊断 + 根因)

**Meta 误区(系统性):规则只进了 memory,没进代码/skill/检查脚本——缺少"规则→落地→验证"的闭环 owner。**
昨日复盘产出 R1/R3/R4 并写入 memory,但今日核查:R1 代码未落地(`server.js:3698` 原样)、R3 未落地(oldjump 继续刷工单)、R4 未落地(infra 类仍开重复自动工单、30 张 todo 堆积)。**"写进 memory 的防复发规则"被默认当成"已解决",但它只是建议,没有执行者把它变成代码/检查门,于是同族故障按量级复发,部分还恶化(all_blocked ×3.3)。** 这正是误区库应命名的新条目:**「沉淀≠落地」/「复盘写了规则就当闭环」**。

分项根因:
1. **重启韧性两端不安全(P1)**:重启只"杀旧 pid + 起新进程",不保证端口释放、连续失败只 cooldown 不升级。R1 未落地。
2. **running 文件与 engine 生命周期竞态(P2)**:运行中 running 队列文件可被并发清扫移除,导致活 engine 被 `queue-running-missing` 连带终止;固定名 `oldjump` 残留队列无自动 pause 出口,反复重排。
3. **资源锁持有审计缺失(P3)**:all_blocked 风暴只有计数无 reason 细分,无法定位长期持锁者;故障窗口放大锁竞争。
4. **任务级超时契约未在运行态生效 + 长运行 runner 挂起(P4)**:codex/codex-privileged 600/900s 挂起喂 no-progress 判死;长驻 worker 未加载昨日的任务级门槛补丁。
5. **路由无就地 failover(P5)**:prefer 首选通道不可用时不自动降级,直接 exit 1 + retry 空耗。
6. **长驻 worker 无滚动重载闭环(P6,元根因)**:落盘补丁不重载即不生效。

## 3. 防复发规则(可执行、可验证)

- **NR1 · 复盘规则落地追踪门(最高优先,针对 Meta 误区)**:每条防复发规则产出时必须绑定一个 owner 维修工单 + "落地验证"出口;次日复盘**强制核查上一日每条规则的落地状态**(代码/测试/配置 diff 为证),复发即判"未落地"。把"规则落地率"作为复盘固定指标。验证:本复盘已逐条核查 R1/R3/R4/R5 落地情况(均未落地或仅部分),并落入第 4 节缺口清单。
- **NR2 · console 重启端口安全门(R1 升级为带证据的维修工单)**:重启起新进程前必须 `kill` 旧 pid → 轮询端口直至 free(超时 SIGKILL 升级)→ 再 `bind`;连续失败 N 次升级告警主人而非无脑 cooldown。证据指向 `server.js:3698`。验证:模拟"旧进程占端口"断言新实例不抛 EADDRINUSE 且 N 次失败触发升级。
- **NR3 · running 文件/engine 生命周期防竞态 + 残留队列自动 pause(R3 升级)**:running 队列文件移除前必须确认无对应 active engine;固定名 queueId 反复失败且零产物 N 次后自动 `queue.paused` + 升级,停止刷工单。验证:并发"移除 running 文件 + active engine"断言 engine 不被误杀;同签名失败计数达阈值触发 pause。
- **NR4 · 模型路由通道健康 + 就地 failover**:runner 收到"No available channel"等通道级错误时,自动降级到该角色 `prefer` 列表下一项,而非 exit 1 + retry;路由层对 prefer 首项做轻量健康预检。验证:mock zhipu 返回 no-channel,断言请求落到 prefer 次选且任务不失败。
- **NR5 · all_blocked 锁持有者审计**:`resource.scheduler.all_blocked` 必须带 reason/当前持锁者快照;风暴达阈值时输出长期持锁 top-N。验证:风暴事件 reason 非空且能定位持锁队列/任务。
- **NR6 · 核心引擎/队列改动后长驻 worker 滚动重载(P6,提级)**:`repair-202606230045` 提优先级;把"补丁落盘后空闲窗口滚动 supersede/drain 重启 worker"做成标准维修动作,有 active running 不强杀、等空闲窗口安全替换。验证:改动核心模块后,空闲窗口确认 worker pid 更新、新逻辑生效。
- **NR7 · infra 类故障分流(R4 续)**:重启/SIGTERM/端口占用/队列文件缺失类只重入队、不开重复自动工单,与业务 `node_failed` 分流;复盘统计 todo 悬挂工单数 + 最老悬挂时长,超 X 小时升级。验证:infra 类不再产生 30 张同签名 todo 堆积。
