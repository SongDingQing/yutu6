### P4. 资源锁授予无公平性,锁等待超时被误称 worker crash

- 现象:ceo/orchestrator 任务等待 `engine/agents/queue-state/brief-status` 写锁 30 分钟后 timeout,事件流记录为 `worker-crash` 链路并自动开高优维修票。
- 根因:锁授予只看当前 writer/readers,waiter 文件不参与授予顺序;长持写锁被心跳续租时不会 stale,老等待者在每次释放瞬间可能输给后到竞争者,形成写者饥饿。标签层又把锁等待超时包装成 queue worker crashed,混淆真实进程崩溃与锁超时。
- 维修结果:维修主管已结案并确认系统性根因,但未改核心锁代码,等待主人拍板后再派维修员处理。
- 防复发规则:资源锁授予必须引入 FIFO/bounded-wait/aging,存在更早且资源集合冲突的 waiter 时后到者不得抢占;对超长写锁设软上限或拆细资源域;锁等待接近阈值前发 P1;`resource.lock.timeout` 与真实 worker crash 分类分账。
- 未闭环缺口:公平锁、超时预警、分类修正和回归测试均未落地。

### P5. project-route parent/downstream 终态非幂等,done 后仍可被 failed 覆盖

- 现象:4 个 CEO parent 任务在下游 supervisor `queue.completed ok=true` 后仍写出 parent `task.failed`,原因均为“下游 done gate 未通过”;其中 `88569a5b` 先写 `project.route.done` 与 `task.done`,12 秒后又写 `project.route.failed` 与 `task.failed`,并同时存在 done/failed 队列文件痕迹。
- 根因:project-route waiter 在 worker stale/re-spawn 或 fallback wake 后没有对 parent 终态做幂等 fence,也没有把“下游已 done 但全量测试命令红灯/退出码 null”与“下游业务失败”分开;done gate 使用全量 `node tests/run.js` 时,已知全局红灯会把下游完成转译成 parent 失败。
- 维修结果:无当天维修票;相关下游任务多已在 status-rollup 记录完成,但 parent fail 痕迹仍在事件/队列目录中。
- 防复发规则:parent 终态写入必须 compare-and-set,`done` 后不得再写 `failed`;done/failed/canceled 终态目录必须互斥,清理 stale tombstone;project-route waiter 每次 wake 后先重读 parent 和 downstream 最终状态;done gate 失败若来自已知全局红灯或命令退出码 null,应归为 gate/test-infra 缺口,不得覆盖已完成业务终态。
- 未闭环缺口:需要修 project-route 终态 fence、stale worker 写终态保护、done gate 全量红灯分账和旧 failed tombstone 清理。
