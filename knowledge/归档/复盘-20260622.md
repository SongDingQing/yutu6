## 2. 根因分析

**共同根因:重启/清扫把"还活着的引擎"当成"该清的死进程"杀掉。**
- 当日 5 次 watchdog 重启,原因为 `console-not-running`、`boardReviewControl hotfix reload`、`worker-heartbeat-stale (supervisor-控制台 / it_engineer / ceo)`。每次重启的 `cleanup-runtime` 步骤会清运行态 → 正在跑的 engine 收到 SIGTERM(P1)→ 记录的 enginePid 失效(P3)→ 续约心跳中断按超时判死(P2 的一部分)。
- **误区命名:`重启即清扫,清扫即杀活`** —— 把"重启控制台/清运行时"当成无副作用的安全操作,实际它对在途引擎是一次批量 kill。这与经验库已有三条([[孤儿清理误杀活引擎]]、detached 进程卡死重启杀不掉、反复治标重启)是**同一族系统性问题的新化身**:症状从"杀错"演化为"重启级联杀错 + 工单刷屏"。
- P2(480s 无进展)双重成因:① 部分是重启把引擎杀了后残留 running 项被超时回收;② 部分是长跑 codex 节点 8 分钟内确无 stdout 进展被判死——**无进展判死未与心跳续约联合判定**,心跳还新鲜也照杀。注:当日「误判阈值审查」已把 watchdog 的 heartbeat 与明确进展分开判定,但**两者尚未做"心跳新鲜则豁免无进展死"的联合门**。
- P1 工单刷屏根因:SIGTERM 由基础设施重启造成,**不是任务自身失败**,但失败分类一律走 `node_failed` → 耗尽重试 → 开维修工单。基础设施级中断与业务失败未区分。
- P4:root 任务在下游队列结果就绪前被读取/超时,属路由等待与下游可用性的时序竞态,单发、未复现,本轮仅记录观察。

## 4. 防复发规则(可执行、可验证)

1. **重启前优雅静默在途引擎,而非直接清杀。** `cleanup-runtime`/console 重启必须先尝试 drain/quiesce(给在途 engine 一个收尾窗口或重连),只 sweep **心跳确已失联**的;心跳新鲜的引擎不在重启清扫范围内。
   - 验收:构造"重启时存在心跳新鲜的 running engine"烟测,断言该 engine **未收到 SIGTERM**、未产生 `node.fail`。
2. **基础设施重启导致的 SIGTERM 不计入业务失败、不开维修工单。** 当 `node.fail` 的 reason 命中"被信号 SIGTERM / 运行清扫"且时间落在一次 watchdog/console 重启窗口内,标记 `retryKind=infra_restart`,**重新入队但不递增 maxRetry、不触发 repair.ticket**。
   - 验收:重启窗口内的 SIGTERM 失败 → 仅 `queue.retry(infra_restart)`,`repair.ticket.created`=0。
3. **"无进展判死"必须与心跳续约联合判定。** 480s 无 stdout 进展时,若 `engine_heartbeat_at` 仍在续约阈值内,延长宽限(或仅告警),只有**心跳也 stale** 才判死。
   - 验收:心跳新鲜但 8 分钟无 stdout 的长跑节点 **不被** `node_failed`。
4. **维修工单生命周期收口。** 自动工单需有终态:基础设施类(infra_restart)在确认源任务已重试成功后自动 resolved/closed;监管/质量运营每日 sweep `status: todo` 超 24h 的工单并归类(待修/可关闭/转人工)。
   - 验收:本轮 4 张悬挂工单本日内被归类;次日复盘 todo 工单数 < 本日。
5. **少重启、修心跳本身。** `worker-heartbeat-stale` 触发的重启是治标——优先排查 worker 为何停止续约(主循环退出/阻塞),让 codex/维修员改根因,而非靠 watchdog 反复重启控制台(延续 [[反复治标重启消耗老板]] 教训)。
   - 验收:`watchdog.restart` 因 heartbeat-stale 的次数周环比下降;heartbeat 续约连续性指标可观测。
