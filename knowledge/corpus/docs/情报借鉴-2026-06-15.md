## 1. 控制平面 · 消息投递队列(Pydantic AI Harness)

- **来源**:`github.com/pydantic/pydantic-ai-harness/issues/89`(未核实)
- **要点**:命令队列拆两种语义 ——
  - **steering**(打断式):工具执行后、下次 LLM 调用前送达,用于"中途转向";
  - **follow-up**(排队式):当前任务干完才送达,用于"顺序喂任务";
  - 主循环两个固定消费点:每轮工具执行后查 steering;无活可干时查 follow-up;
  - 投递支持 **one-at-a-time / all** 两种。
- **评估:✅ 拟采纳(全部)**。与原 §8"命令队列 + 打断/引导"同构但更精确,直接替换。
- **改动点(§8)**:单一"命令队列 + 中断信号" → **两条队列 + 两个固定消费点**;默认 `one-at-a-time`,`all` 作可选(批量转向慎用)。

## 2. 文件即大脑 / 分工 · Agent Harness(Microsoft Agent Framework)

- **来源**:Microsoft Agent Framework — Agent Harness / FileMemoryProvider(未核实)
- **要点**:状态落 `agent-file-memory/{session}/`;scale-to-zero 后文件系统原样恢复;Handoff 移交;BackgroundAgents 并行分叉。
- **评估:✅ 拟采纳(目录约定 + Handoff);⚠️ BackgroundAgents 谨慎采纳**(mini 单机资源有限,并行须受调度器/配额上限约束,非无限分叉)。
- **改动点(§2 / §9 / §10)**:workspace 增 **会话隔离目录** `sessions/{session}/`(对标 agent-file-memory);把"agent 醒来读文件"明确为 **scale-to-zero → 文件原样恢复**;§9 移交协议正式命名 **Handoff**;调度器支持 **受限并行(BackgroundAgents)**。

## 3. 上下文管理 · 增量 delta(ACE)

- **来源**:ACE,`arXiv 2510.04618`(未核实)
- **要点**:Generator / Reflector / Curator + 增量 delta;共享大脑文件 **增量追加 / 局部修订**,严禁整篇重写摘要;设专门 Reflector/Curator 维护趋势摘要层。
- **评估:✅ 拟采纳(增量更新);🔁 Reflector/Curator 角色:采纳但合并** —— 不新增两个独立层,由总管/主管 **兼任 Curator 职责**,避免角色膨胀;负载高时再独立。
- **改动点(§1 / §3)**:§3 规则二改为 **增量追加 / 局部修订,严禁整篇重写 `status` / `status-rollup`**;§1 给总管/主管加 **Curator 职责**(增量维护趋势摘要)。

## 4. need-to-know / 多模型 · 上下文模式(Microsoft Conductor)

- **来源**:`github.com/microsoft/conductor`(未核实)
- **要点**:三上下文模式 `accumulate` / `last_only` / `explicit`(explicit ≈ 员工只看自己依赖);逐 agent 指定模型/厂商;零 token 确定性路由;human gate;并行失败策略。
- **评估:✅ 拟采纳(全部)**。`explicit` 正是我们的 need-to-know,直接采纳为默认。
- **改动点(§3 / §5 / §10)**:§3 上下文传递显式化为三模式,**默认 `explicit`**(= need-to-know),`accumulate` 仅总管级、`last_only` 用于流水线;§5 增 **逐 agent 选模型/厂商 + 零 token 确定性路由**;§10 增 **human gate**(高危/拍板)与 **并行失败策略**(fail-fast / 收集全部)。

## 5. 多模型 / 配额故障转移 · LiteLLM + Ollama

- **来源**:LiteLLM + Ollama 混合网关(LiteLLM 真实开源,形态未核实)
- **要点**:LiteLLM 统一出口 + fallback 链;Ollama 兜本地。
- **评估:✅ 拟采纳(作为推荐实现形态);⚠️ 标注**:LiteLLM 增一层依赖 / 潜在单点,作为**可选**统一网关,不绑死。
- **改动点(§5 / §8)**:§8"配额感知故障转移 + 三路兜底"落为 **LiteLLM 统一出口 + fallback 链(订阅 → API → Ollama 本地)**;§5 多模型经 LiteLLM 接入,Ollama 兜本地/离线。
