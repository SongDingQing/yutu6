# 02 指挥层 · 任务规格、验收与多智能体编排
适用:orchestrator / secretary / board-* / reasoning-architect / governance / insight-scout

## 一份好任务的四要素(brief 质量决定产出质量)
1. **位置**:哪个文件/区段("改 X 文件的生图路由",不是"改生图功能")
2. **场景与约束**:具体到边界情况
3. **背景与影响面**:为什么改、会波及什么;进阶=要求先报 blast-radius 再放行
4. **验收标准**:机器可判定——"跑哪条命令、期望什么输出",不是"功能正常"四个字

## 拆分与规划
- 判据一条:**完成后能用一次独立验证确认对不对吗?** 能→粒度合适;要验证十件事→拆;一句话说清 diff→直接干
- 大功能:先让 AI **反向面试你**挖难点 → 产出 SPEC(含涉及文件/out-of-scope/端到端验证) → 拍板 → 干净会话执行
- 新模块第一问固定:"给 2~3 种方案及利弊,先别写代码"(防第一方案锚定)
- 指向参考实现而非描述风格:"参考 X 的模式实现,不引入新库"

## 验收模板(写进每份 brief)
- 正向用例 2~5 条可脚本化;边界/错误路径至少 2~3 条(未登录/并发重复/上游失败)
- **回归项**:相邻功能各验一条(AI 单点修改常改一漏三)
- 证据要求:贴原始输出/截图(UI 深浅色各一)
- bug 类:先贴失败复现,修后贴同一路径转绿
- **两个实证反模式**:让写代码的 AI 自拟验收(bug 被"验证"成正确);验收写"验证通过即可"(AI 挑最便宜路径)

## 人审节点(只拦不可逆高爆炸半径,其余自动过)
生产部署 / 删除覆盖数据 / 改数据格式 / 花钱超阈值 / 对外发送 / 触碰鉴权·存储·部署三高危面。审批请求必须带"将执行什么+diff 预览"。逢步必审=审批疲劳瞎批。(玉兔6 对应:元宵采纳/不采纳卡)

## 多智能体架构(2025-2026 业界最大共识)
**多数场景,单 agent+好工具+好上下文 > 多 agent**(Anthropic 与 Cognition 双双背书)。拆分是有成本的决策:多 agent token 消耗约为单聊天 15 倍。

**值得拆**:天然可并行互不依赖 / 上下文隔离有价值(脏探索留子 agent) / 权限隔离(不可信内容降权只读) / 读写分离(多个只读可并行,**多个写者不行**)。
**不值得拆**:"感觉更高级" / 子任务高度耦合要频繁互通(错误级联放大) / 单 agent 加个工具就能解决。

**五个编排模式**(够用即止):链式(步骤固定)/路由(按类型分发)/并行(量大互独)/编排者-工人(动态拆分)/评审-优化循环(有明确标准)。五个都不够再考虑自由协作(几乎轮不到)。

**上下文工程**:子 agent 收"目标+输出格式+边界"的**任务简报**,不传全史;大产物写外部存储只传引用;子 agent 只回蒸馏摘要(内部烧几万 token 可以,返回千 token 级)。Cognition 两原则:共享完整上下文含决策依据;两个写者各自基于不完整上下文行动,冲突不可避免→**写者唯一化或先抢锁**。

**失败模式三巨头**:错误级联(上游错误被下游当事实)→关键交接设校验门;成本爆炸(无预算门的递归拆分)→编排层硬预算上限;责任不清→每 agent 单一职责+独立可测。

## 工作流编排 12 原则精华(队列/引擎设计)
- **两层分离**:确定性编排层与副作用执行层分开;编排代码禁止直接 fetch/随机数/取当前时间
- **幂等是重试的前提**:幂等键=任务ID+阶段号;创建前先查;删除时 not-found 视为成功
- **一张任务表+不可变终态**:用户重试=新任务行不是复活旧行;前端报失败先查表和产物(账本才是真相)
- checkpoint 卡阶段边界,恢复前验证产物真实存在,干净跑完必须清 checkpoint
- **轮询为权威,推送为加速**;长步骤心跳+进度(30s 更新 heartbeat_at),扫描器只在心跳超时才判死
- 部署要排空在飞任务;人审门放动作前
- **硬栅栏并行是坑**:单个卡死拖死整批 → 子任务独立超时+容错聚合(手册记录 2026-07-07 两条工作流被单 agent 卡死 2 小时的实证)

## 玉兔6 差距(待老板拍板立项)
- 任务表统一 heartbeat 字段 + "卡在 working 超时"对账扫描器
- 编排层 per-任务 token 预算与告警(防成本爆炸)
- checkpoint 清理纪律核查
