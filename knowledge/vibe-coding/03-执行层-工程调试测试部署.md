# 03 执行层 · 工程、调试、测试、部署
适用:repair-lead / repair / it-engineer / quality-ops / gui-desktop-control(调试节)

## 调试纪律
- 修 bug 三件套:症状原文 + 疑似位置 + "修好"的定义;**先写失败复现测试再修,转绿为准**
- **修根因不压症状**:prompt 必带"不许压制错误或跳过测试";review 看到 skip/放宽校验一律打回
- 给 AI 运行时的"眼睛":前端看 console/network,后端喂日志切片——静态猜远不如实证
- 专项猎杀吞异常:grep 空 catch 块和"只打日志不上抛";点验全过但功能没发生几乎都源于此
- 高频 commit 当存档点,信息写给未来的 AI 看

## 文件存储工程(玉兔6 同为文件型存储,直接适用)
- **原子写四步**:写同目录临时文件→fsync→rename 覆盖→fsync 目录(跨文件系统 rename 会退化);替换前 JSON.parse 自校验;旧文件轮转 .bak
- 收敛成一个 `saveAtomic(path,obj)`,grep 全仓确认无裸 writeFile;**写入者身份必须与读取者一致**(root 写用户态文件→属主漂移→全站挂的实证)
- 数据格式变更:**expand→migrate→contract 三阶段,永不与代码同车**;加 schemaVersion;禁止一次变更中重命名/删除字段

## 部署纪律(三条+一条命根子)
1. **唯一管道**:任何"直接 scp/就地改生产"视为事故(双写通道=部署互踩的根因)
2. **不可变制品**:lockfile 提交且禁止随手重生成;部署用 `npm ci` 不用 install;钉死 Node 版本
3. **冒烟放行门**:部署脚本末尾固化——健康检查(含版本号)+完整登录序列+一次幂等写;任一失败→自动回滚→告警
4. **随时可回滚**:releases/ 目录+软链;`ln -sfn` 不是原子的(正解 `ln -s new tmp && mv -T tmp current`);保留 5 个 release;回滚演练至少做一次
- **门要先自检再下结论**:门自身故障(超时/环境缺)与被测系统故障分离输出(APP_FAIL 触发回滚 / INFRA_FAIL 只告警)——防"质量门把自身故障当质量结论"
- 小批量高频比攒大版本安全(DORA 实证:精英组变更失败率约 5%);feature flag 让"部署≠发布"
- 配置优先级链必须显式:启动时打印(脱敏)最终生效配置及来源;凭据类显式排除 env 覆盖

## 测试体系(quality-ops 重点)
- 金字塔:单元(毫秒/上千个)/集成(秒级/适量)/E2E(分钟/全站≤3条);**主力层=API 级集成测试**(真实 bug 多在组件边界)
- **测行为不测实现**:断言可观察行为;绑死实现的测试 AI 重构就红,AI 的自然反应是改测试→安全网变噪音
- 存量无测试代码:**特征测试**先锁现状(录"输入→实际输出"当断言);写完故意改坏一行确认会红,不会红的直接删
- 回归集事件驱动:每次线上 bug 先写复现测试(红)→修(绿);每条对应真实事故,含金量最高
- **安全不变量测试**:枚举全部路由发无凭据请求,断言全拒;白名单显式化——新加裸奔路由必红
- 冒烟必须从外面打真端口(TLS/cookie/env/属主这些"部署面"问题只有这层能抓)
- **Mock 纪律**:只 mock 不拥有的东西(外部 API);mock 自己的存储层=仪式
- **不给 AI 设覆盖率指标**(会刷高覆盖零断言),改设"关键路径必须有带断言的测试"清单

## 技术债(repair-lead 排债依据)
- 债的利息只在触碰时产生——烂但不动的代码让它烂着是理性决策
- **热点法**:改动频率×复杂度,只还"又复杂又高频改"的(热点占极小部分对应 25~70% 缺陷);月度 Top10 榜,停留两月立为偿还项
- **安全债例外**:利息由攻击者决定不由改动频率决定——公网可达>认证边界>权限操作>内部逻辑,季度暴露面清单一律排队列前列
- 重构嵌进功能任务(预备性重构 diff≤总 diff 20%);三次法则才抽公共;禁止大爆炸重写(绞杀者模式替代)
- 债的度量单位是**事故和返工工时**,不是"代码难看";同一根因第二次出现自动升级偿还项
