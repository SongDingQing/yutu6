## 1. 日常运营流程

后台秘书的职责不是做项目实现,而是把老板或前台 Cowork 的自然语言指令补成 CEO 能执行的 task 信封,并维护队列、公告板和共享能力目录的日常运营。实际工作流如下:

1. 接到老板/前台指令后,先读当前注入的背景包或运行 `node projects/控制台/secretary-tools.js context`。默认 compact 模式足够时不要把全文案例塞进信封;需要原文再按路径读取或用 full 模式。
2. 判断任务类型。非维修任务统一走 `secretary -> CEO -> 项目主管/专职队列`;维修、救火、权限、重启、孤儿进程、内存止血、系统授权等走 `repair-lead/repair` 或交主人手动。
3. 补 task 信封:写清 `goal/projectId/bounds/acceptance/case_reference`。没有明确项目时默认 `控制台`;边界必须含 Starlaid 排除、密钥不回显、登录授权交主人手动。
4. 如果任务涉及 UI、自动优化、架构设计、队列或 agent 体系,要读 `board/learning-cases/` 相关原则,并在信封里保留参考案例。涉及前门治理时,链路里保留 `secretary -> CEO -> supervisor` 的 taskId/queueId。
5. 用 `secretary-tools.js queue-enqueue --agent ceo --goal "..."` 把补全信封交给 CEO,由 CEO 判断路线和派发。不要直接越过 CEO 派给 worker,除非当前任务明确是维修员或工具队列的特权流程。
6. 派单后用 `queue-status` 看是否进入 CEO 或下游主管队列;重要任务还要在 brief/status/rollup 或事件日志里保留可追踪证据,但当前这份交接任务的 CEO brief 明确要求只写本文档。
7. 完成后交付时给可核路径、刷新时间、命令和输出摘要。不要把"已写方案"说成已落盘;没有改文件时 changed_files 为空。

## 4. 与各角色协作经验

- CEO:负责路线判断和项目制分派。秘书要给 CEO 可执行信封,不要替 CEO 做项目拆解。CEO brief 会落到 `projects/控制台/brief.md`,后续主管照 brief 执行。
- supervisor-控制台:负责把 CEO brief 变成 review-loop,再派 implement/review 节点。给它的验收要结构化、可核路径明确,否则容易被 done gate 打回。
- worker_code:能读写文件和跑测试,适合控制台代码、配置、文档、测试最小落地。给它任务时要明确 changed_files、测试、结构化验收表和边界。
- ui_optimizer:适合 UI 自动优化和学习案例回灌。它常需要同时改 prompt/test/case/event,但涉及 `workspace.html`、server、engine、队列语义或成本的内容要 owner_decision。
- repair-lead / repair:处理卡死、进程、launchd、权限、重启、孤儿进程、内存止血、长期 worker reload 和既有红灯。秘书只能提单或路由,不能冒充维修完成。
- memory_officer / memory-officer:历史上存在命名并存和 hidden alias 问题。未来要写长期记忆或技能规范时,先查当前角色治理方案,不要新增一个同义队列。
- 董事会/board:事前评审会给风险和修订建议。执行时优先采用最终董事的合并修订;runner 缺席、额度不足或认证失败不等于方案否决,但要记录。
- 前台 Cowork:负责深度交互、规格、定时、可视化沟通。后台秘书只做补背景、路由、队列和公告板运营,不要把前台任务直接变成私自实现。

谁需要重点盯:

- 历史 `tests/ceo-serial-lock.test.js:513` 红灯仍在维修单中,任何声称全量 `node tests/run.js` 绿的报告都要核实。
- Peekaboo 新截图经常受无显示器或授权影响;不能把 failure.json 当成完成的截图证据。
- enabled 公告板卡很多,不要重复启用或重复派同一洞察;先用 queueId 和 brief/status 查是否已有产物。
- prompt 或策略改动不代表长驻 worker 已加载;需要运行态生效时走 worker reload 维修链路。

## 5. 给继任者的建议

- 先看 `context_budget`。只要 compact 在预算内,继续用摘要;需要案例原文时按路径局部读。
- 每个信封都保留四件事:目标、边界、验收、证据链。前门治理类任务还要保留 `secretary -> CEO -> supervisor` 的 taskId/queueId。
- 不要把公告板当待办黑洞。enabled 卡片要核 queueId、brief、status 产物;todo 维修卡要交 repair-lead/repair。
- 不要把 Starlaid 作为可碰项目;只允许作为排除边界出现。
