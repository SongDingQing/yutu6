## 4) 维修工作的实操要点与坑

1. **先还原链路,再判级。**  
   每张工单先读 `board/repair-tickets/<id>.md`,再读 `engine-events.jsonl`、`engine-runs/<task>/`、`queues/<agent>/{queued,running,done,failed}/`。必须写清谁派给谁、brief 传了什么、期望返回什么、实际返回什么。不要只看最后一个 reason,因为最后 reason 经常是 gate/runner 包装后的二次症状。

2. **严重/小问题的分界要看系统面,不是看改动量。**  
   只改两行 `done-gate.js` 也可能是严重系统问题,因为影响所有 review-loop;只改一个工单状态则通常是小问题。涉及 engine/queue/runner/done gate/资源锁/服务/权限/模型路由,或 24h 反复出现、影响多个 agent、存在需求传递遗漏,都按严重处理并做全局链路核查。

3. **done gate 失败不要默认放宽 gate。**  
   维修期最常见误判是把 gate 失败当 gate 太严。正确做法是先确认 gate 保护的契约是否合理:若合理,修 LLM 信封、复审提示、证据水合或匹配逻辑;若自动生成的验收行制造了额外义务,才修触发条件。所有 gate 修复都要补正反例测试。

4. **SIGTERM/timeout 要分桶。**  
   SIGTERM 可能来自重启、清扫、主动取消、资源压力、running 文件缺失;timeout 可能是进程未启动、运行太久、末尾截图卡住。必须读 `process.log`、running/failed queue JSON 和事件序列。`engine_signal=SIGTERM` 不等于业务失败;`spawn codex 失败:超时` 也不等于 Codex 没启动。

5. **派 `repair` 前要避免同域死锁。**  
   如果 repair-lead 当前已经持有 `board`、`queue-state`、`brief-status`、`memory` 等写域,再派 repair 可能只是在同域锁后排队。派工必须写清:要查/改什么、不准碰什么、验收命令/证据、高危操作停下确认、Starlaid 排除、敏感凭据不回显。需要大量写码、重启服务、清进程、授权点击或长验证时再派;两行门禁/配置低风险小修可由主管直接做。

6. **复核不能信自述。**  
   声称改文件就核 `git diff`/文件行;声称测试通过就核命令、退出码、关键输出;声称服务恢复就核进程/HTTP/API/事件行;声称只是分析就核来源文件和推理链。`repair-ticket-complete` 的结果必须包含链路证据、需求传递判断、严重度、根因、处理、复核验证、架构判断、知识沉淀候选。

7. **旧工单清账要小心。**  
   当前 67 项非 done/closed 中有 23 个 `archived` 和一批 `oldjump/ceoenq` 旧竞态项。它们不一定需要再修,但状态枚举会影响统计。继任者清账时应按“同 fingerprint/同队列/同已修机制”批量归档或关闭,不要逐个重跑老业务任务。

8. **不要把红线文字本身当项目内容。**  
   历史 Starlaid 守卫误杀说明,扫描 goal/planText 时要区分“主动处理某项目”与“红线排除某项目”。维修交接期继续只处理控制台;看到 Starlaid 字样先判断它是边界声明还是主动任务,后者直接停。
