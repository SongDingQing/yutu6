# 玉兔6模型与智能体中枢：架构与迁移

日期：2026-07-21  
状态：Phase 1 已落地，影子前门运行；new-api 保留为兼容上游与一键回滚点。

## 1. 产品结论

新组件命名为 **玉兔6 Model Fabric（模型与智能体中枢）**。

它不是“再装一个 Nexent”，也不是给 new-api 换皮。它把模型网关、Agent 目录、
能力目录、A2A 发布、健康路由、用量账本和玉兔6硬完成判定组合成一个本机控制面。

准确定位是：**受治理的 AI 运行控制平面**。模型网关只是数据面，Agent Builder
只是设计面；玉兔6还必须覆盖持久化执行、权限、证据、评测、恢复和审计。

“更强”限定在玉兔6的实际目标：

| 能力 | new-api | Nexent | Model Fabric |
|---|---|---|---|
| OpenAI 兼容模型前门 | 强 | 有 | 有，且调用方与具体网关解耦 |
| 渠道/模型路由 | 强 | 依赖模型配置 | provider deployment + 熔断 + 有界 failover |
| Agent 可视化/目录 | 无 | 强 | 直接发现玉兔6真实角色，不复制第二份 |
| A2A | 无 | 强 | Agent Card + 任务入口，落回玉兔6控制面 |
| 共享工具/能力 | 无 | Skill/MCP | 复用 capability_registry，后续接 MCP |
| 真完成判定 | 无 | 平台自身状态 | 玉兔6 review + done gate |
| 默认常驻资源 | 低到中 | 高，多容器/数据库 | 低，单个零依赖 Node 进程 |
| 可替换性 | 自身是网关 | 重型平台 | provider、外部平台和 UI 都可替换 |

Phase 1 只是可运行内核，不宣称已经在渠道管理、计费后台或零代码 Agent 设计体验上
全面超过成熟产品。是否替代 new-api 必须由连续健康、真实协议覆盖和账本对账证明，
不能由架构文档自行宣布。

## 2. 架构原则

1. **统一目录，不统一成黑盒**：模型调用和长任务编排分开运行。
2. **逻辑模型不等于具体 deployment**：角色只选能力/模型别名，路由器决定上游。
3. **外部输出只是证据**：Nexent/A2A/模型都不能直接写 `task.done`。
4. **渐进披露**：先给 Agent 轻量能力目录，用到时才读完整模块说明。
5. **元数据可观测，内容默认不落盘**：计量不保存 prompt、回复、工具参数和密钥。
6. **迁移必须可逆**：在 direct provider 覆盖、健康和账本对账通过前，不删除 new-api。
7. **规划与执行分离**：ReAct/模型负责提出步骤，持久化状态机负责副作用、重试和恢复。
8. **验证失败关闭**：评测器缺席、异常或证据不足不能自动算通过。
9. **密钥最小作用域**：客户端凭据不透传，上游凭据只在 provider adapter 内解析。
10. **运行快照不可变**：每次受控任务逐步冻结模型、Agent、Skill、工具、权限和输入哈希。

## 3. 运行分层

```text
React 控制台 / CLI / 外部客户端
        |
        +-- OpenAI compatible /v1 -------------------------------+
        |                                                        |
        |  Model registry -> policy router -> health/circuit -> provider adapters
        |                                                        |
        |                         +-- new-api compatibility upstream
        |                         +-- direct OpenAI-compatible providers
        |                         +-- future local inference
        |
        +-- Agent/A2A control API -------------------------------+
                                  |
                                  +-- shared/agents
                                  +-- capability_registry
                                  +-- external platform catalog
                                  +-- 玉兔6 queue / engine / review / done gate
```

三条真相源严格分开：

| 平面 | 真相源 | 能做什么 | 不能做什么 |
|---|---|---|---|
| 模型数据面 | Model Fabric 路由与用量账本 | 选 provider、熔断、流式转发、计量 | 不能宣布任务完成 |
| Agent 控制面 | Agent/能力目录与 A2A task id | 发现能力、提交任务、查运行状态 | 不能直接改队列终态 |
| 执行证据面 | 玉兔6 queue/engine/review/done gate | 持久化执行、复核、恢复、回滚 | 不能信任外部自述替代证据 |

### 3.1 模型面

- `GET /v1/models`
- `POST /v1/chat/completions`
- `POST /v1/responses`
- `POST /v1/embeddings`
- `POST /v1/images/generations`
- `POST /v1/audio/speech`
- `POST /v1/audio/transcriptions`

第一阶段 `new-api-compat` 是兼容 deployment。调用方现在就可以改用
`http://127.0.0.1:3020/v1`；以后加直连 provider 时无需再改调用方。

### 3.2 控制面

- `/api/fabric/overview`：模型、provider、Agent、能力、平台和迁移状态。
- `/api/fabric/routes/plan`：只读解释路由，不消耗模型 token。
- `/api/fabric/health/run`：显式健康测试。
- `/api/fabric/usage`：脱敏计量。
- `/api/fabric/agents/<id>/run`：受控拉起普通 Agent。
- `/a2a/agents/<id>/*`：自动发布本机 Agent Card 与任务入口。

维修、桌面操作等特权角色不允许从通用入口直接启动。

### 3.3 双模编排

```text
目标
  -> 动态规划(ReAct/模型，可替换)
  -> 受控执行计划(确定性节点)
  -> queue lease + heartbeat + idempotency
  -> evidence manifest
  -> review + done gate
  -> done / retry / repair / owner decision
```

- 探索、研究和方案比较允许动态 Agent 迭代。
- 写文件、发布、维修、授权等副作用必须进入确定性节点。
- 节点状态、租约和证据落盘；模型进程退出后仍可恢复。
- 外部 A2A Agent 的 `completed` 只作为一条证据，不映射为玉兔6 `done`。

### 3.4 能力契约

统一目录中的模型、Agent、Skill、工具、MCP 和 A2A endpoint 后续共同使用以下字段：

```text
id / version / kind / input_schema / output_schema / capabilities
source / health / permissions / approval_level / secret_scope
timeout / retry_policy / side_effects / evidence_contract
```

目录采用渐进披露：首轮只给 `id + summary + capabilities`，真正选择后才读取完整说明，
降低上下文占用，也避免把所有 Skill 一次塞给模型。

## 4. 路由与故障模型

每个逻辑模型有一个或多个 deployment：

```text
logical model
  -> provider + upstream model + priority + capabilities
```

选择顺序：

1. 过滤禁用 provider 与能力不匹配 deployment。
2. 跳过尚在 cooldown 的 provider。
3. 按优先级、健康状态和最近延迟评分。
4. 网络失败、超时、429 和有限 5xx 才故障转移。
5. 普通 4xx 不盲目换上游，避免错误请求重复计费。
6. `max_attempts` 有硬上限，防止模型池故障时请求风暴。

健康既由 30 分钟主动探测维护，也由真实请求被动更新。连续失败达到阈值才熔断，
单次慢响应不会整家 provider 被永久禁用。

## 5. 账本契约

策略账本记录：

```text
event_id / task_id / trace_id / endpoint / requested_model
selected_provider / selected_model / fallback_candidates / decision
```

用量账本记录：

```text
provider / model / http_status / duration_ms / token counts
failover_count / billing_mode / error_class
```

明确不记录：

```text
prompt / response / tool args / tool results / Authorization / key / cookie
```

## 6. 替换 new-api 的四阶段

### Phase 1：影子前门（已落地）

- Fabric 运行在 `3020`。
- 所有模型可从 Fabric 目录发现。
- 请求仍落到 `new-api-compat`，但路由、健康和账本由 Fabric 接管。
- new-api 原端口、数据库和管理页不改。

### Phase 2：逐模型直连

- 按 provider 新增只引用本地 KEY 名的 adapter。
- 每次只迁一个低风险模型。
- 同请求只走一条生产路径，不做双倍计费的自动镜像。
- 对比成功率、延迟、token 数和错误分类；不比较 prompt 正文。

### Phase 3：Fabric 成为模型池主入口

- `model-routing.yaml` 与 API runner 的 base URL 改到 `3020/v1`。
- new-api 只作为低优先级兼容 deployment。
- 控制台模型池页改由 Fabric 概览驱动，new-api 后台降为“兼容层”。

### Phase 4：可选退役 new-api

只有同时满足以下条件才允许退役：

1. 所有生产模型有健康的直连 deployment。
2. 图像、流式、embedding 等实际使用端点均通过。
3. 连续观察期内 Fabric 账本与供应商用量可对账。
4. 回滚演练已验证。
5. 主人明确批准。

## 7. 与 Nexent 的关系

Nexent 保留为按需的 Agent 设计/知识库/市场实验平台，通过 A2A 被 Fabric 发现。
玉兔6不复制 Nexent 的 PostgreSQL、Elasticsearch、Redis、MinIO 与多服务常驻栈。

从 Nexent 吸收：

- A2A 与 Agent Card。
- 统一工具、Skill、记忆和版本目录的产品表达。
- 渐进式 Skill 披露。
- 反馈、评估与控制面分层。
- 调度任务的原子认领、租约续期、并发上限与过期 worker fencing。
- 上下文稳定前缀、动态区和硬预算裁剪。

不照搬：

- 第二套任务真相源。
- 外部平台自行决定玉兔6任务完成。
- 每个 Agent 复制一套模型密钥。
- 为本机轻量需求常驻完整容器集群。
- Skill/Python/Bash 直接继承宿主环境运行。
- 验证器异常时 fail-open。
- 凭据或自定义认证头通过管理接口明文返回。
- 只有 LLM 二值判断、没有确定性断言的发布门禁。

## 8. 超越 new-api 与 Nexent 的目标能力

### P0：成为可替换生产前门

1. 原生协议适配：OpenAI Chat/Responses/Realtime、Anthropic Messages、Gemini，
   以及 embedding、rerank、图像、音频；不能永远只靠 OpenAI 兼容转译。
2. 策略路由：按能力、健康、延迟、成本、地域和数据等级选择 deployment。
3. FinOps：按项目、Agent、任务和模型预算，提供硬上限与异常用量熔断。
4. 双模编排：动态规划 + 持久化状态机，具备幂等、断点恢复、补偿和人工检查点。
5. 隔离执行：工具运行进入受限子进程/容器，使用网络白名单、最小文件写权和 secret scope。
6. 不可变运行清单：保存版本与哈希，不保存 prompt/response 正文，支持事故重放。

### P1：形成质量闭环

1. 多指标评测：确定性断言、成功率、工具轨迹、安全、引用、成本和延迟。
2. 影子/灰度/回滚：只对脱敏基准样本做影子对比，生产请求默认不双发、不双倍计费。
3. 三层记忆治理：语义、情景、程序性记忆分开，带来源、TTL、人工晋升和遗忘。
4. 产品级观测：trace、metric、log、成本与上下文组成关联到同一 task/trace id。
5. 数据与引用血缘：原文、切片、检索、重排、引用与 ACL 全链可追踪。

### P2：可信生态

1. Agent/Skill/MCP 包签名、SBOM、权限清单、兼容矩阵和安装前安全扫描。
2. A2A Agent Card 签名、委托身份、预算上限与远程能力降级。
3. 版本化 Agent 市场只分发描述和签名包，不分发模型密钥。

## 9. 替代判定

Model Fabric 只有满足以下量化条件，才有资格接替 new-api 的 `3000` 端口：

| 指标 | 门槛 |
|---|---|
| 生产协议覆盖 | 当前调用端点 100% |
| 生产模型直连覆盖 | 100%，且每个关键模型至少一个健康 deployment |
| 稳定观察 | 连续 14 天无 P0/P1 路由事故 |
| 成功率 | 不低于 new-api 基线，差值不超过 0.2 个百分点 |
| P95 延迟 | 不劣于基线 10% 以上 |
| 账本对账 | 调用数/token 与供应商侧可解释一致 |
| 回滚演练 | 5 分钟内恢复旧入口 |
| 最终授权 | 主人明确批准 |

满足前仍采用 `3020` canary。真正切换时可以停止 new-api 后让 Fabric 监听 `3000`，
也可以继续保留 `3020` 并只改调用方；端口选择不影响控制面设计。
