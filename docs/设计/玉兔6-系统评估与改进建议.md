# 玉兔6 系统评估与改进建议(设计者视角)

> 秘书作为项目设计者的全面审视。**先不处理,供你定优先级。** 日期 2026-06-20。
> 基于实查:核心代码 ~4783 行、9 个 agent、文件式队列引擎、三视图 + 任务板、维修员 v2、记忆系统(新)、知识库 RAG 栈、统一密钥保险库。

## 一、总体评价
玉兔6 已经是一个**能自我运转的多智能体系统**:能接任务、拆解、路由、执行、失败自愈、对外通知。架构选型克制(文件式、零依赖、原子认领),这点很对——简单的系统才扛得住长期演进。
当前阶段的主要风险不在"功能不够",而在**韧性和可观测性还没跟上功能的增长速度**。

## 二、做得好的(亮点)
- **文件式零依赖队列**:原子 `rename` 认领、心跳续约 + 孤儿清理,简单可靠。
- **角色化多 agent + 路由**:层级清晰,职责分明。
- **维修员 v2 闭环**:失败自动开单→入队→修复→根因/架构判断→飞书,真正的自愈。
- **密钥治理**:统一保险库 `~/.config/yutu6-secrets/`(chmod 700、不进 git),规范。
- **记忆系统底座**(新):分层 + 记忆官,方向对。

## 三、改进点(按优先级)

### P0 · 韧性(最该先补的"安全网")
1. **几乎没有自动化测试** —— 4783 行核心代码,只有配置校验(agents-check),**没有引擎/队列/状态机的单元或集成测试**。现在每次改动(尤其引擎核心)只能靠 Peekaboo 截图肉眼验,**回归风险随规模恶化**。建议:给队列引擎、状态机、路由补一套冒烟+回归测试,改动前自动跑。
2. **没有自动备份** —— 队列、记忆、config、工单全是本机文件,只有**手动**迁移包。一次误删/写坏/磁盘故障就丢数据。建议:定时快照(队列+memory+board+config)到外接盘/云,留多版本。
3. **JSON 写入非原子** —— `queue.js` 的 `writeJson` 用 `writeFileSync`,**进程在写一半时崩溃会损坏队列/状态文件**。建议:写临时文件 + `rename` 原子落盘(认领已经用了 rename,写入也该一致)。

### P1 · 能力释放(已有基础设施没用上)
4. **RAG 知识库建好了却没接入** —— `knowledge/` 有完整 RAG 栈(ingest/query/embed/kb.sqlite),但**没有任何 agent 在干活时调用它检索**。等于盖了图书馆没人借书。建议:和记忆系统**打通**,agent 干活前统一检索"知识库 + 记忆"。
5. **成本不可观测** —— 有 token usage 痕迹但**没有聚合**。你很在意省钱,却看不到钱花在哪。建议:成本面板(各 runner/任务的 token + 生图花费),按天/任务汇总。
6. **监控告警薄弱** —— 有 `/api/probe` 探活,但**没有系统级健康看板 + 主动异常告警**(引擎挂、队列堆积、连续失败)。建议:健康看板 + 异常主动飞书告警,别等你发现。

### P2 · 长远演进
7. **全串行的吞吐瓶颈** —— 刚改全串行(防死锁,当下正确),但任务一多会排长队、变慢。记住这是**临时取舍**;将来在"无依赖任务/独立项目可并行 + 死锁检测"上再放开。
8. **单点风险** —— 全在一台 Mac mini,机器挂=全停。手机元宵端(cc-connect)缓解远程访问;备份缓解数据单点;长远可把关键状态同步到第二处。
9. **记忆质量要持续打磨** —— 记忆系统新建,**检索注入的相关性 + 提炼的准确性**要盯,否则记忆会变噪音(定期整合就是为这个)。
10. **前端单文件膨胀** —— `workspace.html` 已 1489 行,继续堆功能会难维护(你偏好单文件,这是权衡,先记着)。

## 四、你可能想不到的几个深层点
- **"测试网"缺失是当前最大的隐性债**:系统越聪明、改得越频繁,没有测试网就越容易"修一个坏三个",而且不易察觉。这是我最担心的一条。
- **你其实有"两套记忆"没打通**:老的 `knowledge/` RAG + 新的 `memory/`。应统一成"一个检索入口",否则各自为政、都发挥不出来。
- **系统现在是个"高速黑盒"**:出问题靠翻 jsonl。一个把"健康 + 成本 + 任务流"合一的可观测面板,会让你第一次真正"看懂"系统在干嘛。
- **让系统自己体检**:可以设一个定时"系统体检"(交给监管/质量运营角色),定期产出健康报告(失败率/成本/堆积/回归),**主动**发现问题——而不是等你撞见。这是从"自愈"走向"自省"。

## 五、我的建议(不急着全做)
先补 **P0 安全网**(测试 + 备份 + 原子写)——它们不增加功能,但决定系统能不能扛住后面越来越快的迭代。
再做 **P1**(打通 RAG+记忆、成本面板、健康告警)——这几样能立刻让你"省心 + 看得懂"。
P2 随手机端、随规模再推进。
