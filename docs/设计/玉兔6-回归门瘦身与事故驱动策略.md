# 玉兔6回归门瘦身与事故驱动策略

日期: 2026-07-16

## 目标

约束门不是越多越安全。有效门禁应满足三个条件:

1. 能指回具体事故、失效模式或基础安全责任。
2. 只在匹配风险的任务上运行。
3. 能量化命中、误伤、耗时，并能休眠和恢复。

本轮目标是在不削弱真完成、密钥保护和运行稳定性的前提下，降低正常任务的误拦截、重复复审、全量回归和模型输入长度。

## 门禁状态机

### active

会执行，失败可阻断。只允许两类:

- 基础安全门:密钥卫生、任务真实完成、队列状态完整性。
- 有具体事故记录，且已有正例、反例回归的风险门。

`active + block` 注册时必须带 `incident_refs`。生产 HookRegistry 开启来源强制校验后，无事故来源的新硬门不能上线。

### shadow

会执行、会记录 `observedOk` 和 `gate.incident`，但不会阻断任务。适用于:

- 新门上线前观察误报率。
- 仍有争议的分类规则。
- 外部发布、通知等不应反向否定业务完成的附属流程。

### dormant

完全不执行，但策略、事故来源、恢复条件仍保留。适用于:

- 已被主门覆盖的重复 hook。
- 没有事故依据的预防性限制。
- 长期无命中、误伤高于收益的门。

## 回归测试分层

### 专项

优先运行与 changed_files 和事故类型直接相关的单测。例如:

```bash
node tests/gate-policy.test.js
node tests/hardening-hooks.test.js
```

### smoke

普通任务的快速合同检查:

```bash
node tests/run.js --profile smoke
```

只跑完成协议、门禁策略、证据执行、loop/context 预算等轻量合同。密钥保护等生产功能没有关闭；这里只是不在每个普通任务里重复测试扫描器自身。

### lean

核心引擎、队列、门禁和路由改动的事故型安全网:

```bash
node tests/run.js --profile lean
```

只包含有明确事故依据的完成门、队列恢复、路由和密钥卫生测试。其余测试为“例行休眠”，不会删除，仍可在 full 唤醒。

### full

继续保留兼容入口:

```bash
node tests/run.js
node tests/run.js --profile full
```

只用于跨引擎改造、发布前、基线刷新、人工全检或专项回归显示存在跨域风险时。不能再把它当作所有小任务的默认完成证据。

本机 2026-07-16 实测:

- smoke: 11 个文件，1.83 秒，全部通过。
- lean: 22 个文件，16.81 秒，全部通过。
- full: 97 个文件，41.07 秒，全部通过。
- 普通任务从 lean 下沉到 smoke 后，回归等待约减少 89.1%。
- 相对 full，smoke 的实测等待减少 95.5%；86 个测试在例行任务中休眠，文件数减少 88.7%，仍可按风险唤醒。

## 循环与上下文预算

Loop engineering 不再对所有普通任务默认开启:

- 显式 `loopEngineering:true` 始终开启。
- UI/视觉/设计、架构、多智能体、skills/prompt、hook/门禁/路由等迭代收益明确的任务开启。
- 普通修复、配置、只读核查默认单轮 review-loop。
- 定时、洞察和自动优化线路沿用轻量模式，除非显式覆盖。

循环评分标准只从结构化验收表的真实要点提取，不再把表头、模板路径和 `---` 分隔线当标准。

知识注入同样按职责收缩:

- 执行节点按需注入少量知识和历史教训。
- 董事、主管复审和 orchestrator-plan 只核任务、实际文件、diff、命令和测试，不再重复灌入生成阶段知识。
- 默认历史教训从 top3/1200 字收缩为 top2/700 字，可通过环境变量临时调大。
- 默认知识库仅 top1/280 字；只在生成或执行节点注入。策略实现位于 `shared/engine/context-budget.js`。
- 实时董事样本的旧信封共 6147 字符，其中无关知识库/教训占 1791 字符（约 29.1%）；新策略在董事节点完全移除这两块。
- CLI handoff 默认 `auto`:目标超过 2500 字符时仅内联一句意图、任务稿绝对路径和指纹；短任务、董事节点、HTTP 文本 runner 保持全文，读取或指纹异常自动回退。

证据命令也按范围执行:

- `node tests/<专项>.test.js` 自动复核。
- `node tests/run.js --profile smoke|lean|gate|runtime|routing|full` 属于显式范围，可自动复核。
- 裸 `node tests/run.js`、`npm test`、`pnpm test`、`yarn test` 不在收尾阶段自动重跑，只记录为未分档证据。
- 对应策略位于 `shared/engine/regression-command-policy.js`，避免模型运行一次、DoneGate 又运行一次。

主 DoneGate 内的事故型硬回归也采用同一生命周期。每条子规则必须声明
`mode`、`reason`、`incident_refs`；只有 `mode=active` 的规则会阻断。当前
`queue_merge_integrity` 因真实队列假完成事故保持 active，未来若事故消失或
误伤率超过收益，可直接转 dormant，规则与历史证据仍保留。

## 事故记录与生命周期

Hook 失败或 shadow 观察到不一致时，HookRegistry 自动写:

- `hook.executed`:模式、实际判定、有效判定、耗时。
- `gate.incident`:gateId、原因、是否阻断、事故来源。

主 DoneGate 继续写 `done_gate.blocked`。

质量运营使用 `projects/控制台/tools/gate-load-audit.js` 汇总:

- 各 gate 运行次数、命中、平均和最大耗时。
- 未分档全量回归的历史使用量。
- lean/full 文件数与例行休眠比例。
- 无命中或重复门的休眠候选。

门禁恢复顺序:

1. 新事故进入事件日志或维修工单。
2. 补最小复现与正反回归。
3. dormant -> shadow，观察误报。
4. 误报可接受且确有漏网风险时，shadow -> active。
5. 超过策略中的无命中天数后重新审核；没有继续存在的事故依据则回到 dormant。

## 不休眠的底线

- 密钥、token、私钥泄漏检查。
- 主 DoneGate 的 implementation/review/证据一致性。
- 队列原子状态、心跳、孤儿恢复。
- 明确发布任务的签名、安全和版本门。

性能优化不能以再次允许假完成、泄密、误杀活进程或丢队列状态为代价。
