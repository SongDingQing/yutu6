# 玉兔6 · 洞察员审查负载与硬化架构

- 日期: 2026-07-16
- 状态: v1 已落地低风险项；中风险路由项待主人拍板
- 审计对象: `board/insights/agent-harness-deep-research-20260714.md` 的 50 条 AHR 建议及其 6 个实施主题包
- 机器审计: `projects/控制台/artifacts/insight-workload/latest.{json,md}`

## 1. 结论

50 条建议本身大多有价值，主要问题是“建议进入实施”之后没有按风险和产物类型降档。6 个主题包即使只是只读审计、设计稿或 PoC，也普遍走了董事会、CEO、主管、实现、复审、结构化证据和部分 loop engineering，累计占用普通执行链约 5 小时 46 分。

本轮架构方向不是删除 done gate，也不是降低真实改码任务的质量，而是把机械检查前移到脚本，把任务上下文切到当前任务，把模型评审保留给语义风险。

### 2.3 已验证的限制反噬

1. 董事评语提到“UI/队列消费者”时，纯引擎任务被追加视觉证据行。
2. 主管读取追加式全历史 `brief.md`，旧任务目标和旧验收可混入当前任务。
3. AHR-26..30 已形成实施包和测试，但最终被与当前任务不一致的验收行打回。
4. AHR-17..25 为纯架构 PoC，却为了误触发的视觉行生成 648KB 视觉证据目录。
5. 公告板已有至少两组语义重复的洞察卡，只因 ID/slot 不同而重复进入系统。
6. “50 条候选账本”标题与“15–50，50 只是上限”的规则冲突，会给研究角色形成凑数暗示。

## 3. 目标分层

### L0 · 确定性入口

由脚本完成：

- 候选数量、ID 唯一性、来源数；
- AHR 编号与主题包覆盖；
- 公告卡内容指纹去重；
- 队列终态、失败原因、主题包累计时长；
- 报告格式和秘密材料粗筛。

这一层零模型调用，失败直接给结构化 reason。

### L1 · 风险与产物分类

每个洞察任务标记：

- `mode`: `light | deep | audit`
- `deliverable`: `research | proposal_only | canary | runtime_change`
- `risk`: `low | medium | high`
- `ownerApproved`: 是否已经由主人点启用

分类结果进入信封和事件日志。分类仅决定评审预算，不改变 done gate 的证据真实性要求。

### L2 · 最小语义复核

模型只回答脚本无法回答的问题：

- 证据是否真的支持建议；
- 是否越过角色边界；
- 是否会改变权限、路由、并发、全局 hook；
- 是否值得主人拍板；
- 回滚是否可信。

不要再让三个角色各自复算行数、编号、重复卡和队列用时。

### L3 · 运行时执行门

只有 `runtime_change` 才进入完整董事会、CEO、主管、实现、复审和 done gate。研究报告、只读审计、纯方案稿仍须有可核证据，但不应默认制造 canary、截图或多轮 skill 改写。

## 4. 评审预算

| 类型 | 默认链路 | 模型预算 | 视觉门 |
|---|---|---:|---|
| 定时 light scan | 洞察员 agent-once | 1 个角色，2–3 来源，最多 3 候选 | 仅原始目标明确要求视觉时 |
| 主人点名 deep research | 洞察员 + 静态脚本 + 质量/监管各一次语义复核 | 8–15 来源，15–50 候选 | 研究网页本身不是 UI 交付，不自动触发 |
| 主人启用 proposal_only | 当前仍走既有链；建议后续改成一次实现 + 一次复核 | 不启用 loop engineering | 只看老板原始目标/正式验收 |
| canary | 一次实现 + 一次硬评估 | 固定样本与预算 | 仅 canary 真的验证 UI 时 |
| runtime_change | 完整链 | 按风险分配 | 保留现有硬门 |

## 5. 本轮已落地

### 5.1 角色提示词按流程裁剪

`projects/控制台/engine-runner.js` 只加载本流程会实际运行的角色：

- review-loop: `worker_code + supervisor`
- agent-once: 当前角色；需要 CEO 时再加 `orchestrator`
- project-route: `orchestrator + board_*`

旧任务文件不改，新任务开始生效。

### 5.2 任务级 brief

历史总账继续追加到 `projects/<project>/brief.md`，但每个任务另写：

`projects/<project>/artifacts/task-briefs/<taskId>.md`

主管只读取任务级 brief。CEO 规划只读取当前信封和项目 `status.md`，不再把 2.2MB 历史 brief 当热输入。

这同时解决：

- 旧验收污染；
- 重复 token；
- 主管长期持有 `brief-status` 读锁导致后续 CEO 写 brief 被串行阻塞。

### 5.3 视觉门输入隔离

结构化验收的任务类型只依据：

- 老板原始目标；
- orchestrator 正式 acceptance。

董事评语和路由说明不再作为视觉分类输入。真实 UI 目标仍会注入 Peekaboo/Codex 视觉证据行。

### 5.4 洞察卡内容指纹

自动洞察卡新增稳定 `insight_fingerprint`。不同 slot、不同 ID 但标题/目标相同的卡会以 `duplicate-content` 跳过，不再重复进入公告板。

自动洞察卡同时显式写入：

- `loopEngineering=false`
- `insightWorkload.mode=light`
- `insightWorkload.proposalOnly=true`

### 5.5 15–50 非配额校验

报告标题改为通用 `## 候选账本` 或实际数量前缀。validator 接受 15–50 条，不再要求标题固定写 50。

### 5.6 洞察负载审计器

新增：

`node projects/控制台/tools/insight-workload-audit.js --report <report.md>`

固定输出：

- `projects/控制台/artifacts/insight-workload/latest.json`
- `projects/控制台/artifacts/insight-workload/latest.md`

## 6. 仍需主人拍板

以下会改变现有治理路线，本轮不自动启用：

1. 主人已手动启用且标为 `proposal_only` 的洞察卡，是否跳过一次重复董事会评议。
2. `proposal_only` 是否从完整 review-loop 降为“一次实现 + 一次主管硬复核”，禁止自动生成 canary/fixture/skill patch。
3. 资源锁是否改为“拿到 engine slot 后再持有写锁”，或等待 slot 时释放可重建锁，避免空等任务阻塞整条 CEO 队列。
4. 首周质量运营仍保持全量链路覆盖，但是否把机械核对全部交脚本、模型只审异常链和语义风险。

## 7. 指标与回滚

### 观察指标

- 新任务 `vars.agentPrompts` bytes 与 role 数；
- supervisor task input 是否只含任务级 brief；
- `resource.scheduler.all_blocked` 中 `brief-status` 冲突次数；
- 非 UI 任务视觉行误注入次数；
- 洞察卡 `duplicate-content` 跳过数；
- proposal_only 平均耗时、模型节点数和产物数。

### 回滚

- 提示词裁剪：让 `promptSelectionForSpec()` 返回 `null` 即恢复全量加载。
- 任务级 brief：主管输入恢复 `projects/<project>/brief.md`；历史总账从未停止写入。
- 视觉隔离：`buildSupervisorAcceptance()` 恢复使用 `supervisorGoal`。
- 卡片去重：移除 `insight_fingerprint` 检查；已有卡不受影响。
- validator：恢复固定标题检查不影响历史报告内容。

## 8. 设计原则

1. gate 应验证结果，不应把每个任务都变成一场治理演示。
2. 能由脚本确定的事实，不调用模型重复判断。
3. 历史总账是冷证据，不是每轮热上下文。
4. 董事可提出风险，但不能通过评语关键词偷偷改写任务类型。
5. 主人批准一个提案，不应自动等价于批准无限迭代、视觉制作和全链 canary。
6. 不保存或要求隐藏思维链；保留任务 prompt、可观察输出、命令摘要、证据引用和结构化结论即可。
