# 元宵补齐飞书交互能力：原生决策卡、问题处理报告与 typed-card 闭环

## 任务元数据

- `projectId`: `元宵`
- `owner`: 元宵项目主管
- `route`: 秘书 -> CEO -> 元宵项目主管 -> 移动端开发 / 桥接后端 -> 质量复审
- `priority`: 高
- `releaseImpact`: `minor`
- `external_release`: 主人确认后执行

## 背景与目标

玉兔6目前同时通过飞书和元宵向主人发送消息、决策请求与处理结果。飞书已经具备原生交互卡片、按钮回调、幂等决策和维修报告附件；元宵已有聊天、任务队列、typed-card 后端、文件/文档中心和维修 HTML 投递基础，但部分生产脚本仍退化为 Markdown 文本，要求主人回复“采纳 N / 不采纳 N”，没有形成移动端原生的一键操作闭环。

本任务以“业务结果对齐”为目标，不机械复制飞书专属的群聊、@、租户或已读语义。完成后，主人应能在元宵内直接完成决策、查看进度与失败原因、打开问题处理报告，并在断网重连或重复点击时保持一致结果。

## 实施阶段

### P0：真实能力矩阵与统一合同

1. 逐项运行或阅读现有实现，产出“已支持 / 部分支持 / 缺失 / 不适用”的证据矩阵。
2. 复用元宵现有 `typed_card`、`/api/v1/cards`、`/api/v1/cards/answer` 合同，不另建平行消息协议。
3. 统一卡片字段：`cardId`、`type`、`title`、`summary`、`status`、`actions`、`artifactRefs`、`taskId`、`idempotencyKey`、`createdAt`、`updatedAt`。
4. 决策动作只传不可伪造的卡片/动作标识；本地 HMAC secret、API token 不下发到 Android 客户端。

### P1：Android 原生卡片

实现并统一视觉：

- `approval/decision`：同意、拒绝、可选原因；点击后原卡变为已同意/已拒绝。
- `progress`：阶段、进度、最近更新时间、可展开详情。
- `failure`：明确原因、证据摘要、重试或创建维修工单动作。
- `report`：问题概述、严重度、根因、处理过程、验证证据、剩余风险、打开 HTML/PDF。
- `artifact/checkpoint/trace`：统一摘要、文件入口和链路入口。

UI 沿用元宵既定消息规范：全宽、无头像、消息间有清晰分隔、适当行距；不使用 WebView 模拟卡片，不用 URL 跳转代替本地动作。

### P2：玉兔6发送与回调闭环

1. 将 `notify-yuanxiao-approval.js` 从“发送 Markdown + 等主人文字回复”升级为优先提交 typed decision card。
2. 保留旧版本客户端的文字回复降级通道，但必须能明确识别为 fallback，并可观测。
3. 卡片点击通过元宵桥接服务映射到玉兔6本地决策回调；复用现有 decision callback 的 fail-closed、幂等和审计语义。
4. 重复点击、网络重试、桥接进程重启均不得重复入队或重复执行。
5. `repair-report-delivery.js` 在保留固定 HTML 文档的同时发送 typed report card；报告更新时更新同一卡片，不重复刷屏。
6. 失败卡可安全触发现有维修工单入口；高危、删除、上传、登录、授权仍需主人确认。

### P3：离线、通知与兼容

1. 断网后卡片和未完成动作持久化；恢复连接后按 cursor/revision 补齐，动作重放保持幂等。
2. Android 系统通知显示卡片标题和必要摘要，点击进入元宵对应会话/报告，不要求复制飞书群聊语义。
3. 老版本 App、桥接服务暂时不可用、未知卡片类型均有可读文本降级，不得整页崩溃。

### P4：文档、测试与本地构建

1. 新增或更新：
   - `docs/chang-e-android-control-plane/feishu-yuanxiao-capability-parity.md`
   - API contracts / agent control plane contracts
   - 玩家可见版本日志（只写可公开功能，不写密钥、内部防线或服务器信息）
2. 增加自动化测试，至少覆盖：
   - 决策同意/拒绝；
   - 无效动作 fail closed；
   - 重复点击只执行一次；
   - 报告卡和 HTML 产物一致；
   - 失败卡创建维修工单；
   - 离线重连与动作重放；
   - 未知卡片和旧客户端 fallback；
   - 消息无头像、全宽、分隔与行距规则。
3. 运行玉兔6相关专项测试、元宵 Python/Android 测试与 `ops/scripts/verify_release_candidate.py`。
4. 构建本地 Android 安装包并记录路径、版本号、SHA-256、测试证据。未经主人确认，不 push、不发布、不覆盖线上包。

## 验收标准

1. 主人在元宵里收到真实决策卡，点击“同意”或“拒绝”后无需浏览器、无需回复文字，任务只流转一次。
2. 卡片立即展示终态；重复点击、断线重试、服务重启不会产生第二次副作用。
3. 真实维修结案在元宵出现结构化报告卡，能查看摘要并打开对应固定 HTML/PDF；报告正文包含根因、过程、验证与剩余风险。
4. 真实失败任务出现 failure card，原因不再只有 `SIGTERM` 等裸状态，并能安全转维修流程。
5. 旧客户端仍能收到可读文本；任何密钥、token、cookie、私钥不进入客户端、日志、任务产物或 Git。
6. 能力矩阵中的每个“完成”都有文件、测试或真实冒烟证据；主管复审必须对照实际结果，不能只读执行者自述。
7. 本地 APK 构建成功；外部发布保持待主人确认。

## 边界与回滚

- 不改变现有 `/api/*` 的兼容语义；只能向后兼容扩展。
- 不删除现有 Markdown/HTML 降级能力，待新版稳定后另行决策是否清理。
- 每阶段独立提交或形成可回滚 diff；若原生卡片链路异常，可切回文字 fallback。
- 不读取或回显密钥；不在任务正文记录主机公网地址、账号、token 或 cookie。
