# 维修员 Repair Agent · 设计规格

> 由 codex(特权版)担任,处理「秘书(Cowork 前台)够不到」的活。秘书出规格 → 独立 codex 落地。出规格 2026-06-19。

## 背景
反复出现「秘书够不到、得给老板一段话转交独立 codex」的活:重启服务、系统授权(屏幕录制/飞书)、改引擎核心、docker 部署、紧急修 bug(如孤儿清理误杀活 engine)。把这个模式正式化成「维修员」角色,少让老板当传递人。

## 定位(只接秘书够不到的)
- **进程/服务级**:重启控制台 server、拉起/清理 worker、launchd 服务。
- **权限/授权**:屏幕录制/辅助功能、飞书授权、用 computer use 点系统设置。
- **引擎核心修复**:改 server.js / ceo-worker.js / 引擎、修系统级 bug。
- **部署/环境**:docker、装 skill、配 key。
- **版本管理协作**:Gitee/Git 的 push、pull、远端同步、四段版本号提交与安全回滚已改由 `it-engineer`/IT 工程师负责。维修员只在页面完全无法修复时通过 `node projects/控制台/secretary-tools.js it-rollback-request --target <version-or-commit> --reason "<原因>"` 发起协作,不直接改写 Git 历史。
- **紧急救火**:系统不稳、任务被误杀、崩溃/重启循环。
- (文件级 / 前端 / 配置类仍由秘书直接派 CEO,不进维修。)

## 担任 = codex 特权版
- 有 computer use + 本地权限 + 能重启进程;**区别于 `worker_code` 的 sandbox codex(受限)**。
- **在系统外/特权运行,不被系统清扫(孤儿清理)误杀** —— 关键,否则救火的人自己被杀。

## 工单通道
- 秘书识别「够不到」→ 写维修工单 `board/repair-tickets/<id>.md`(问题 + 事件证据/路径 + 期望 + 红线),并在事件/公告板标一条「维修工单」。
- 维修员(独立特权 codex)消费工单 → 处理 → 回报(写结果 + 飞书通知老板,飞书已通)。
- **v1 半自动**:秘书写工单 + 独立 codex 消费;**v2**:常驻特权 codex worker 监听 `repair-tickets/` 自动接单(系统稳定后再做)。

## UI
工作区「公共协作区」加「维修员」节点(🔧 扳手,运维橙),显示状态/当前工单。

## 红线(特权角色,务必)
- **高危/不可逆操作(删数据、移动资金、改密钥分发)仍须老板确认**;
- 密钥不回显;Starlaid 排除;不破现有功能;
- 维修员独立运行、**绝不被系统孤儿清理误杀**(依赖当前紧急修先把误杀 bug 修好,维修员才能安全常驻)。

## 落地顺序
1. 先修完当前紧急 bug(孤儿清理误杀活 engine),系统稳定;
2. 再落地维修员:角色注册(runners.yaml + config.json,runner=特权 codex)+ repair-tickets 工单 + UI 节点 + 秘书写工单约定;
3. v2:常驻特权 codex worker 监听 repair-tickets 自动接单。
