# 任务板 v2 + 全串行执行 · 设计(秘书思考)

> 老板要求(2026-06-20):① 任务板以「CEO 收到的任务」为单位,不是各 agent 各自报状态;② 一个任务内多 agent 协同的状态,显示在该任务卡内部;③ 担心并发死锁(A 等 B、B 又等 A),**先全部串行**,以后有机会再议并行。

## 现状(已查证)
- 引擎 `ENGINE_MAX_CONCURRENCY=2` + 各 agent 队列各自拉 worker → **并发**;死锁隐患真实存在。
- 任务板「进行中」现在按 **agent** 显示:同一个任务流经 CEO→主管→程序员,会显示成好几张不同 agent 的卡,看不出"这其实是一个任务"。

## 一、执行模型:全串行(防死锁)
1. **并发度 = 1**(`ENGINE_MAX_CONCURRENCY=1`)。
2. **全局单活跃任务**:一个 CEO 任务从激活,到它**所有节点**(含路由给主管/程序员的子节点)全部完成/失败之前,**不启动下一个 CEO 任务**(全局"活跃任务锁")。
3. 任务内部节点按 **DAG 依赖顺序**串行推进,同一时刻只有一个节点在跑。

→ 同时只有**一个任务、一个节点**在动 = 无并发 = **无循环等待死锁**。代价是慢(任务排队等),但稳、可预测。老板已接受。

## 二、任务板:以「CEO 任务」为单位
- **进行中区** = CEO 任务队列全景:**执行中(1)+ 排队(N)全部列出**,第一个标「执行中」、其余「排队中」。(老板:正在处理 1 个 + 队列 9 个 = 进行中页面有 10 个)
- **任务卡**:
  - 外层一行:目标精简(**问题 + 解决方案**)+ 状态 + 实时时长。
  - 展开内层:该任务流经的 **agent 节点链**,每节点显示 `[谁]` + `[状态]`:✅完成 / 🔵运行中 / ⏳等上游 / ⚪待开始。
    - 例:`CEO 规划 ✅ → 主管 ✅ → 程序员 🔵 → 主管复审 ⏳`
  - 串行下,老板举的「4 agent:1 完成 2 运行 1 等待」实际会是「1 完成、1 运行、其余待开始/等上游」(同一刻只 1 个在跑)。
- **数据来源**:`engine-events.jsonl`(node.start/end/fail + edge)+ `engine-runs/cr-*/` 聚合成「任务 → 节点」树。

## 三、布局(承接上一轮)
过往(上,3 个可滚回看)/ 待办(中,缩短)/ 进行中(下,**放大、主角**)。

## 四、死锁防范小结
- 全串行 = **结构性免疫**(没有并发就没有循环等待)。
- 任务内是 DAG(无环),按拓扑序执行不会死锁。
- **未来若上并行**(以后):需 ① 依赖图显式声明 ② 死锁检测(等待图找环 / 超时重试)③ 资源有序分配。先不做。
