# Claude Code & Codex 用量获取 · 方案研究

> 老板点名让秘书研究:怎么拿到 Claude Code / Codex 的具体用量(5小时+周额度+刷新倒计时+当前消耗),给用量面板用。日期 2026-06-20。
> 结论:这俩比 GLM(走 new-api 有现成 usage)复杂得多,得分层。下面是详细可落地方案。

## 一、难点(为什么复杂)
1. **官方查用量是交互命令**:Claude Code 用 `/usage`、Codex 用 `/status`,只能在**活跃交互会话**里跑,显示 5h 剩余+周+重置。但玉兔6 是 `claude -p` / `codex exec` **非交互**跑的,**抓不到这俩命令的输出**;也**没有非交互的官方用量 API/命令**。
2. **额度是共享的**:Claude Code 的 5h/周额度,和你个人的 **Claude.ai、Cowork 共享**——玉兔6 自己累计的 ≠ 你的全部消耗(你自己用也占额度)。
3. 所以"剩余额度+重置倒计时"这种**官方限额状态**,没有干净的程序化接口,得想办法。

## 二、三个数据源(分层)

### 源 A · 自累计(可靠、可程序化)——"玉兔6 自己用了多少"
- **Claude Code**:每个会话把调用记成 **JSONL 到本机 `~/.claude/`**(projects/会话日志),每条含 input/output token usage。解析它,按 **runner / agent / 时间**累计。
- **Codex**:同理,`~/.codex/`(会话/日志)累计。
- 产出:玉兔6 各 agent/模型的真实 token 消耗——**这正是面板要的"用在哪些智能体上、各花了多少"**,而且准确。

### 源 B · 官方限额状态(5h/周剩余 + 重置倒计时)——较难,三条路
1. **首选:定时 pty/expect 抓交互命令**。用伪终端(script/expect/node-pty)每 5–10 分钟自动起一个 `claude` 跑 `/usage`、起一个 `codex` 跑 `/status`,抓屏解析出"5h 剩余 / 周剩余 / 重置时间"。能拿到**官方真实**状态。
2. **备选:估算**。记录玉兔6 **首次调用时间**作 5h 窗口起点 + 已知套餐额度,推算剩余 + 倒计时(不依赖抓屏,但只覆盖玉兔6 部分,不含你个人消耗)。
3. **兜底:API 响应头**。若 `claude -p` / codex 能在 verbose/debug 模式暴露 `anthropic-ratelimit-*` 类响应头,从里面取限额(需验证 CLI 是否透传)。

### 源 C · GLM-5.2——已现成
走 new-api,usage 从 new-api 的 sqlite 直接拿(`/api/newapi/overview` 已能取 todayQuota/logs/pricing)。面板上显示为**「已付费·买断额度」**(不是计费扣钱)。

## 三、推荐方案
1. **自累计(源 A)做主力**:解析 `~/.claude`、`~/.codex` 日志,准确展示**各 agent/模型的 token 消耗**(面板核心)。
2. **限额状态(源 B)**:定时 pty 抓 `/usage`、`/status` 拿官方 5h/周剩余+重置倒计时;抓取失败用**估算**兜底。
3. **GLM(源 C)**:new-api 现成,显示已付费额度。
4. 面板**注明**:"Claude/Codex 额度与你个人 Claude.ai/Cowork 共享,此处为系统侧视角"。

## 四、落地步骤(交给 CEO→开发)
1. 本机确认 `~/.claude`、`~/.codex` 实际目录结构 + JSONL 字段,写解析器。
2. 写累计聚合(按 模型 / agent / 时间窗)。
3. 写 pty 定时抓 `/usage`、`/status` 的小工具 + 解析(带容错:CLI 版本变了不崩)。
4. 面板整合三源(GLM new-api + claude/codex 自累计 + 限额状态)。

## 五、风险/注意
- **pty 抓屏较 hacky**:CLI 版本/措辞变了可能失效 → 必须容错 + 估算兜底。
- **共享额度**:自累计只反映玉兔6 部分,面板要说明。
- **5h 窗口起点**:要准确记录玉兔6 首次调用时间。
- 这是"尽力而为"的可观测,不是官方精确接口——但**自累计那部分(各 agent 花了多少)是准的、够用的**,限额倒计时尽量逼近。
