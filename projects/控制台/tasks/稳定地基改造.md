# 稳定地基改造 · 设计规格(给 codex 落地)

> 流程:秘书出规格 → 老板转交 codex 落地 → 秘书验证。
> 目标文件:`projects/控制台/server.js`、`projects/控制台/ceo-worker.js`(必要时 `shared/engine/queue.js`)。零依赖。
> 出规格时间:2026-06-19。

## 0. 现状诊断(已读码确认)
**底子(保留):**
- `ceo-worker.js` 是通用队列 worker:`main()` → `acquireWorkerLock()`(每 agent 一个 `.worker.pid` 互斥)→ `sweepStaleRunning()` → **`while(true)` 常驻轮询** `Q.claim()`,无任务 `sleep(800)`。**不会空闲退出。**
- `server.js` 用 `ensureQueueWorker(agent)` 以 **`detached:true` + `child.unref()`** spawn worker → worker 不随 server 重启而死。
- 同一 agent:单队列 + 单 worker → 多 caller 入队天然**串行**。✅

**三个缺口:**
1. **无持续监督**:worker 若 crash(`main().catch`→`exit(1)`)或被杀,只有 `ensureWorkersForBacklog()`(**仅 server 启动时 line 383 调一次**)或 HTTP `/api/queue` 入队(`ensureQueueWorker`)才补拉。**直接写队列文件入队 + 当时无 worker → 永不被消费**(#22 卡住根因)。
2. **假并发**:全局单锁 `artifacts/engine-runner.lock.json`(`acquireEngineLock`,single-flight)。多 agent worker 能并行抢单,但执行引擎时抢同一把锁 → **全局串行**。
3. **重启误杀在跑任务**:`sweepStaleRunning()` 在 worker 启动时,把"running/ 有项但无活动 engine 锁"的任务直接 `Q.finish(...,'failed')`。server/worker 一重启,正跑的任务可能被误判失败。

## 1. 目标
1. **持续监督**:任何途径入队(含秘书直接写文件)、worker 万一挂,都能在 ≤10s 内被拉起消费。后台独立于网页稳定。
2. **多 agent 受控并发**:并发上限可配 `ENGINE_MAX_CONCURRENCY`(默认 **2**);同一 runner 类型(尤其 **claude** 登录态)串行保护,避免登录/配额/资源冲突。
3. **对状态修改/重启健壮**:在跑任务不因 server/worker 重启被误判失败 —— engine 子进程还活着就不碰,真没了**重新入队重试**(不直接判死)。
4. 同 agent 串行**保持不变**。

## 2. 改造点(只动 server.js / ceo-worker.js,必要时 queue.js;零依赖)

### A. 持续监督 —— server.js
- server 启动后增加常驻定时器:`setInterval(ensureWorkersForBacklog, SUPERVISE_MS)`,`SUPERVISE_MS = env.WORKER_SUPERVISE_MS || 10000`。
- `ensureWorkersForBacklog()` 已会遍历 `queues/` 下各 agent、对 `queued||running` 非空者 `ensureQueueWorker`;改为**定时反复**执行即可。
- 效果:秘书直接写文件入队后 ≤10s 被扫到拉起;worker 崩溃 ≤10s 被补拉。
- 备注:本期用 server 内定时器(改动最小)。后续可升级为**独立 watchdog 进程**(launchd 单独托管,与 server 彻底解耦)——列入后续,不在本期。

### B. 多 agent 受控并发 —— ceo-worker.js(替换全局单锁)
- 现 `acquireEngineLock(entry)` 为全局单锁。改为**并发槽 + 同类型串行保护**:
  - 槽目录 `artifacts/engine-slots/`,槽文件 `slot-<k>.json`(k=0..N-1),内容 `{ownerPid, queueId, runnerType, started_at}`。
  - `acquireEngineSlot(entry, runnerType)`:
    1. 清理:删除所有 `pid` 已死(`!pidAlive`)的槽文件(容灾)。
    2. 同类型串行:若 `runnerType ∈ SINGLEFLIGHT_RUNNERS`(env `RUNNER_SINGLEFLIGHT`,默认 `claude`),且已有**存活**槽的 `runnerType` 相同 → `sleep` 重试(保证该类型一次一个)。
    3. 抢槽:在 0..N-1 找一个不存在或已死的槽位,用 `writeFileSync(flag:'wx')` 原子占用;成功即获并发权,返回 release()(删自己的槽)。满则 `sleep(500~1000)` 重试。
  - `N = env.ENGINE_MAX_CONCURRENCY || 2`;`RUNNER_SINGLEFLIGHT = (env || 'claude').split(',')`。
  - `runnerType` 推断:由 spec.role → runner 映射(参照 cli-runner.js 的 roleMap;orchestrator/supervisor/worker_* → claude,codex 类 → codex…)。**至少对 claude 做 single-flight。**
  - 平滑兼容:`ENGINE_MAX_CONCURRENCY=1` 时退化为现状全局串行;旧 `engine-runner.lock.json` 可保留为兼容或迁移到槽机制(择一,注释说明)。

### C. 孤儿任务可恢复 + 清理孤儿进程(治内存)—— ceo-worker.js `sweepStaleRunning`
> 背景:实测 #22 在 review-loop 多轮迭代时遇 worker 重启,被本函数误判 `failed`("发现无活动 engine 锁的 running 项,已标记中断");同时旧 engine 子进程(codex)未被清理,孤儿堆积占大量内存。本节同治这两点。

worker 启动 sweep 改为:对每个 running 项:
- `cancel_requested` 的 → 按 canceled 收尾。
- 否则先**清理孤儿 engine 子进程**:若该项关联的旧 engine pid(槽/锁记录的 `enginePid`)`pidAlive`,`killProcessGroup(enginePid,'SIGTERM')`(5s 后 `SIGKILL`)。旧 worker 已死、无人收尾该 engine,留着只会占内存(codex 尤甚)/重复写结果。**这是治"孤儿 codex 堆积占内存"的关键。**
- 然后**不直接 failed,而是 `requeue` 重试**:移回 `queued`(保留 payload + `retry=(retry||0)+1`),`emit('queue.recovered',{queueId,retry})`;仅当 `retry > QUEUE_MAX_RETRY`(env,默认 2)才 `finish(...,'failed')`。
- 需要 `queue.js` 增 `requeue(root, agent, id, patch)` 辅助(running/<id>.json 移回队列目录、status=queued、合并 patch)。
- 正常运行中在跑任务只认 `cancel_requested` 中断;其它队列状态文件变动不影响在跑 engine。
- 双保险:`acquireEngineSlot` 清理槽时,若某槽 owner worker 已死但其 `enginePid` 仍存活,一并 `kill` 该孤儿 engine,防堆积。

### D. 健壮性杂项
- 文件读写全程 try/catch(现状已大量,保持)。
- worker spawn 继承 server env(登录态/proxy),保持不变。
- 强调并注释:worker + engine 是 detached 独立进程,网页仅只读轮询 `/api/events`、`/api/queue`;**网页不更新/关闭不影响后台执行**。

## 4. 自测(落地后必跑,把结果回报)
1. **worker 自愈**:`kill` 掉 ceo worker 进程,≤15s 内 server 定时器自动补拉(`queue.worker.spawned` 事件 + `.worker.pid` 复活)。
2. **直接写文件即消费**:无 worker 状态下直接往 `queues/ceo/` 写一个任务文件,≤15s 被 claim 执行。
3. **多 agent 并发**:`ENGINE_MAX_CONCURRENCY=2`,同时入队 ceo + governance 各一个任务,engine-events 时间**重叠**(并发);设 `=1` 则串行。
4. **同类型串行保护**:同时入队两个 ceo(claude)任务,二者仍**串行**(不并发,保护登录态)。
5. **重启不丢任务**:任务执行中重启 server,在跑任务**不被误判 failed**,而是恢复/重试,最终状态正确。
6. **回归**:`git diff` 仅限上述文件;review-loop、队列多面板、双视图(若已合)均不回退。
