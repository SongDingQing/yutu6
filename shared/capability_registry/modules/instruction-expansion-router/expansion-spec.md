# 补齐规范(expansion-spec)

> 前门总管(Codex CLI)收到人的指令后,**派单前**按此把它补成完整 task 信封。
> 目标:让下游 agent 拿到的永远是"目标+边界+输入+验收"齐全、可独立执行的信封。

## 输入形态(都要能接)
- **口语 / 简略**:"把那个修一下""整理下""做完发我"。
- **带截图 / 文件**:用户附了图或文件 → 把路径写进信封"输入",并指明"必须对照该证据"。
- **跨项目 / 含代号**:出现 Simulaid / YuanXiao / 元宵 / 玉系列代号 → 先走对应 expander(优先级 2–3)。
- **多义 / 缺信息**:不臆测 —— 列进"待澄清",必要时先问一句再派。

## 补齐步骤
1. **定意图 + 选来源**:按 INDEX 的 4 级优先级,选显式点名 > 项目 expander > 跨项目 wrapper > 全局兜底。命中即停在该级。
2. **填信封**(§4 模板,缺项不要留空,写"无"或"待澄清"):
   ```
   # 任务:<一句话标题>
   - 目标:<要什么,可验证>
   - 边界:<明确"不用管 X、不要碰 Y",并写密钥、授权和高危操作边界>
   - 输入:<逐条列要读的文件/截图路径>
   - 验收:<怎样算做完;带视觉产物必须"渲染自查+对照用户截图">
   - 截止:<有就写,没有写"无">
   ```
3. **标注**:用了哪条补齐来源、置信度(高/中/低)、待澄清项。
4. **派单**:把信封交给目标 runner/agent(经 routing 选择)。

## 硬规矩
- **每次任务最多一个可见"补齐稿"块**;项目 expander 已出,全局 router 不再重复(避免串味,§3)。
- **不臆测关键信息**:缺目标/输入/验收且无法合理推断 → 先问,别硬派。
- 带视觉/带证据的任务,验收必须包含"对照用户提供的证据"(§17 监管硬门)。
- 不在信封里塞密钥值;需要密钥的步骤只引用保险库位置(见 `shared/config/environment.md`)。

## 例:口语 → 信封
输入(用户对前门):"控制台那个 codex 路由还没通,你看下"
补齐稿(全局 router,置信度高):
```
# 任务:排查控制台 codex runner 未通
- 目标:让 codex 真对话在控制台返回真实回复(核心 runner 全绿)
- 边界:只动 config.json 的 codex cmd / server.js 最小改动;不接飞书;密钥不回显
- 输入:projects/控制台/{config.json,server.js,artifacts/console-verify.txt}
- 验收:POST /api/chat runner=codex 返回真实答案、done code 0;结果写进 console-verify.txt
- 截止:无
```
