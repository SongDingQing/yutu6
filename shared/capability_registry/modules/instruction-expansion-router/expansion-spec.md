# 补齐规范

## 输入

- 口语或简略请求
- 截图、文件或链接
- 已知项目 ID
- 可能有歧义或缺少关键输入的任务

## 步骤

1. 确定意图、项目和可用能力来源。
2. 生成任务信封：
   ```text
   # 任务: <标题>
   - 目标: <可验证结果>
   - 项目: <projectId 或 system>
   - 边界: <不应修改的范围>
   - 输入: <文件、截图、接口或无>
   - 验收: <命令、证据和通过标准>
   - 风险与确认: <需要主人确认的操作或无>
   ```
3. 标注补齐来源、置信度和待澄清项。
4. 秘书将信封交给 CEO；CEO 决定系统部门或 `supervisor-<projectId>` 路线。

## 硬规则

- 不臆测账号、密钥、业务规则或不可逆操作授权。
- 视觉任务必须对照用户证据做渲染验证。
- 信封只引用 KEY 名和本地私有配置位置，不包含值。
- 项目专用命令只能来自已安装并登记的项目能力包。
- 同一任务不重复生成补齐稿。
