## 一、模型动态切换(三层,声明式、零 token 路由)

源:`shared/routing/model-routing.yaml`(声明)+ `shared/routing/failover.js` + `shared/engine/cli-runner.js`(执行)。

### 层1 · 分角色选模型(已有)
`roles.<role>.prefer = [候选链]`,判断层用强模型(Opus/Codex),执行层用快省模型(GLM/Haiku)。
编排层只读表选 provider/模型,本身不烧 token。

### 层2 · 失败转移 failover(✅ 已落地 v0.0.3.5)
首选 runner 失败(非0/超时/信号 → 含 429/5xx/不可用)时,按 prefer 顺序**降级到下一候选**重试,而非整节点失败触发维修循环。
- 首选 = 现 roleMap runner,**行为不变**;prefer 候选只在失败后降级。
- provider→runnerId 映射:`subscription.codex→codex`、`api.zhipu→zhipu-glm`、`subscription.claude→claude`、`api.deepseek→new-api`;无本地 runner 的(openai/ollama/anthropic)跳过。
- 真实链路:worker_code `codex→zhipu-glm`;supervisor `claude→codex`;quality_ops `zhipu-glm→claude→codex`。
- 事件:`runner.failover{from,to,reason}`;开关 `YUTU6_RUNNER_FAILOVER=0` 可关。

### 层3 · 预测性切换 predictive(⬜ 未实现,设计)
`failover.predictive: true` + 触发器 `quota_low`:把"剩余限流余量"当路由信号,**耗尽前提前切**,不被动 fallback。
- 缺口:目前没有"配额余量"实时信号源。需要 runner 暴露/估算余量(如 newapi gateway_quota、订阅额度倒计时——见任务10)。
- 落点:`failoverCandidates()` 里加"primary 余量低于阈值 → 把 primary 降到候选末尾、首选次优"的预判分支。
- 依赖任务10(模型用量 plan% + 刷新倒计时)先把余量信号接进来。

### 动态切换的"开关"在哪
- 静态调整:改 `model-routing.yaml` 的 prefer 顺序(声明式,改完即生效,无需改码)。
- 运行时:`YUTU6_RUNNER_FAILOVER` 总开关;每角色 `config.json roleRouting.<role>.runner` 定首选。

## 二、知识库能含多少"工作准则"

### 两套系统,别混
- **memory/**(精炼长期记忆):人审过的稳定准则/偏好/项目约束,**进 context 每会话加载**(MEMORY.md 索引)。容量受 context 预算硬约束 → 只能放**最关键**的几十条。
- **knowledge/**(RAG,`kb.sqlite` via `ingest.py`/`query.py`):海量语料,**按需检索**,不占常驻 context。容量近乎无限(磁盘级),但**当前是纸面工具**(见缺口)。

### 缺口(任务7,需专门一轮)
`query.py` 在 agent 的 prompt/envelope 里**零引用** → 智能体从没被告知能查知识库、也没工具接到它。知识库现在=只进不出。

### "岗位宪章 + RAG 顶岗"设计(建议)
1. **岗位宪章**(放 memory/ 或 config,常驻):每角色一份精炼准则(≤1 屏),进每次该角色的 envelope。这是"必须遵守"的硬准则。
2. **RAG 顶岗**(放 knowledge/,按需):大量案例/历史/细则,**运行前按任务 goal 检索 top-K 注入 envelope**(检索增强),或给 tool-harness 角色一个 query 工具。这是"参考资料"。
3. 工作准则归属判定:**会反复用、要硬遵守 → 宪章(memory);量大、偶尔参考 → RAG(knowledge)**。

落地顺序:先把 query.py 打通(node 子进程调用 + envelope 注入 top-K),再分角色写岗位宪章。embed provider 需先确认可用(API/本地)。
