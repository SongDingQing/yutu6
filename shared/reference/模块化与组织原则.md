# 玉兔6 · 模块化与组织原则(所有 agent 必读)

> 用户定调(2026-06-18):整个玉兔6 架构要**清晰、模块化、可拆解**;**减少过度繁复逻辑**;优化智能体处理**性能**、降低**出错概率**。磁盘大,但仍要**有逻辑地用**(冷数据归档,别堆乱)。
> 本文件是组织真相;新建/移动文件前先对照它。

## 1. 五条设计原则
1. **模块化可拆解**:每个能力 = 一个可独立理解、可单独替换的模块,有清晰边界与 I/O 契约。任何模块都应能被拎出来单测/替换,不牵一发动全身。
2. **核心与项目分离**:**智能体架构核心**(引擎/路由/能力库/知识库)归 `shared/`,**与任何具体项目无关**;**项目代码**各自归 `projects/<名>/`,彼此独立。核心被项目消费,核心不依赖项目。
3. **少繁复逻辑**:能声明式就别命令式(`shared/routing` 写死固定流程,引擎零 token 走);能确定性就别让 LLM 现想;能硬化成脚本/软骨就硬化(§16)。复杂度是出错与耗时的根源。
4. **性能 / 少出错**:need-to-know 上下文从严(只喂依赖)、稳定前缀吃 KV-cache、attempt+事件日志可回溯、跑前 dry-run 校验、验收证据门。**先验证再交付**。
5. **磁盘有逻辑地用**:热数据在工作区可读位置;冷数据(历史快照、迁移记录、旧资料)进 `knowledge/corpus/` 或归档区;大二进制/资产不进 git(`.gitignore`)。空间紧张时按"冷→先归档/冷备到玉兔2"。

## 2. 目录真相(谁归哪)
```
玉兔6工作区/
├── board/                      # 老板层:direction / status-rollup / decisions / 搭建路线图
├── shared/                     # ★ 智能体架构核心(与项目无关、可拆解模块)
│   ├── engine/                 #   编排引擎(声明式路由+护栏+事件日志+信封运行时)← 核心基建
│   ├── routing/                #   声明式控制平面:model-routing / runners / flows
│   ├── agents/                 #   系统级智能体定义(总管/主管/质量运营/监管)← 阶段3
│   ├── capability_registry/    #   能力目录 + 模块(含 instruction-expansion-router)
│   ├── config/                 #   环境配置(environment.md / machine.json / ssh-and-remotes)
│   └── reference/              #   蓝图、监管误区库、本组织原则
├── projects/                   # ★ 各项目代码,彼此独立
│   ├── 控制台/                 #   系统 WebUI(消费 shared/engine)
│   └── <projectId>/            #   由项目部门生成器创建的业务项目
├── knowledge/                  # 知识库(kb.sqlite + ingest/query)+ corpus 冷档
│   └── 归档/                   #   ★ 答复/分析文档归档(见 §3)
├── wiki/                       # 原文真相(人可读 markdown)
└── project-packs/              # 可选项目能力包约定
```
> 核心规矩:**项目代码不入 shared/;核心基建不入 projects/。** 引擎已从 `projects/控制台/engine` 提升到 `shared/engine`,即此原则的落地。

## 3. 答复 / 分析文档归档约定
总管写给老板的**实质性分析/答复/方案**(不是随手对话)要存成文件,别只留在聊天里。归位规则:
- **属于某项目**的分析 → `projects/<projectId>/artifacts/`(如官网引擎选型 → `projects/website/artifacts/引擎选型分析-*.md`)。
- **跨项目/架构级**的分析、答复、决策背景 → `knowledge/归档/`(按日期+主题命名)。
- **方向/趋势/路线图** → `board/`。
- 命名带日期(`-YYYY-MM`)便于追溯;过期版本不删、标注 superseded。

## 4. 新建模块的最低要求(可拆解的保证)
每个 `shared/` 下的模块至少有:① 一句话职责;② I/O 契约(输入什么、输出什么);③ 与谁解耦(不依赖谁);④ 可单独验证的方式(测试/冒烟/dry-run)。达不到就不算"模块",是"耦合代码"。
