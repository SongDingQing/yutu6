# 多智能体架构设计

> 一套「多智能体 · 多模型 · 随意切换」的本地交互系统设计。
> 核心思想:**把上下文隔离当成设计的出发点,而不是缺陷。**

---

## 0. 一句话概括

每个智能体(agent)是一段**独立的上下文**,彼此不共享记忆,甚至可能来自不同厂商(Claude / OpenAI / 本地模型)。
它们唯一共享的是**文件系统**。所以:

- **文件 = 共享大脑**。沟通和状态都落在文件里,不靠对话记忆。
- **信息向上走 → 逐层压缩**(只传摘要 + 文件路径,不传正文)。于是领导天然只看到趋势。
- **信息向下/横向走 → 按需拉取**(缺了才问,问到才知道)。于是员工天然只看到自己那一摊。

这正好对应你的两句话:

> 「我是领导,不需要知道项目细节,我只知道趋势。」
> 「我是员工,不需要知道别的部门,只在需要时找领导问。」

## 1. 组织结构:四层

> **📎 借鉴(2026-06-15 · ACE,未核实)**:Reflector/Curator 角色**采纳但合并** —— 总管 / 主管兼任 **Curator 职责**,以增量 delta 维护趋势摘要层(不新增独立角色,负载高再独立)。详见 `情报借鉴-2026-06-15.md`。

| 层级 | 角色 | 看得到什么 | 负责什么 | 建议模型 |
|------|------|-----------|---------|---------|
| 老板 | **你(人)** | 只读 `board/`(趋势 + 决策) | 下方向、看趋势、拍板 | — |
| 领导 | **总管 Orchestrator** | `board/` + 各项目的 `brief/status` | 把方向拆成项目、派给主管;把各项目进展摘成趋势回报你 | 强模型(Opus 档) |
| 中层 | **主管(一个项目一个)** | 只看**自己项目**的全部 + 产物 | 把项目拆成任务、派给员工、审产物、把结果摘成项目进展上报 | 强模型 |
| 执行 | **员工** | 只看**自己任务**的文件 + 指定输入 | 做一件具体的事;缺跨域信息时写进 `questions.md` 向主管问 | 快/省模型(Codex 写码、Haiku/本地跑窄任务) |

关键点:**没有谁能看到全局**。每一层只拿到「它被允许读的路径」,而不是别人的上下文内容。

## 2. 目录结构 = 把「谁该知道什么」写进文件夹

```
玉兔6工作区/
├── board/                    # 你 ↔ 总管:只放趋势与决策
│   ├── direction.md          #   你下的方向、目标、约束
│   ├── status-rollup.md      #   总管给你的趋势汇总(无细节)
│   └── decisions.md          #   待你拍板 + 已决议(追加日志)
├── projects/
│   └── <项目名>/
│       ├── brief.md          # 总管→主管:范围 + 验收标准
│       ├── status.md         # 主管→总管:进展摘要(滚动覆盖)
│       └── tasks/
│           └── <任务名>/
│               ├── task.md       # 主管→员工:一件事 + 输入路径 + 边界
│               ├── result.md     # 员工→主管:产物摘要 + 产物路径
│               └── questions.md  # 员工→主管:需要时才问的跨域问题
│       └── artifacts/        # 实际产物(代码/数据/文档)—— 共享文件
└── shared/
    ├── reference/            # 全局只读资料(规范、API 文档等)
    └── glossary.md           # 统一术语,防止各 agent 自说自话
```

> 产物(代码、数据)放在 `artifacts/`,是真正「跨模型共享的文件」。
> 而 `task.md` / `result.md` / `status.md` 是**信息在层与层之间流动的信封**。

## 3. 上下文管理:三条规则

> **📎 借鉴(2026-06-15 · ACE + MS Conductor,未核实)**
> - **ACE ✅ 增量更新**:下方规则二升级为「**增量追加 / 局部修订**」,**严禁整篇重写** `status` / `status-rollup`(防摘要漂移、丢信息)。
> - **Conductor ✅ 三种上下文模式**:`explicit`(**默认**,= need-to-know,员工只看依赖)/ `accumulate`(仅总管级)/ `last_only`(流水线)。
> 详见 `情报借鉴-2026-06-15.md`。

### 规则一 · 读取范围按层封顶(强制 need-to-know)
给每个 agent 的不是内容,而是**它被允许读的路径白名单**:

- 你 → 只读 `board/`
- 总管 → `board/` + 各项目 `brief.md` / `status.md`(**不读**具体任务)
- 主管 → 自己项目的全部 + `artifacts/`(**不读**别的项目)
- 员工 → 自己的 `task.md` + 其中点名的输入文件 + `shared/reference/`

### 规则二 · 过界就压缩(让上层永远停在趋势)
信息穿过层级边界时**只传摘要和指针,不传正文**:

- **派单(向下)** = 目标 + 边界(明确写「不用管什么」) + 输入文件路径 + 验收标准 + 截止
- **汇报(向上)** = 做了什么(1–3 行) + 产物路径 + 待解问题 + 风险标记

这就是为什么领导只看到趋势:细节在每一层被摘要掉了,只剩结论和指针往上走。

### 规则三 · 无状态 agent + 有状态文件
每个 agent 的生命周期是:**醒来 → 读自己的文件 → 干活 → 写结果 → 退出**。
记忆全在文件里,不依赖对话历史。配合 git:

- 一个任务一次提交 → 可回滚、可审 diff
- **单写主原则**:一个文件只由一个任务负责写;员工不改别人的文件,跨文件改动上交主管协调,避免互相覆盖。

## 4. 信封模板(协议)

**派单 `task.md`**

```
# 任务:<标题>
- 目标:<一句话说清要什么>
- 边界:<明确写"不用管 X、不要碰 Y">
- 输入:<需要读的文件路径,逐条列出>
- 验收:<怎样算做完>
- 截止:<时间>
```

**汇报 `result.md`**

```
# 结果:<标题>
- 做了什么:<1–3 行>
- 产物在哪:<文件路径>
- 待解问题:<有就写,没有写"无">
- 风险:<低/中/高 + 一句原因>
```

**提问 `questions.md`**(员工→主管,默认隔离、按需才问)

```
[待答] <我缺的跨域信息是什么,为什么需要>
[已答] <主管回填的答复>
```

## 5. 多模型如何接上

> **📎 借鉴(2026-06-15 · MS Conductor + LiteLLM/Ollama,未核实)**
> - **逐 agent 选模型/厂商 + 零 token 确定性路由**(Conductor):路由表写死、可预测;判断层强模型、执行层快/省模型。
> - **LiteLLM 统一出口 + fallback 链**(订阅 → API → Ollama 本地兜底);LiteLLM 作**可选**网关、不绑死(避免单点)。
> 详见 `情报借鉴-2026-06-15.md`。

因为上下文本就隔离,**跨厂商混用是零成本**的:主管(Claude)写 `task.md`,员工(Codex)读它就开工——文件协议就是它们之间的「普通话」。

- 判断、压缩、消歧 → 强模型(总管、主管)
- 写代码 → Codex
- 跑窄任务 / 省钱 / 离线 → Haiku 或本地模型(Ollama)

「随意切换」在实现上 = 一个本地网页当统一控制台,前面是聊天界面,后面按角色把消息路由给不同模型。

# 第二部分:生态架构与借鉴融合(v2)

> 玉兔6 已从"单机多智能体"扩展为**可远程审计的 agent control room**:
> 本机工作区与 runner = 执行核心,可选远端服务 = 中继,网页或移动客户端 = 控制面。
> 本部分固化控制平面、接入同步、可靠性,以及对**璇玑 / OpenClaw / Hermes** 的借鉴。

## 8. 控制平面(运行时编排)

> **📎 借鉴(2026-06-15 · Pydantic AI Harness + MS Agent Framework + Conductor,未核实)**
> - **队列拆两种语义(Pydantic AI Harness ✅,取代下方笼统的"命令队列 + 中断")**:
>   - **steering**(打断式)— 工具执行后 / 下次 LLM 调用前送达,用于中途转向;
>   - **follow-up**(排队式)— 当前任务干完才送达,用于顺序喂任务;
>   - 主循环两固定消费点:每轮工具执行后查 steering;无活可干时查 follow-up;默认 `one-at-a-time`,`all` 可选。
> - **配额故障转移**落为 **LiteLLM + fallback 链**(订阅 → API → Ollama),见 §5。
> - **Handoff 移交 + 受限并行 BackgroundAgents**(MS Agent Framework,⚠️ 受调度器/配额上限约束);**human gate + 并行失败策略**(Conductor)纳入 §10。
> 详见 `情报借鉴-2026-06-15.md`。

把"排队、打断、引导、路由"抽成**独立于模型**的一层:

- **命令队列**:所有指令进 FIFO 队列(带幂等键),上一个完成自动喂下一个。
- **打断 / 引导**:中断信号在"流式 chunk 之间、工具调用之间"检查,命中即停在安全检查点,把"进度摘要 + 新指令"插队续跑。
- **配额感知故障转移**:某模型 429 / 额度满 → 轻量层同档换 provider(无感),不行则上升到主管重新分配;额度恢复后回切。
- **三路兜底**:订阅(吃性价比)→ API(稳、可预测)→ 本地 5090(离线 / 降本)。
- 队列与中断状态都落事件日志,可暂停恢复。

## 9. 接入与同步层:客户端 ↔ 可选中继 ↔ 本机工作区

三段式,**真相分层**:

- **本机工作区**:跑 agent、持有文件工作区,是执行产物的真相源。
- **可选远端中继**:中转消息、文件和看板镜像;不跑 agent,也不取代本机执行真相。
- **网页或移动客户端**:瘦控制面,负责发命令、传文件和查看状态。

**同步三原则**:① 命令式(端发命令进队列,不直接改状态);② append-only 事件日志 + cursor 增量(断线重连不丢不重);③ 不用时间戳排序,用单调 version / event offset。

## 10. 可靠性与控制面(control room)

目标:每个 agent 的权限、状态、队列、产物、失败原因、验收证据、下一步,都**看得清、控得住、追得回**。

- **任务状态机 + attempt**:queued→assigned→running→awaiting_verify→done / failed / paused / cancelled;任务与"运行尝试 attempt"分离(重试 / 换 runner / 兜底各算一次 attempt,失败原因挂 attempt)。
- **验收证据**:每个 done 挂测试输出 / 截图 / diff —— 回答"凭什么算完成"。
- **Artifact ledger**:产物内容寻址 + 不可变 + 溯源(hash / 大小 / 类型 / 产出 task);大文件断点续传 + 双重 hash。
- **断线恢复**:客户端、中继和 runner 链路各自使用 cursor 增量;离线命令本地排队、上线消费;控制面显式显示链路健康。
- **Codex = best-effort runner**:以 bridge I/O 为权威,Codex 内部历史仅参考;可脚本化 CLI / API runner(Hermes、Claude Code)作一等公民,GUI 作二等。
- **移动端 IA**:Agents / Tasks / Artifacts / Memory / Approvals / Audit 视图,默认老板视图、按需下钻。

## 15. 本地知识库子系统(wiki + RAG + 图谱)

把第 11 章的"知识图谱长期记忆"落成**可跑的本地子系统**。核心理念:**wiki 原文 / RAG 向量 / 图谱关系 是同一份真相的三种索引**,真相是工作区里的 Markdown。嵌入式优先 —— 单文件、零运维、跟工作区 / git / 冷备一起走。

### 15.1 选型表

| 用途 | 起步(嵌入式,推荐) | 规模 / 多端时升级 |
|---|---|---|
| 主库 / 关系 / 任务 ledger | **SQLite** | PostgreSQL |
| 全文检索 | **SQLite FTS5** | PG tsvector / Meilisearch |
| 向量(RAG) | **sqlite-vec**(或 LanceDB) | pgvector / Qdrant |
| 图谱 | **SQLite 表 + 递归 CTE** | KùzuDB / Neo4j |
| Embedding 模型 | 本地 **Ollama**(nomic-embed-text·768 维 / bge-m3),或丢给 5090 | 云 embedding API |
| 图谱可视化 | **Cytoscape** | Cytoscape |

> 起步阶段几乎全在一个 `kb.sqlite` 文件里(关系 + 全文 + 向量 + 图谱),零常驻服务;等需要远端或移动客户端共享同一份知识,再升级常驻库或由本机 bridge 包成 API。

### 15.2 数据流

**写入(ingestion · 文件变更触发)**

1. 扫 `wiki/` 下 Markdown,按 `path + hash` 做**增量**检测(只处理变更文件)。
2. 变更文件 → 按标题 / 段落**切块**(约 300–600 token)。
3. 每块 → 本地 embedding → 写 `vec_chunks`;同时写 `chunks` + 触发 FTS5。
4. 每块 → LLM 抽「实体-关系-事件」→ upsert `entities / relations / events`,并写 `mentions`(出处指回 chunk)。
5. 文件被删 / 改 → 级联清理旧块、旧边。

**检索(GraphRAG)**

1. 问题 → query embedding。
2. **混合召回**:向量 top-k(`vec_chunks`)+ 全文 top-k(`chunks_fts`),融合去重。
3. 从命中块的 `mentions` 找**种子实体** → 沿 `relations` 扩展 1–2 跳(跳过 `status='deprecated'`),拉相关实体 / 事件及其出处块。
4. 汇总上下文(原文片段 + 图谱邻域)**带出处**喂 LLM 生成答案。
5. 用户纠正 → 把对应 `relations.status` 置 `deprecated` 并记 `corrections`,下次自动避开(璇玑式永久修正)。

### 15.3 最小起步方案

**目录结构**

```
玉兔6工作区/
├── wiki/                     # 原文真相:人可读可编辑的 Markdown
│   ├── people/example-user.md
│   ├── projects/example-project.md
│   └── ...
└── knowledge/
    ├── kb.sqlite             # 单文件库:关系+FTS5+向量+图谱
    ├── schema.sql            # 建表 DDL(下方)
    ├── ingest.py             # 写入管道(扫 wiki→切块→embedding→抽实体→入库)
    └── query.py              # 检索(向量+全文+图谱→合并→回原文)
```

**SQLite 表结构(`schema.sql`)**

```sql
-- 需加载 sqlite-vec 扩展;FTS5 为 SQLite 内置
PRAGMA foreign_keys = ON;

-- 原文档(对应 wiki/ 下的 markdown)
CREATE TABLE documents (
  id    INTEGER PRIMARY KEY,
  path  TEXT UNIQUE NOT NULL,         -- 相对 wiki/ 的路径
  title TEXT,
  hash  TEXT,                         -- 内容哈希,增量更新用
  mtime REAL
);

-- 切块
CREATE TABLE chunks (
  id     INTEGER PRIMARY KEY,
  doc_id INTEGER NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
  ord    INTEGER,
  text   TEXT NOT NULL
);
CREATE INDEX idx_chunks_doc ON chunks(doc_id);

-- 全文索引(FTS5,external content 关联 chunks)
CREATE VIRTUAL TABLE chunks_fts USING fts5(
  text, content='chunks', content_rowid='id', tokenize='unicode61'
);
CREATE TRIGGER chunks_ai AFTER INSERT ON chunks BEGIN
  INSERT INTO chunks_fts(rowid, text) VALUES (new.id, new.text);
END;
CREATE TRIGGER chunks_ad AFTER DELETE ON chunks BEGIN
  INSERT INTO chunks_fts(chunks_fts, rowid, text) VALUES('delete', old.id, old.text);
END;

-- 向量索引(维度按 embedding 模型,nomic-embed-text = 768)
CREATE VIRTUAL TABLE vec_chunks USING vec0(
  chunk_id  INTEGER PRIMARY KEY,
  embedding FLOAT[768]
);

-- 图谱:实体 / 关系 / 事件
CREATE TABLE entities (
  id      INTEGER PRIMARY KEY,
  name    TEXT NOT NULL,
  type    TEXT,                       -- person/project/tool/concept...
  summary TEXT,
  UNIQUE(name, type)
);
CREATE TABLE relations (
  id       INTEGER PRIMARY KEY,
  src      INTEGER NOT NULL REFERENCES entities(id),
  dst      INTEGER NOT NULL REFERENCES entities(id),
  type     TEXT,                      -- 偏好/属于/去过/否决...
  weight   REAL DEFAULT 1.0,
  status   TEXT DEFAULT 'active',     -- active / deprecated(被用户否决)
  evidence INTEGER REFERENCES chunks(id)
);
CREATE INDEX idx_rel_src ON relations(src);
CREATE INDEX idx_rel_dst ON relations(dst);
CREATE TABLE events (
  id       INTEGER PRIMARY KEY,
  summary  TEXT,
  ts       REAL,
  evidence INTEGER REFERENCES chunks(id)
);

-- 实体↔块 提及(连接图谱与原文)
CREATE TABLE mentions (
  entity_id INTEGER REFERENCES entities(id),
  chunk_id  INTEGER REFERENCES chunks(id),
  PRIMARY KEY(entity_id, chunk_id)
);

-- 用户反馈修正(璇玑式)
CREATE TABLE corrections (
  id              INTEGER PRIMARY KEY,
  target_relation INTEGER REFERENCES relations(id),
  note            TEXT,
  created_at      REAL DEFAULT (julianday('now'))
);
```

**检索流程(`query.py` 的核心步骤 + 关键 SQL)**

```sql
-- ① 向量召回(sqlite-vec KNN;:q 为问题向量)
SELECT chunk_id, distance
FROM vec_chunks
WHERE embedding MATCH :q AND k = 8;

-- ② 全文召回(:kw 为关键词/问题)
SELECT rowid AS chunk_id
FROM chunks_fts
WHERE chunks_fts MATCH :kw
LIMIT 8;

-- ③ 命中块 → 种子实体
SELECT DISTINCT entity_id
FROM mentions
WHERE chunk_id IN (/* ①②融合后的 chunk_id 列表 */);

-- ④ 沿关系图扩展 1–2 跳(跳过被否决的边)
WITH RECURSIVE graph(eid, depth) AS (
  SELECT :seed_entity, 0
  UNION
  SELECT CASE WHEN r.src = g.eid THEN r.dst ELSE r.src END, g.depth + 1
  FROM relations r JOIN graph g
    ON (r.src = g.eid OR r.dst = g.eid)
  WHERE g.depth < 2 AND r.status = 'active'
)
SELECT DISTINCT eid FROM graph;

-- ⑤ 取邻域实体的出处块,回到原文给 LLM(带 documents.path 做引用)
```

合并 ①② 用加权或 RRF 去重;把 ④⑤ 的图谱邻域 + 原文片段拼成上下文,连同 `documents.path` 出处一起喂 LLM 生成答案。用户纠正 → `UPDATE relations SET status='deprecated'` + 记 `corrections`。

### 15.4 注意

- **换 embedding 模型 = 重建向量索引**,维度也要改(`FLOAT[768]`),选型尽量一次定。
- 图谱抽取靠 LLM,有噪声 → `corrections` + `status` 软删除是必备的纠错通道。
- 多端时 DB 真相在 mini,云 / 手机走 bridge API(见 §9)。
- `kb.sqlite` 纳入快照 + 冷备到玉兔2;`wiki/` 本身进 git。

## 16. 硬化(Hardening):把流程从"柔"压成"硬"

> **「硬化」是玉兔6 的核心省力机制**,由「**质量运营工程师**」执行(分**系统级 1 个**横切全局 + **项目级 N 个**各盯一个大项目,见下)。它把 agent 的工作沿一条**从柔到硬的连续谱**逐步压实:
>
> **LLM 实时推理(最柔,临场思考)→ + Skills / 记忆(像笔记 + 记忆,仍需主动调用与思考)→ 硬化 → 肌肉记忆 / 软骨 / 骨头(越来越硬、越来越省)。**
>
> 直觉:有些东西现在要 LLM **一直"绷着"**(每次都耗 token 去维持和推理);硬化就是把这些**一直绷紧的肌肉变成骨头** —— 从"持续耗能维持"变成"结构性存在",结实、可靠、零 token。
>
> 来源:用户提出的"硬化"框架(2026-06-15)+ ACE Reflector/Curator 思路。与 §1 的 Curator 区分:Curator 炼"知识 / 记忆"(笔记层),质量运营工程师炼"流程 / 动作"(肌肉→骨头层)。

### 双层配置:系统级 1 个 + 项目级 N 个

- **项目级质量运营工程师(硬化本项目)**:精力全在硬化自己这个项目 —— 读本项目**全部交互过程**、定期分析,把项目内的重复 / 确定性流程硬化成骨头 / 软骨 / 肌肉记忆。
  - **配备条件(成本门槛)**:不是每个项目都配;只有项目 **agent 数 ≥ 阈值** 或 **token 消耗 ≥ 阈值** 时才配 —— 它要吃大量上下文、本身有成本,小项目不划算。
  - **工作方式**:有一套**标准工作方式(SOP)**,但具体判断要靠它对**大量项目上下文**的理解,所以是个"重"角色、按需启用。
- **系统级质量运营工程师(维护基础路由)**:**全局唯一**,横切所有项目;不盯单项目细节,而是接收各项目级上报的**跨项目共性优化**,验证后**升级基础路由 / 通用硬化库**,把共性能力下沉为平台级,惠及所有项目。
- **上报 → 升级回路**:某项目级硬化出一个优化,若发现**多个项目级在做同一件事**(共性),就上报系统级;系统级把它从"各项目各自硬化"升级成"基础路由里的一个标准模块" —— **一次下沉、处处受益**。这正是 §1 need-to-know + 上报的延续。

### 三种硬化形态(按硬度与灵活度)

| 形态 | 是什么 | 特性 | 例子 |
|---|---|---|---|
| 🦴 **骨头** | 完全固定的脚本 / 确定流程 | 最硬、零 token、最可靠 | "先 X 再 Y"的固定两步 → 一个两行脚本 |
| 🦵 **软骨** | 可配置 / 可路由的模块(参数化 skill、带分支) | 硬但可变通,可重新变软 | 按参数走不同分支的 skill / 路由表 |
| 💪 **肌肉记忆** | **不写成脚本,而是重设计流程让某段思考变得不必要**(其他部分已替它完成) | 卸载认知,而非固化动作 | 状态机已明确阶段 → agent 不必再"判断现在第几步" |

> 关键区别:骨头和软骨是**把动作固化**;肌肉记忆是**消除思考的必要性**——"我不需要这里的某些思考,因为流程其他部分会替我完成"。三者都省 token,但路径不同。

### 触发 / 输入
- 主触发:**项目架构更改**时(或工作流稳定 / 里程碑后);也可周期性或手动。增量分析,不全量(它自己也耗算力)。
- 输入:§10 事件日志 / audit、任务 ledger、各 agent 的 `task.md` / `result.md` 与工具调用序列。

### 质量运营工程师做什么
1. 找**高重复 + 确定性**的动作序列 → 硬化成 🦴 骨头(脚本)或 🦵 软骨(参数化 skill,进 capability_registry §11)。
2. 找**本可消除的思考**(被状态 / 上下文 / 其他模块保证的判断)→ 重设计流程做成 💪 肌肉记忆,把这段推理卸载掉。
3. 产出**硬化建议清单**:候选形态 + 命中次数 + 预计 token 节省 + 风险;经 human gate 批准后落地并更新工作流。

### 边界(别硬化过头)
- **只硬化"高重复 + 确定性 / 可被保证"的**;需要判断、易变的留给 LLM —— 别把灵活性也硬化掉(骨头太多会僵)。
- **门槛触发**:同一模式出现 ≥ N 次才硬化,避免过早优化。
- **可回退 + 可观测**:硬化路径失败要能 fallback 回 LLM;骨头 / 软骨要带测试与验收证据(§10),软骨保留"重新变软"的余地。
- **human gate**:改工作流 / 高危硬化需人批准(§10)。

### 收益
token ↓、延迟 ↓、确定性更强、审计更清晰 —— **越用越硬、越用越省、越用越快**。这是玉兔6 把"经验"沉淀成"结构"的核心机制。

> **联动**:质量运营师监测到**同一问题反复出现**时,除了硬化,还应**建议启用监管人员(§17)** 做复盘 —— 效率问题归硬化,反复出错归监管。

## 17. 监管人员:复盘反复问题与重大漏洞,从流程上防复发

> **治理角色**,与质量运营师并列、但管另一头:质量运营师保"效率"(把流程硬化省 token),**监管人员保"正确性 + 不复发"**。它专门盯两类信号 —— **同一问题反复 N 次未根治**、**重大漏洞修复** —— **跳出"改症状"做 meta 复盘**,并把教训固化成流程,减少复发。
>
> 来源:用户提议(2026-06-15),源于一次 SVG「元素碰撞反复搬家」的真实踩坑。

### 动机示例(真实案例)
反复改一张 SVG 的"元素重叠",改了很多版,碰撞只是换位置(连线 → 发光 → 光点 → 贴字),根治不了。**根因不在某处坐标,而在系统性认知误区**:① 追症状不追根因(只改最显眼那处);② 只按几何算间距、忽略 glow 外溢;③ **不渲染自查、只靠坐标推断,没去看用户给的错误截图**。改 N 次症状都没用 —— 根因在方法。

### 触发
- **重复失败检测**:同一问题 / 任务的 attempt 反复失败 ≥ N 次(用 §10 的 attempt 计数识别"这是第几次");
- **重大漏洞**修复;
- 质量运营师监测到重复问题时**上报 / 建议启用**(§16 联动)。

### 职责
1. **误区诊断(meta 分析)**:不改症状,而是判断是否陷入系统性误区,并给误区命名(如"未看证据只靠推断""追症状不追根因""盲调不验证")。
2. **复盘(对事不对 agent)**:对反复问题 / 重大漏洞做根因复盘,产出"现象 → 根因 → 避免方法",归档进知识库(§15)。
3. **流程固化防复发**:把复盘教训变成**规则 / 检查项**,交质量运营师硬化成 skill / 检查脚本,或写进 L0/L1 规则(如:"改带视觉的产物,必须**渲染自查 + 对照用户提供的截图**才算 done")。
4. **方案设计介入**:下次再遇反复修改 / 重大漏洞,**在动手前介入**,先把对的处理方案设计出来(而非让执行 agent 直接瞎改);可结合**多智能体辩论**(璇玑式)做多视角校验。

### 与辩论 / 验收证据的关系
- **辩论**(过程中多视角)= 预防单点盲思;**监管人员**(事后复盘 + 下次事前介入)= 纠正 + 制度化。互补。
- 强依赖 §10 **验收证据**:监管人员可设"硬验收门" —— 带视觉的产物**必须附渲染截图、且对照用户给的证据**才算完成,从源头堵住"盲调交付"。

### 边界
- 只在"反复 / 重大"时介入,不干预正常任务(免拖慢);
- 复盘**改流程、不追责**;产出的规则要可执行、可验证,避免空泛的"以后小心"。

## 18. 声明式控制平面 v2:结构已知就用确定性路由

> **核心转变**:**结构已知的流程,不要再让 LLM 每步现想**。评审循环、研究-汇总、计划-实施这类**固定流程**,用声明式编排写死;**编排层本身不调模型、不烧 token**,只在任务**真需要临场重规划**时,才把"下一步去哪"交给 LLM。与 §16 硬化同源:硬化固化**动作**,声明式路由固化**流程**。
>
> 来源:微软 Conductor / Bifrost + Manus / Anthropic 上下文工程(详见 `设计更新-2026-06-15-声明式控制平面.md`;Conductor/Bifrost/STATE-Bench **未核实**)。

### 18.1 声明式确定性路由(YAML + Jinja2)
固定流程写成节点图:节点 = agent 任务 / 工具 / human gate;边 = **Jinja2 表达式**条件转移(读上一步结果与状态)。编排器只算表达式、选下一节点,**零 token**。

```yaml
nodes:
  - { id: implement, agent: engineer }
  - { id: review,    agent: reviewer }
  - { id: human,     type: human_gate }
  - { id: done,      type: end }
edges:
  - { from: implement, to: review }
  - { from: review, to: implement, when: "{{ review.pass == false and loop < max_loops }}" }
  - { from: review, to: human,     when: "{{ review.severity == 'high' }}" }
  - { from: review, to: done,      when: "{{ review.pass == true }}" }
```

**何时交回 LLM**:仅当"没有预定义边匹配 / 需临场拆解新任务"时,调一次 LLM 的 **planner 节点**做重规划,产出新子图,再交回确定性编排执行。

### 18.2 human gate 是一等节点
人工介入不是事后补丁,而是**流程图里的一类节点**:打断、引导、审批都走它。命中 → 暂停、落事件、等用户在控制面确认或给引导 → 按输入选下一条边。对接 §8 的 steering/follow-up 与 §10 的 Approvals。

### 18.3 agent 隔离 + 上下文默认从严
- **隔离**:每个 agent 独立 session、独立 system prompt、独立模型/后端,**彼此不共享对话历史**——杜绝上下文串味。
- **可见性三档,默认从严**:`accumulate`(全量,仅总管级)/ `last_only`(只看上一步,流水线)/ **`explicit`(只看指定依赖,默认)**。这把 §3 的 need-to-know(领导看摘要、员工看自己任务)落到节点级强制。

### 18.4 多模型 v2(per-agent + 预测性故障转移)
- **per-agent 覆盖 provider/模型**:每个 agent 可独立指定厂商 + 模型,混用订阅版(Copilot/Claude)、API、本地(Ollama / vLLM 走 OpenAI 兼容端点)。
- **故障转移做成基础设施**(经统一网关 Bifrost / LiteLLM 出口):429 / 5xx / 超时 / 模型不可用 → 自动转下一后端;**更进一步:把"剩余限流余量"当路由信号,在额度耗尽前提前切换**(预测性,而非被动 fallback)。升级 §5 / §8 的兜底链。

### 18.5 上下文工程(省 KV-cache + 常驻记忆)
- **稳定前缀吃 KV-cache(Manus)**:每个 agent 的 prompt 维持**稳定前缀**(系统/规则/工具定义不抖动),命中 KV-cache、降本提速;易变内容放后段。
- **自动注入 AGENTS.md / CLAUDE.md**:发现工作区里的说明文件 → 作为该 agent 的**常驻记忆**注入(放进稳定前缀)。

### 18.6 可观测:事件 pub/sub
执行引擎只**发事件**(节点开始/结束/失败/等待),展示层(网页/移动端/审计)**订阅**,二者解耦——换前端不动引擎;事件流天然就是 §10 的审计与 cursor 同步源。

### 18.7 三道护栏
1. **防失控**:每条流程设**最大迭代数**(loop ≤ N)+ **墙钟超时**,超限强制停 + 报 human gate。
2. **运行前校验(validate / dry-run)**:跑前先静态校验——路由可达(无死节点/死循环)、引用完整(agent/工具/文件都存在)、表达式可解析;不过不准跑。
3. **定期回归(pass^5)**:用 **STATE-Bench 式 pass^5**(同组任务连跑 5 次的**全过率**)定期回归,验证"文件态记忆"是否真让 agent **更稳(方差小)、更省轮数**——比单次通过率更能暴露不稳定。

### 18.8 与既有章节的衔接
声明式路由是 §8 控制平面的**上层骨架**:steering/follow-up 仍是消息通道,声明式图管"结构已知"部分,LLM 只补"结构未知"部分。默认从严的上下文(§3)、per-agent 模型与限流余量切换(§5)、pub/sub 审计(§10)都在**节点级**生效。
