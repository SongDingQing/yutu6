# 失败案例库(玉兔项目 · 避坑)

> 来自迁移记录 `08` §9。这些是**监管人员误区库的项目级初始条目**;新机遇到同类问题,先查这里再新建复盘。

| 案例 | 症状 | 原因 | 怎么避免 |
|---|---|---|---|
| Hermes/Codex 联系不精准 | 玉兔不知道任务该交给 Codex | 能力目录 / handoff 规则不够显式 | 保留 multi-agent-collaboration-contract;装后做 handoff smoke test |
| 队列消息太吵 | 忙碌时多条飞书 / 语音提示 | 排队提示没合并、还触发语音 | 队列提示要短;排队中不触发"飞书来消息了" |
| 语音回复后唤醒失败 | 播报后再次唤醒不稳 | 播放 / 监听状态切换未恢复 | 装后必做"播报后再次唤醒"测试 |
| ASR 误识别 | Codex / 项目名 / 人名被听错 | 没有持久热词 / 纠错闭环 | 热词库迁移结构,内容用户确认后导入 |
| 深度解读上下文污染 | 新问题引用上一问 | 默认带入 history | 默认 `contextMode=new`,点"参考上一问"才 followup |
| YuanXiao placeholder host | APK 访问占位 host | 发布前没扫占位配置 | release candidate 必查 placeholder / base URL / metadata |
| 后台打断 Codex session | 移动端任务等不到回复 | 后台直接打活跃会话 | 保护 session 默认排队;直连须显式允许 |
| 队列不可见 | 用户不知任务已排队 | queued 元数据没传到 UI | 队列 notice 带 queue id / position / task summary |
| 任务位置误导 | 旧 blocked 任务让排名很大 | 全局队列与用户视角混用 | UI 显示 scoped position,运维留 global |
| 交付走错 worker | `煮元宵` 只交接不交付 | 用了 read-only handoff worker | delivery 命令走 delivery-worker lane |
| 聊天确认阻塞 | agent 反复问是否继续 | 把自然语言确认当审批 | 能执行就执行;真实审批用 typed card |
| 报告太长发不出 | 飞书摘要反复压缩 | 一开始写太长 | 摘要先短写,正文放文件路径 |

> 对接监管人员 SOP(`监管人员复盘SOP.md`):反复出现的问题先查本表;命中即按既有避免法,不命中再开新复盘并回填本表。
