# 监管人员复盘 SOP + 重复失败检测规则

> 配套设计文档 §17。监管人员是"管正确性 / 防复发"的治理角色,只在**反复失败**或**重大漏洞**时介入。
> 本文件是它的工作手册:怎么数 attempt、什么阈值触发、复盘怎么做、模板长什么样。

---

## 1. 触发条件(满足任一即启用)

- **重复失败**:同一"问题指纹"的失败计数 ≥ 阈值(见 §2)。
- **重大漏洞**:严重度=高(数据损坏 / 安全 / 不可回滚的破坏),**立即触发**,不等计数。
- **人工 / 质量运营师上报**:质量运营师监测到同一问题反复出现时建议启用。

---

## 2. 怎么数 attempt(重复失败检测规则)

核心难点:判定"这是**同一个**问题的第 N 次",而不是把不同问题混在一起。

### 2.1 两个计数
- **attempt_count(任务级)**:同一 `task_id` 的运行尝试次数(直接来自 §10 的 task/attempt)。
- **recurrence_count(问题级)**:同一**问题指纹**在滚动窗口内被"报告→修→又被报告"的轮数。用于抓"换了 task 但还是同一个病"(例:SVG 碰撞每次在不同位置,但都是同一类问题)。

### 2.2 问题指纹(聚类键)
```
指纹 = 产物标识(文件/对象) × 症状类别 × 反馈类型
```
- **产物标识**:如 `logo.svg`、某接口、某模块。
- **症状类别**:把同义反馈归一,如「碰撞/重叠/糊在一起/贴着/overlap」→ `collision`。
- 维护一张**症状同义词表**(随复盘增长),把用户原话映射到统一类别。

### 2.3 触发阈值(可配)
| 参数 | 默认 | 说明 |
|---|---|---|
| `recurrence_N` | **3** | 同一问题指纹反复 3 轮 → 触发 |
| `attempt_N` | **3** | 同一任务连续失败 3 次 → 触发 |
| `severity_high_N` | **2** | 高危 / 重大漏洞 2 次即触发 |
| `window` | 本项目近 **7 天 / 20 任务** | 统计滚动窗口 |

### 2.4 数据来源
每个 attempt 落 §10 事件日志,至少带:`task_id`、`产物`、`症状标签`、`结果(pass/fail)`、`用户反馈摘要 + 截图链接`、`本次改了什么 + 依据`。检测器扫这些算两个计数。

---

## 3. 复盘流程(七步)

1. **冻结**:先停止继续"改症状"(防止碰撞再搬家)。
2. **取证**:拉齐这条问题线的**全部 attempt**(每次做了什么、依据什么、结果、用户反馈与截图)。
3. **元分析**:对照 §5 误区库,判断陷入了哪种认知误区。
4. **根因**:写「现象 → 根因 → 为何之前没解决」。
5. **对策**:给出 ① 这次怎么**一次修干净** ② **防复发规则**(交质量运营师硬化) ③ **验收门**。
6. **落地**:更新规则 / 检查项;下次同类任务**事前介入方案设计**。
7. **归档**:复盘记录进知识库(§15);更新误区库与症状同义词表。

## 5. 误区库(初始条目 · 可追加)

监管人员用它给误区**命名**。初始条目来自首个真实案例(SVG 碰撞反复搬家):

| 误区 | 识别特征 | 对策 |
|---|---|---|
| **追症状不追根因** | 每次只改截图里最显眼那一处 | 整区域**所有元素对**两两排查,一次修干净 |
| **盲调不验证** | 没渲染 / 没跑就交付,全靠推断 | 强制产物**自查**(渲染出来看)再交付 |
| **只信自算、忽略证据** | 用自己的计算否定用户给的截图 | **证据优先于推断**;先看用户截图再动手 |
| **几何忽略外溢** | 按边缘几何算间距,忽略 blur/glow 外溢 | 间距 = 几何边 + glow 外溢量,且留富余 |
| **过早收敛** | 第一个方案就动手,没比选 | 关键改动用**多智能体辩论**先比方案 |

---

## 6. 与系统的接口

- **输入**:§10 事件日志 / attempt 计数、用户反馈(含截图)。
- **输出**:复盘记录(→ §15 知识库)、防复发规则(→ 质量运营师硬化 §16)、验收门(→ §10)。
- **介入点**:下次同类任务的**方案设计阶段(动手前)**;可调用多智能体辩论做多视角校验。

> 一句话:**反复失败 → 元分析定误区 → 一次修干净 + 固化成规则 → 下次事前堵住**。
