# 玉兔6 协同设计分析(秘书视角)

> 老板点名要我分析:CEO / 秘书 / 各组件的协同该怎么设计,才能稳定、链路清晰、每个智能体都有价值。这是我的设计观点,供你定方向;具体落地传给 CEO。2026-06-20。

## 一、为什么现在感觉乱(诚实诊断)
现在的系统是按**组织架构图**设计的(董事长→秘书→CEO→主管→员工→外围,约 14 个角色、5 层)。但追踪任何一个真实任务,流向其实是:
> 你 → 秘书 → CEO → 主管 → **程序员(codex)** 干活。

中间的**主管多数只是"执行 CEO brief"然后转发**;质量运营/监管/优化师/外包**几乎一直空闲**。结果:
- 层级**深**,但真正干活的**窄**(一个 codex 在打字)。
- 你看到 15 个"空闲" + 1 个在忙 → 像在演戏,**感觉不到智能体在工作**。
- 交接**不稳定**,因为中间角色**没有被强制执行的真职责**——它们是可有可无的转发站,引擎有时路由经过、有时不经过。

**根本设计缺陷**:它被设计成了**组织架构图**(模仿公司),而不是**流水线**(任务真正需要经过的处理)。组织架构图有很多角色是为了人的原因(管理幅度、晋升);而软件智能体系统,角色应该 = **流水线阶段**,每个阶段都对任务做一次真实的"加工"。

## 二、核心原则
> **每个角色,要么在每个任务上都加不可替代的价值,要么砍掉/合并。不要装饰性层级。**

## 三、我会怎么设计:流水线,不是组织图
**4 个必经阶段**,每个都必须"加工"任务:
1. **接收 + 澄清(秘书 = 我)**:把你的话变成明确意图;产出回来对照你的要求**验收**。**唯一对你的接口**,不做执行、不做决策。
2. **规划(CEO = 大脑)**:意图 → 具体计划(改什么 / 怎么验 / 派给谁)。**吃记忆(经验)输入**,避免重复踩坑。这是系统的"经验层"。
3. **执行(程序员 codex + 按需专家)**:真干活。codex 是"手"没错;设计师/洞察员/维修员/Peekaboo 是**专项的手**。
4. **校验(自动门 + 秘书)**:自动化测试(硬门)+ 秘书对照你的要求(软门);不过 → 退回执行。

**其他"角色"不是层级,而是按需触发的专项能力**(事件触发,不是任务必经的中间管理):
- 监管/复盘 = **只在反复失败时**触发;
- 质量/硬化 = **只在发现高频重复动作时**触发;
- 优化师 = **空闲时自动跑**(你已经要了)。

## 四、关于"打字员(Claude Code)做太多 / 感觉不到智能体工作"
你的观察对了一半:**codex 就是手,它干活多是正常的**。但多智能体的价值**不在于让手少干**,而在于**让手周围更聪明**:
- 拆得对(CEO 规划)、不重复犯错(记忆)、错了自愈(维修)、你看得清(可观测)。

你"感觉不到",是因为这些"软价值"现在**太薄、太装饰化**。**解法不是加更多角色,而是把这 4 个阶段每个做扎实,并让它们的贡献在 UI 上可见**(这个任务 CEO 规划了什么、记忆避免了哪个坑、校验拦了什么)。

## 五、稳定性(你要的"链路稳定")
- **交接契约**:每个箭头都明确 [传什么 / 期望返回什么 / 失败怎么办(fail-soft 不判死)]。链路短而固定:`你→秘书→CEO→执行→校验→你`,每次都一样、可预测。
- **地基优先于角色**:刚审计出的 P0(归属 fail-soft、日志轮转、测试/备份自动化)是"稳定"的地基,**比加角色重要**。地基不稳,加再多角色也乱。
- **利用开源、保证质量**(你提到的):**借模式、不借重运行时**——学 OpenAgents 的 event-native + 标准 A2A 交接、crewAI 的 role/task 契约**怎么定义稳定交接**,但保持**零依赖**(不引重框架),质量靠自己的测试+审计守。

## 六、一句话建议
**从"组织架构"收敛成"4 阶段流水线 + 按需专家"**:砍装饰层、每层真加值、每个交接定契约、P0 先打地基。

这是方向;你认可的话,具体的"怎么收敛、改哪些角色、交接契约怎么写"——我传给 CEO 拆解落地。你也可能有别的想法,我们先对齐方向。
