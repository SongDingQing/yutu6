# 玉兔6 · 董事会(重要架构多模型评议)设计方案

> 老板要:重要系统架构决策前,4 个模型交叉评议、互相挑刺、避免理解偏差,最多 3 轮。**结论默认安全直接跑,只有 3 轮后 Opus 仍判风险才找老板。** 供过目,认可后落地。
> 秘书拟 · 2026-06-21

---

## 一、定位

重要系统架构决策前的**多模型评审委员会**——同一条(秘书补齐后的)指令,4 个不同模型交叉消化、**互相挑刺**,避免单一理解偏差,提前揪出风险和优化点。

## 二、成员(4 位董事)

| 董事 | 模型 / runner | 角色 |
|---|---|---|
| DeepSeek | new-api | 评审 |
| GLM-5.2 | zhipu-glm(免费) | 评审,主力 |
| GPT-5.5 | codex | 评审 |
| Opus-4.8 | claude | 评审 + **最终决策者(3 轮后)** |

## 三、触发(控成本)

秘书补齐指令后,判断是否涉及**重要系统架构**(改引擎 / 队列 / 路由 / agent 体系 / 数据架构 / 版本发布等):

- **是** → 接入董事会。
- **普通任务** → 不接入(省 opus / codex 贵额度)。

## 四、流程(最多 3 轮,默认直接执行)

1. 秘书把补齐后的指令交给董事会。
2. 4 位董事**各自挑刺**:揪理解偏差 / 风险 / 漏洞,给具体优化建议。**前几轮务必积极找问题**,不许敷衍。
3. 秘书整合 4 方意见 → 优化指令(第 1 轮)。
4. 仍有可挑 / 分歧 → 再迭代(累计**最多 3 轮**)。
5. 收敛后:**结论默认足够安全 → 直接走 CEO 执行,不打扰老板**。

## 五、决策边界(关键,老板更正)

- **默认直接运行**:评议收敛后的结论默认足够安全,**直接执行,不用每次找老板确认**。
- **唯一例外**:**3 轮之后由 Opus-4.8 作最终决策**——若 Opus 判定结论**仍有误判风险**,才飞书发老板一张**需点击的决策卡片**,由老板拍板;否则照常直接执行。
- **硬上限 3 轮**,防止无限讨论烧额度。
- 只在重要架构触发;普通任务不接入。

## 六、董事 prompt 要点(重要)

- 前几轮**当严格评审**:主动挑刺、质疑假设、找理解偏差和风险、提具体可改的优化点。
- **禁止敷衍**:不许像自动优化师那样每轮都"挺好、没什么可调"。若真没问题,要说明为什么真没问题。
- 后期(接近收敛)再判断是否足够安全。
- **Opus 作为最终决策者**,3 轮后专门评估"还有没有误判风险",有则发卡片上报老板。

## 七、落地清单(认可后执行)

1. 建董事会 agent(4 成员,分别绑 deepseek / zhipu-glm / codex / claude;opus 兼最终决策)。
2. 秘书加"重要架构识别"判断 + 接入董事会钩子。
3. 评议流程引擎:收集 4 方 → 整合 → 迭代 ≤3 轮 → **默认执行**;或 opus 判风险则发老板**需点击的决策卡片**。
4. 飞书:需点击的决策卡片(仅这一种情况用)。
5. UI:重要架构任务显示"董事会评议中(第 X/3 轮)"。
6. 评议记录沉淀(决策 + 理由进 `memory/decisions.md`)。
