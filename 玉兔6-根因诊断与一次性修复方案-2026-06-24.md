> 日期 2026-06-24

## 0. 一句话结论

系统现在**没有任何一环把「声称完成」和「现实」对账**:验收只读 agent 自己写的作业本(`result.md` 里的 JSON),修复只「派单/自报」,完成既不进 git 也不重载进常驻进程。于是修复任务被**执行层杀掉 → 被验收层盖章 → 被落地层挡住**,系统永远在「找到根因→宣布修好→其实没动」,同一个 bug 跨 5–6 天反复复发,维修单堆到 70 张。

**先做 P0-A + P0-B,假修复当场堵掉大半。**

## 2. 三层真根因(带证据)

### 第一层 · 执行被自己杀掉 → 对应「给了不执行」

真实失败里约 **63%** 不是任务做错,而是编排层把**正在跑的引擎误杀**:

| 失败原因(工单 `reason` 原文) | 工单数 | 性质 |
|---|---|---|
| `running 队列文件已不存在,终止对应 engine 并释放 slot` | **23** | 并发清扫把 running 文件移走 → 引擎发现没了 → SIGTERM |
| `…无节点进展超时判死` | ~12 | 看门狗因「进展字段没刷新」误判超时 |
| `engine-runner 退出码1 未写明原因` / `被信号 SIGTERM 终止` | ~9 | 黑盒退出 / 外部 kill,run 目录不存在、worker 日志 0 字节 |
| `node_failed`(泛化) | 9 | |
| `检测到 Starlaid 或无法安全确定项目归属` | 4 | 归属判死(已修,见 §1) |

- `engine_signal`:**SIGTERM 40 次 vs none 30 次**——多数失败是「被杀」不是「报错」。
- `memory/experience.md` 自己命名了这一族:**「重启即清扫,清扫即杀活」**、**「队列清扫器把 owner-worker pid 当 engine pid」**(误判 engine 不存在 → 重新入队 running 文件 → 原 engine 轮询发现 running 消失 → `queue-running-missing` → 被 SIGTERM,「已 5 次,属系统性竞态」)。
- **派发断链**:`engine-runner.js:1003` CEO 只把子任务塞进 `supervisor-<项目>` 队列就返回「等下游」;但 `server.js:58 PERSISTENT_QUEUE_AGENTS` **默认只有 `repair` 常驻**,`supervisor-*` 不常驻、靠 backlog 探测按需拉起。没人来取 → 子任务一直 `queued` → 父任务阻塞至超时 = **「派了、然后就没了」**。
- 重复失败被 `dedupe-cooldown` **静默压住(命中 25 次)**,所以表面很安静,严重性被掩盖。

### 第二层 · 验收与现实脱钩 → 对应「说解决了其实没解决」

整条「完成 → `true_done` → 版本+1」**只校验 agent 自己写进 `result.md` 的 JSON**,从不重跑命令、不复现 bug、不看文件改没改:

- `shared/engine/cli-runner.js:758-760`:`extractJson(stdout)` 把 agent 自报的 JSON **原样并入权威上下文** `ctx.implementation` / `ctx.review`。
- `shared/engine/done-gate.js` **全文无任何 `spawnSync/execSync/child_process`**——零进程执行。
- 所谓「硬证据」`evidenceHasHardPointer` `done-gate.js:222-238` / `evidencePointerIsVerifiable :429-449`:**只要 evidence 里有字面量 `exit_code:0` 且文本含「test/PASS/通过」之类词,就算测试通过**。
- `validateChangedFiles :911-930`(尤其 `:921`):只 `fs.existsSync`,**只查文件「存在」、不查「改过」**;一个早就存在、这次一行没改的文件写进 `changed_files` 也能过。
- 收敛分数是 **LLM 自评分**:`loop-engineering.js:82-105 scoreFromReview` 直接读 `review.score`;`:331-332` 兜底「`review.pass=true` → 自动给 `target_score`」;done-gate `:877-903` 只校验这个自评分导出的 `converged`。
- `completion_hash`(`projects/控制台/version-progress-hook.js:310-321`):只 SHA256(agent 自报字段 + gate 判定),**不含任何文件内容/命令产物**,空手也能算出合法 64 位 hash。
- 维修类更直接:`ceo-worker.js:2742` 把 **「维修工单已创建/接单」这句话本身叙述成「闭环」**;关单靠 `repair-ticket-complete` 工具自报,**从不复测原失败是否消失**。
- **铁证**:`tests/repair-policy.test.js:113-173` 就是一条官方测试——手写假 task、evidence 里写 `{command:'node tests/repair-policy.test.js', exit_code:0}`(**命令从未执行**),断言 `gate.ok === true` **通过**。这条测试等于把漏洞固化进了 CI。

> 注:done-gate 在格式上反而**偏严**,有时把「真完成」按格式误拦(制造一批 `done_gate_failed` 工单)。所以问题是两头:**该严的真值没验(命令/diff),不该严的格式瞎严**。两头都指向同一修法——验收要基于**真实执行**,不是自报 JSON 的结构。

### 第三层 · 完成不落地不生效 → 让一切反复复发(最致命)

- `git log` 全仓**只有 4 个提交,冻在 `v0.0.0.3`**;所有被反复「修」的引擎文件(`ceo-worker.js`、`engine-runner.js`、`cli-runner.js`…)自 `v0.0.0.1` 后**零提交**;`shared/engine/done-gate.js` **从来没被 git 跟踪过**。
- `logs/version-bumps.jsonl` 284 行:**283 行 `skip`(`missing_granularity`/`missing_release_impact`)、只有 1 次真 bump**。那次也只把 `VERSION.json` 写成 `0.0.1.3`,**git 不动、push 被标 `deferred_it_engineer` 从未发生**。→ 现在 `VERSION.json=0.0.1.3` 但 `git HEAD=v0.0.0.3`。
- 常驻 `ceo-worker` 进程**缓存了旧代码**,补丁落盘也不重载。`experience.md`:**「补丁已落盘 ≠ 运行态已生效」**「常驻 worker `require` 缓存了旧队列模块」。
- 系统自己的总结:**「沉淀≠落地」**。`memory/decisions.md:538-539` 写明 06-23 复盘的 R1/R3/R4 规则「**仅写入 memory 未落代码**」,06-24 同族故障复发并恶化(`resource.scheduler.all_blocked` 595→1961,×3.3)。

### 闭环效应:为什么「越用越笨」

修复任务 **①被执行层 SIGTERM 杀掉 → ②被验收层凭自报盖章「已修」→ ③通过的也不进 git / 不重载生效**。三层互相喂料,根因永不消失:维修单越堆越多(70 张,头号指纹 `50b653…` 一个就 21 张、**全 todo 零解决**),`status-rollup.md` 涨到 55KB,上下文越塞越满、信噪比越来越差——这就是「越用越笨」的体感来源。

## 3. 一次性修复方案(按性价比 P0 → P2)

### ⚖️ 铁律(贯穿所有修复,违背则必然复发)

1. **真验证,不信自报**:任何「测试通过/已修复」必须由**系统真 spawn 跑一遍**得到真实退出码,不采信 `result.md` 里的字面量。
2. **真落地**:任何「完成」必须产生**一个真实 git commit**(版本号与 commit 绑定),否则不算 `true_done`。
3. **真生效**:补丁落盘后必须**重载/重启常驻 worker**(或验证热加载),否则视为未生效。
4. **真闭环**:维修类完成必须**重跑原失败场景、确认不再复现**,「创建/接单工单」只能叫「受理」,不能叫「闭环」。

### 🔴 P0-A · 让验收真执行(堵第二层,收益最大)

**目标**:把「自报 `exit_code:0` + 文件存在」换成「真跑命令 + 真比对 diff」。

**改哪里**(`shared/engine/done-gate.js`):
- 顶部加 `const { spawnSync } = require('child_process');`
- `evidenceHasHardPointer :222-238` / `evidencePointerIsVerifiable :429-449`:当 `entry.command` 存在且 `opts.executeEvidence===true` 时,**实际** `spawnSync(entry.command, {cwd: workspaceRoot, shell:true, timeout, maxBuffer})`,用**真实 `status` 覆盖自报 `exit_code`**;真实退出码≠0 直接判 `false`。只放行白名单命令(`^node tests/`、`^npm test`、`^pytest`、`^node .*\.test\.js`),其余要求附 `git diff` / 产物路径。
- `validateChangedFiles :911-930`:对每个 `changed_files`,在 `opts.gitVerify` 下执行 `git diff --quiet -- <file>`(或比对 `git status`/mtime/内容哈希)判定**确有改动**,无改动则打回。
- 调用处把 `executeEvidence:true, gitVerify:true` 传进 opts:`engine.js:210-213`、`ceo-worker.js`(调 `validateReviewLoopCompletion` 处)、`version-progress-hook.js:169`。

**验证**:改完后,把一条「假数据(命令不会真过)」喂给 gate,必须返回 `ok:false`;真实跑通的任务返回 `ok:true`。

### 🔴 P0-B · 让完成真落地 + 真生效(堵第三层)

**目标**:`true_done` ⇒ 一个真实 commit + 版本号绑定真实文件指纹 + 常驻进程重载。

**改哪里**:
- `version-progress-hook.js:310-321 completionHash`:把 `changed_files` 的**真实文件内容哈希**纳入 payload(`fs.readFileSync` → sha256),`MISSING` 或「与上次 bump 完全相同」判 `not_true_done`。
- `handleTrueDone :508-739`:版本 +1 之后,**真正执行** `git add -A && git commit -m "<version>: <summary>"`(沿用 `VERSION.json.remote` 的 gitee 配置);commit 失败则**回滚版本号**、不发 `true_done`。把 push 从「deferred 给 IT 工程师」改成**完成链路内必做**(或显式失败告警,而非静默跳过)。
- **常驻 worker 热重载**:`ceo-worker.js` 收到 `code.reloaded` 事件或检测到 `shared/engine/*`、`projects/控制台/*` mtime 变化时,**安全重启自身**(等当前 running 引擎收尾 → `process.exit` 让 server 拉起新进程,避免「强杀活引擎」)。配合 P1-C 的 pid 修复一起做。

**验证**:跑一次真实任务完成后,`git log` 必须**多一个 commit**、`git HEAD` 版本 == `VERSION.json` 版本;改一个文件落盘后,新任务必须命中**新代码**。

### 🟠 P1-C · 让执行不自杀(堵第一层)

**目标**:消除「清扫/看门狗/重启」对在途引擎的误杀,让 supervisor 队列常驻消费。

**改哪里**:
- **pid 语义**(对应 `experience.md`「清扫器把 owner-worker pid 当 engine pid」):清扫器判定「engine 是否存活」时,必须用 **engine 进程自己的 pid**,不能拿 owner-worker pid 判,误判后**不要重新入队 running 文件**。
- **看门狗**(`…无节点进展超时判死`):判超时前先确认「进展字段本应刷新但没刷新」,把心跳/进展续约和「真卡死」区分开;CLI 长任务要有 keepalive 心跳。
- **「重启=清扫=杀活」**:把「重启控制台/清运行时」改成**有副作用的危险操作**——先排空/等待在途引擎收尾,不做批量 kill。
- **supervisor 常驻**:`server.js:58` 把 `^supervisor-` 前缀 agent 纳入常驻;或 `ensureWorkersForBacklog :709-723` 对 `^supervisor-` **无条件 `ensureQueueWorker`**(不要求 `queued/running>0`),消除「入队到 worker 拉起之间无人消费」的窗口。
- **断链自愈**:`engine-runner.js:1003` 入队后 emit `queue.worker.needed{agent: supervisorQueue}`,并确认它在 server 的唤醒事件集合里,让 server 立刻拉起下游 worker 而非等定时器。

**验证**:连续派 20 条任务,**指纹 `50b65386540b23c8`(running-missing SIGTERM)不再新增工单**;`supervisor-控制台` 队列始终有 worker 在取。

### 🟠 P1-D · 维修闭环必须复测原失败

**改哪里**:
- `shared/agents/repair/agent.json`:`repair-ticket-complete` 前置改为「**必须附原失败场景的重跑命令 + 真实退出码**」(由 P0-A 真执行校验)。
- `ceo-worker.js:2742`:closure 文案与逻辑分离——「已创建/接单」只能写 **`受理中:维修工单…已接单(待复测闭环)`**;「闭环/已解决」只保留给**复测通过**的路径。

**验证**:维修单状态从 `todo→done` 必须伴随一次「原失败重跑且退出码 0」的证据记录。

### 🟡 P2-E · 收口与降噪

- **mock 脚枪**:`config.json:305` 给 `mock` 加 `"disabled": true`;`cli-runner.js:454 resolveRunnerForNode` 开头加:若 `runnerId==='mock'` 或 `cmd[0]==='__mock__'` 且非自测环境 → `return {fail:'mock runner 不可用于真实任务'}`。
- **静默 catch 留痕**:`ceo-worker.js` 关键 `catch(_){}`(尤其 `runEngine`/`handle`/`waitForProjectRouteDownstream`)改成 `catch(e){ eventlog.emit('worker.swallowed_error',{where,reason}) }`,让被吞异常在 `engine-events.jsonl` 留痕。
- **dedupe 升级出口**:`dedupe-cooldown`/`repair.ticket.skipped` 加「同签名复发达阈值(如 ≥3)→ 升级人工/根因工单」的出口,别让「压制」冒充「解决」。

## 4. 落地顺序与总验收(怎么确认这次真解决了)

**顺序**:P0-A → P0-B →(此时已能真验证+真落地,先用它去修后面两条,形成正反馈)→ P1-C → P1-D → P2-E。

**总验收(必须全部为真,不许自报)**:
1. `git log` 提交数 **> 4**,`git HEAD` 版本 == `VERSION.json` 版本(脱钩消失)。
2. `tests/run.js` 全绿,且 `tests/repair-policy.test.js` 已改成「不真跑命令 ⇒ gate 拒绝」(见 §5)。
3. 连续运行 1 天,**头号指纹 `50b65386540b23c8` 工单新增 = 0**;`board/repair-tickets/` todo 数**下降**而非上升。
4. 随便挑一张历史 todo 工单的「已修复」声明,按 P0-A 真重跑——若仍失败,说明那是假修复,据此清理。

## 5. 防回归:把「假修复」测试改成守卫

`tests/repair-policy.test.js:113-173` 现在断言「手写 vars + 字面量 `exit_code:0` 应过门」——**这是把漏洞写进了 CI**。P0-A 上线后改为:不带 `executeEvidence` 时仍过(兼容);**新增**一条:带 `executeEvidence:true` 且命令实际会失败时,断言 `gate.ok === false`。让 CI 永久保证「自报不再能绕过真实执行」。
