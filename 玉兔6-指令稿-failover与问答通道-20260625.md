## 指令稿 A · runner failover(让声明的 prefer/chain 真生效)

**背景**:`shared/routing/model-routing.yaml` 声明了 `roles.*.prefer` 候选链 + `failover.chain` + `failover.triggers`,但 `shared/engine/cli-runner.js:455 resolveRunnerForNode` **只取单个 runner**,失败即整节点 `{fail}` → 触发维修循环。设计与代码脱钩。

**目标**:runner 执行失败(429/5xx/超时/通道不可用)时,按 prefer 顺序降级到下一候选 runner 重试,而非整节点失败。

**改**:
1. `cli-runner.js`:建 `prefer 标识(如 subscription.claude.sonnet) → config.json runnerId` 的映射(读 `config.runners` + `model-routing.yaml`)。无法映射的候选跳过。
2. `resolveRunnerForNode` 改为返回**候选 runnerId 列表**(按 role 的 prefer 顺序;末尾兜底原 roleMap runner),而非单个。
3. `runRunnerOnceSync` / `runCommandWithOutputEvents` 外层加重试循环:依次尝试候选;命中失败触发器(`res.status!=0` / `res.signal` / `ETIMEDOUT` / stderr 含 `429`|`5xx`|`No available channel`|`model_unavailable`)→ 降级下一候选 + `emit('runner.failover',{from,to,reason,attempt})`;候选全失败才返回 `{fail}`。
4. attempt 上限 = 候选数;每次 attempt 留痕。需要落盘的 implement 节点,降级候选仍须满足 `canWriteFiles`(或经 harness 升级)。

**验收(必须真测)**:注入首选 runner 失败(mock 返回非0/超时)→ 断言降级到次选成功 + `emit runner.failover`;全候选失败才整节点 fail。新增测试守卫 `tests/runner-failover.test.js`。

## 指令稿 B · agent 追问 → 挂起 → 续跑(问答通道)

**背景**:agent 是单向流水线,**没有"提问→等答复→续跑"回路**;`questions.md` 只写在 prompt、代码零实现。卡住只能硬猜 / 判死 / 单向飞书。这是"给了不执行 / 越用越笨"的一大根源。

**目标**:agent 执行中遇到需澄清 → 提问 → 任务挂起(**释放并发槽,引擎转去做别的任务**)→ 上级/人答复回填 → 从**原节点**续跑。严格复刻已验证的 `awaiting_human` resume 链路(低风险)。

**改**:
- **A. runner 协议第四种返回**(`shared/engine/cli-runner.js` extractJson 后 ~759 `resolveRunnerResponse`):识别 agent 输出 `{"ask":{question,to,timeout_sec,options?,resume_node?}}` → 返回 `{ask:{...}}`(与 `{vars}`/`{evidence}`/`{fail}` 并列)。`engine.js:8` 协议注释补 ask。
- **B. 状态机**(`shared/engine/taskstore.js` STATES):加 `'awaiting_answer'`(非终态,`TERMINAL_STATES` 不含)。
- **C. 引擎**(`shared/engine/engine.js` agent 节点 runner 返回处理 ~364-465):`out.ask` 时 → `taskstore.update` 暂存 `pending_ask={question,to,node:current,asked_at}` + `emit('node.await_answer')` + `setState 'awaiting_answer'` + `return {ok:false,reason:'awaiting_answer',node:current}`。cursor 停在 current(参照 `engine.js:312-316` awaiting_human),保证答复回来从原节点重跑。
- **D. 暂停判定**(`engine-runner.js:1197 isPausedResult`):加 `reason/state==='awaiting_answer'` → 沿用 `exit(5)` → runEngine paused → 队列 paused 既有路径(无需新退出码)。
- **E. 回填 API**(`server.js` queueMatch 路由表 ~3689):加 action `answer`(`/api/queue/:agent/:id/answer`,body=`{answer,answeredBy}`)→ 写 `entry.pending_answer={text,by,at}` + `resumeTask=true`(参照 `Q.steer` 注入写法;`Q.resume` 的字段保留白名单要含 `pending_answer`/`resumeTask`)→ `Q.resume` → `ensureQueueWorker`。
- **F. 答复注入续跑**(`ceo-worker.js:1777 makeSpec`):已凭 `resumeTask/existingTaskId` 复用同一 taskId;新增把 `latest.pending_answer` 透传进 spec(`spec.injectedAnswer`),`engine-runner` 构造 ctx 时写进 ctx,原节点重跑时作为"追问已答复"上下文(runner buildEnvelope 在 prompt 注入)。
- **G. 释放并发槽**:awaiting_answer 走 `exit(5)` 后,handle 的 `releaseRunResources`(~3797)释放 engineSlot/typeLock/资源域锁,队列项 paused 不占 in-flight → **天然实现"挂起期间引擎转去做别的"**(老板要的"像人一样")。
- **H. 超时降级防新卡死**:`ceo-worker.js` RUNNING_SWEEP 循环加扫 `awaiting_answer` paused 项,超 `timeout_sec` 未回填 → 按策略降级(注入 "no answer, best-effort default" 后 resume / 或 failed + notify)。恢复优先级:resume 时 `setPriority 0`(`Q.jump`)防被新任务插队饿死。

**配套并发优化**(顺带解开"真并发为1"):`ceo-worker.js:76 RUNNER_SINGLEFLIGHT='claude,claude-opus-4-8'` 把所有 Claude 角色全局串行,是真并发为1的元凶。收窄它(若 Claude CLI 本地登录态可并发就移除)→ 让资源域锁(已开)按读写域判定并行;`ENGINE_MAX_CONCURRENCY`(73)按 CPU/配额上调。

**验收(必须真测)**:agent 返回 `{ask}` → 任务 `awaiting_answer` + 槽释放 + 别的任务能跑;`/answer` 回填 → 从原节点 resume 续跑且带答复上下文;超时降级不无限挂。新增测试守卫。
