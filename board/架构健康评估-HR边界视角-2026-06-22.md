## 一、一句话结论(HR 视角)

**今天的 9 类问题,在技术上是 CEO 说的 4 个架构弱点;但在组织上是同一个病:"没有单一归属"。** 代码里"自愈逻辑散在 3 个文件、谁都不是权威",和组织里"没有任何角色为'运行态是否健康'负总责、没有任何角色为'这是补丁还是架构'把关",是同一件事的两次显形。

更要紧的一条 HR 发现:**今天本该拦住"补丁堆叠"的两个治理角色——董事会(逐事评议)和监管(逐事复盘)——一个被 config 关停、一个只在凌晨定时跑,等于在补丁最密集的这一天,系统里没有一个活着的"架构守门人"。补丁能堆起来,不是因为没人会修,而是因为没人被指派去拦。** 这正是老板担心的"混乱机械"的组织成因。

结论与 CEO 一致:**不推倒重来,做"有方向的渐进式收敛"。但 HR 补一句顺序——先补"归属"(指派 owner + 重启守门),再补"代码"(统一锁/抽 domain)。否则代码修得再对,下一批补丁照样无人拦。**

## 三、系统性根源:同一个病的"代码身"与"组织身"

CEO 报告已把技术根因讲透(A 进程生命周期 / B 状态一致性 / C 识别守卫 / D 职责未分层)。HR 不重复,只指出**每个技术根因背后都站着一个"归属缺口"**——这才是 HR 该管的层面:

| CEO 的技术根因 | 同一病的"组织身"(HR 视角的真根因) |
|---|---|
| A 进程生命周期靠猜 | **没有"运行态健康"的单一 owner。** `repair`(维修员)只做事后救火;`server.js`/`watchdog` 是无主基础设施;orchestrator 顺带管自己的 worker。没有任何角色被指派"持续核对期望态 vs 实际态"。 |
| B 5 套锁并存 | **没有"资源租约标准"的 owner。** 对的模式(`resource-locks.js`)写出来了,但没人有权、有责把它定为唯一标准并强制迁移。新模式只活在写它的那个子系统里。 |
| C 识别/守卫到处加正则、默认激进 | **没有"识别/触发决策"的 owner,且默认动作没有"安全侧"约束。** 谁需要分类就地写一段 `if(/正则/)`,无置信度、无作用域、默认归控制台/默认开会/默认阻断。 |
| D 核心机制复制粘贴 | **没有"单一事实源"的 owner 和评审闸。** 没人在评审时拦"这是第二份 spawn/锁/识别实现",于是两份各自长歪(见 `unref()` 漏写)。 |

**贯穿全局的组织根因(老板最该看这一条):**

> 系统有 17 个角色,却**没有一个角色对"架构连贯性 / 这是补丁还是架构"负总责**。本该承担这个职责的两个角色——`board`(董事会:逐架构事项挑刺)和 `governance`(监管:反复失败→复盘→立规)——今天的实际状态是:**board 被 config 关停,governance 只在凌晨 5 点定时跑一次**。也就是说,在白天补丁最密集的时段,**"补丁该不该升级为架构修复"这个判断,没有任何活着的角色在做**。补丁因此成了"默认通过"。这不是能力问题,是**治理在线率问题**。

这就是为什么会"修 A 引出 B、同一处修两次":不是工程师不行,是**没有守门人逼着每个补丁回答那道题**("一次性补丁,还是触及共性根因?")。

## 四、评估:要不要全方位重构?

**与 CEO 一致:不做停机大重构。** 理由相同(系统每天在产、正确模式已知、大重构本身最不稳)。HR 补两点关于"顺序"和"该谁干"的判断:

1. **先治"归属"再治"代码"。** 哪怕一行代码不改,只要先把 owner 指派清楚、把 board/governance 的守门重新上线,补丁堆叠的速度立刻下降——这是**当天就能止血、零代码风险**的动作,该最先做。
2. **架构性修复要落到具体角色的 KPI,否则又会悬空。** 锁统一、抽 domain 是 P0/P1 的工程活,但若不指派 owner 并纳入验收,半年后会变成"又一个写了没铺开的好模式"(resource-locks 就是前车之鉴)。

| 弱点 | 处置(同 CEO) | HR 补充:谁负责(owner) |
|---|---|---|
| A 进程 + B 锁(合并) | 架构性解决 · **P0** | 工程 owner:`it_engineer`/`worker_code` 实现;运行态切换/回滚:`repair`;**新设/明确"运行态健康 owner"**(见第六节归档建议) |
| C 识别/守卫 | 收敛成单一 classifier · **P1** | 工程 owner:`worker_code`;决策守门:`governance`(定"安全默认"规则) |
| D 代码边界 | 渐进抽 domain · **P1→P2** | 工程 owner:`supervisor` 排期 + `worker_code` 实现;评审闸:`board`/`governance` |
| 飞书去重 / 个别正则 | 补丁够用,明确收口 | `repair` 记账,纳入 C 收敛;**注意去重粒度变粗的新风险** |

## 五、进化路线(带"谁负责",这是 HR 的交付重点)

原则同 CEO:**每阶段主动作是"收敛/替换/删除",不是"新增";先回归测试再切换,留开关可回滚。** HR 在每阶段钉上 owner 与验收人——没有 owner 的路线图等于没有路线图。

**阶段 0(今天就能做 · 零代码 · 止血归属):**
- **重启守门(最高优先):** 给老板一个决定——董事会要么按收紧后的条件重开(只"作用域命中+高置信"才开),要么明确指派 `governance` 在每次架构类补丁合入前做一次"补丁/架构"判定。**当前 board 关、governance 只夜里跑 = 守门真空,这条必须当天闭掉。**
- 建《架构债清单.md》,把今天 9 条逐条挂到 A/B/C/D,标"一次性补丁 / 触及共性"。owner:`governance`;录入人:`repair`。
- 验收:此后每个 repair 工单都带"补丁 or 架构"字段且挂到四弱点之一。

**阶段 1(P0 · 进程与锁统一):**
1. 把 `resource-locks.js` 的 lease 模型定为**唯一锁标准**,迁移 engine-slot / runner-type / queue-lease / active-ceo-task 四套旧锁(共 5 套→1 套)。
2. 建单一 `process-supervisor`,收编 `server.js`/`ceo-worker.js`/`watchdog-daemon.js` 三处自愈为一处权威;引擎/worker 主动上报"活着+第几步"。
- owner:`it_engineer`+`worker_code` 实现,`supervisor` 排期,`repair` 负责运行态切换/回滚。
- 验收(同 CEO):slot 崩溃即回收;活 engine 在慢心跳/重启下不误杀;worker 掉 5s 内拉回;卡 running 自动恢复。

**阶段 2(P1 · 识别收敛 + 边界分层):**
1. 单一 `recognition` 模块:projectId/intent/Starlaid 守卫/董事会触发全走它,输出 `{projectId, intent, 置信度, 命中作用域}`,**默认安全侧 + 留痕**;删散落正则。
2. `server.js` 抽 `queue`/`engine`/`taskboard` domain,spawn 只剩一份(顺带消灭 `unref()` 漏写)。
- owner:`worker_code` 实现,`governance` 定"安全默认"规则,`board` 复核。
- 验收:常规修 bug 不再开董事会;Starlaid 误判保持为零且真实 Starlaid 仍阻断;node-bin 类修复只改一处。

**阶段 3(P2 · 出口抑制反向治理):**
逐条审计去重/节流/跳过/开关,每条回答"上游能不能不产生";能从入口消除的,出口抑制降级为纯兜底。**特别复查飞书指纹去掉 reason 后的合并是否漏掉了该告知的失败。**
- owner:`quality_ops` 审计,`repair` 落地。

**"在进化而非堆补丁"的周度指标(同 CEO,HR 加 owner 维度):**
- 复发率(同根因再提单)↓ — 看人:`governance`
- 一个修复要改几处文件 →1 — 看人:`supervisor`
- 出口抑制条数净增量 ≤0 — 看人:`quality_ops`
- 架构债"已收敛/总数"↑ — 看人:`governance`
- **守门在线率 = 架构类补丁中"经过 board/governance 判定"的占比,目标 100%** — 看人:`hr_manager`(本职:边界/在岗)

## 六、HR 专项:涉事角色边界审核 + 归位

按 HR 边界审核必备格式,只审与今天问题直接相关的角色:

**`repair` 维修员**
- 我做什么:卡死/进程/重启/权限/服务修复;问题模式→根因→预防沉淀。
- 我不做什么:不做架构裁决(那是 board/governance)。
- 混淆点:今天事实上承担了"补丁要不要升级为架构"的判断,但这超出其边界且无人复核 → **补丁默认通过**。
- 归位:repair 继续救火 + 录架构债;"补丁/架构"判定权归 `governance`,争议升 `board`。

**`board` 董事会(deepseek/glm52/gpt55/opus48)**
- 我做什么:对重要架构指令逐轮挑刺、裁误判风险。
- 我不做什么:不该被常规 bug 修复触发(根因 C)。
- 混淆点:触发条件过宽 → 误触发 → 紧急关停 → **守门真空**。
- 归位:触发改"作用域命中+高置信"双条件后重开;关停期间其职责**必须显式转给 `governance`**,不能悬空。

**`governance` 监管**
- 我做什么:反复失败/重大漏洞→复盘→立规(§17)。
- 我不做什么:不写业务代码、不救火。
- 混淆点:只在凌晨定时跑,**白天不在岗**;board 关停后无人接替逐事守门。
- 归位:承接 board 关停期的"补丁/架构"逐事判定;持有《架构债清单》。

**`orchestrator` CEO / `supervisor` 主管 / `it_engineer` / `worker_code`**
- 边界本身清晰(拆方向/排期/版本发布/实现),今天问题不在其边界混淆,而在**缺少"运行态健康 owner"和"单一事实源评审闸",导致他们各自的产物无人在架构层统合**。

**未闭环缺口(HR 口径):**
1. **无"运行态健康"角色**——自愈散 3 处的组织对应物。建议补一个"运行态 owner"职责(可挂到 `repair` 扩边界,或在 `process-supervisor` 落地后由其代码化承载),不急于新建 agent。
2. **守门在线率无人负责**——board 关、governance 夜跑时,补丁堆叠无监控。建议纳入 `hr_manager` 周度边界审核指标。
3. **"单一事实源"无评审闸**——第二份 spawn/锁/识别实现无人拦(`unref()` 漏写即后果)。建议作为 §16 硬化项交 `quality_ops`(grep 到第二份实现即告警)。

**后续 HR 招聘/归档建议(遵守"先查重、能复用不新建"):**
- **不新建 agent。** 三条缺口都用"扩现有角色边界 + 加治理指标"覆盖,不扩 agent 编制。
- 归档项(本就是 INDEX.md 列的待归档角色):`worker_code`/`worker_narrow`/`reasoning_architect` 等补全 `agent.json`,顺带把"运行态 owner / 单一事实源评审"职责写进契约。
- 高风险动作(写核心引擎/路由/锁的实现)按既有分级:输出规格卡 → 老板审批 → 再交实现角色。本报告不触发任何 agent 创建。

## 七、防止补丁堆叠的三条护栏(HR 落成"角色+在岗"版)

CEO 报告给了三条机制护栏。HR 把它们落到"谁在岗、何时触发":

1. **逐补丁自答题 + 登记**——每个 repair 完成必答"补丁 or 架构",`governance` 持账。(机制已有,补"在岗判定")
2. **架构不变量清单,新代码对照评审**——持资源必有 `lease+ownerPid+starttime+心跳`;分类/触发必有 `置信度+作用域+安全默认+留痕`;核心机制只许一份实现。守门人:`board`/`governance`。
3. **单一事实源 grep 闸**——spawn/lock/recognition 各只许一个模块导出,评审/CI grep 到第二份即打回。落地人:`quality_ops`(§16 硬化)。

**HR 的元护栏:守门必须始终有人在岗。** 任何治理角色被关停,其职责必须显式转交并记录,绝不允许"关了就没人管"。这是今天最大的教训。

## 八、给老板的话(归属判断)

- **好消息**:这不是能力问题。正确模式已写出(`resource-locks.js`)、每天跑通大量真实任务、Starlaid 边界和部分识别其实已修好(别被复盘里的夸大吓到)。
- **真问题**:**治理"下线"了而没人接班**——董事会关停、监管夜跑,白天补丁无人把关,于是"补丁默认通过",这才是"混乱机械"的成因,不是工程师爱打补丁。
- **建议当天拍三件零代码的事**:① 重开董事会(收紧条件)或明确指派 governance 逐事守门,堵住守门真空;② 建《架构债清单》并把今天 9 条挂进去;③ 把"守门在线率"纳入 HR 周度审核。
