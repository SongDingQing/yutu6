## 一、确认的问题清单

### A. 交接文件夹机制 v1 的设计冲突(按严重度排序)

**A1【最严重】queue entry goal ≤200 字指针化会掏空所有"以文本为判据"的运行时机制,直接削弱假完成拦截。**
依据(全部读码核实):
- done-gate 硬回归触发源只看 `vars.goal/acceptance/bounds/task/changed_files`(`shared/engine/done-gate.js:856-868` `taskHardRegressionTriggerText`)。goal/acceptance 变成指针后,真队列合并任务不再触发 `queue_merge_integrity` 硬门——06-23 第四处教训好不容易把触发源收敛到任务信封文本,信封文本再被抽空,硬门就成了空转。
- 结构化验收表的 requiredRows 从 `vars.acceptance` 文本解析;acceptance 指针化 = gate 无行可校 = 全放行。设计 3.4 说"验收表匹配逻辑不动",但 done-gate 读的是 **ctx vars**,不是 task.md——设计通篇没有"engine 侧读 task.md 水合 ctx"这一步(3.3 步骤 2 只写了 meta.status=claimed)。
- runner 可写性兜底启发式读 ctx 文本(`shared/engine/cli-runner.js:546-555` `nodeNeedsWritableRunner` → `deliveryEvidenceRequiredFromText(ctx.goal, ctx.acceptance…)`),未声明角色的落盘任务会被误判只读、派给纯文本 GLM——精确重演 experience.md 2026-06-22/06-25 两条教训。
- 资源锁域推断在 **worker claim 时读 queue entry 文本**(`projects/控制台/resource-locks.js:158-190` `inferResourceDomains` 拼 `task.goal/message/...`),发生在 engine/ctx 存在之前;entry 只剩一句话时推断必然失败,现状是兜底 `write.push('console-project')`(:189)→ all_blocked 更糟;若叠加问题#2 的"无声明不上锁"修法,则变成静默失去互斥——两个方向都是坑。
- 董事会触发 `BoardReview.shouldRunBoardReview`、直派 `directQueueForGoal`、project-guard 全部同理吃 goal 文本。

**修正建议**:指针化只发生在 `buildEnvelope` 的 **prompt 层**;queue entry 与 ctx 保留全文(entry 体积从来不是报告量化出的瓶颈,信封逐字重复才是)。若坚持 entry 瘦身,必须先给锁域/硬回归/writable/董事会补结构化声明字段(resource 声明、task_domain、assignee)并有 .js 消费点 + 回归(experience.md:193-195「declared-but-not-honored」的镜像风险:文本抽走了、结构化没接上)。

**A2 回退路径在阶段 2 后不成立。** 3.6 说"读 task.md 失败 → 自动降级现状拼接,任务不失败"——但阶段 2 起 entry 只有 ≤200 字 stub,task.md 缺失时**没有内容可拼**,"降级拼接"拼出来的是一句话任务,模型照做产出垃圾,比硬失败更糟。修正:要么 entry 保全文(同 A1),要么 task.md 缺失/指纹不符 = 硬失败 + 事件,删掉"任务不失败"的承诺。

**A3 目录键 `<taskId>` 不成立:taskId 是每 attempt 重新生成的。** 实证:`cr-<Date.now()>-<queueId>` 每次 handle 重生成;experience.md:103 现场证据"progress_task=cr-…-9c01e9e2(旧) / taskId=cr-1782171842664-9c01e9e2(新)"即同一 queueId 两个 taskId。发起方**入队时刻根本没有 taskId**,却要写 `<taskId>/task.md` 并把路径放进 entry——自相矛盾;retry 会产生多个交接夹,meta 状态机跨 attempt 断裂。修正:按 `queueId`(必要时 + rootQueueId)作目录键;meta 增加 attempts 数组;状态机显式允许 `reported/rejected → claimed` 再入(否则测试 1 的"非法迁移拒绝"会把每次 retry 都拒掉)。

**A4 report.md 单文件多写者竞态。** 同一任务的 implement、review、failover 候选(`-foN` 目录)、董事会扇出(`board-review.js:840` runNodeAsync 并发)都会在收尾写 report.md → 相互覆盖;review 合同"读 report.md 逐行复核"可能读到自己上一轮的产物。且改动点 4 挂在 `resultFromRunnerResponse`(cli-runner.js:840-867)层——该函数对**失败候选也执行**(result.md 总是写),照设计会把 meta 误置 reported。修正:`report-<node>-<attempt>.md` 命名 + 只在 result 无 fail 时写 + meta 存指针数组。

**A5 steer 落点自相矛盾 + 指纹口径未定义清楚。** 3.2 的 task.md 模板含"## steer 追加区",3.1/3.3-7 又说写独立 steer.md;若指纹按改动点 10 的"对 task.md 内容哈希"(整文件),steer 一追加指纹即碎,协议门必打回。顺带确认一个**现网活雷**:现状 steer 在检查点直接 `ctx.goal +=`(`projects/控制台/engine-runner.js:577`),而协议门按 `vars.goal` 重算指纹对比 snapshot(`shared/engine/protocol-gate.js:102-121`,消费点 `done-gate.js:1230` + `hardening-hooks.js:52`)——凡被 steer 过且走完成协议校验的任务,收口时必报"规格指纹不一致"。设计方向(steer 出 goal)是对的,但落地必须:指纹 = 「目标+验收两节的规范化哈希」且加正反测试,steer 只进 steer.md;这个现网雷可单独 A 类先修。

**A6 tool-harness 会漏在 kind 二分之外。** 预展开规则按 `runner.kind=cli` / `openai_http` 二分,但 `openai_http_tool_harness` 的 planner 阶段是纯文本模型且与 executor **共用同一 prompt**(`cli-runner.js:869-925` 及 async 版):harness 拿到指针信封,planner 读不了文件只能盲规划。且 failover 候选链跨 kind(codex→zhipu-glm),信封必须逐候选按 kind 重建——这与问题 #17"按 node+attempt 缓存 buildEnvelope"**内部冲突**,缓存键必须含 runnerId/kind,否则指针信封会被喂给纯文本候选。

**A7 meta.json 多写者无守卫。** engine 写 reported/accepted、发起方写 created、sweep/requeue 后新 attempt 写 claimed——原子写只保证单次完整,不保证顺序,last-write-wins 可让状态倒退。参照本次新发现的队列字段清理漂移(见 C5),meta 必须单一 owner + 读-校验-写(带 attempt 号的 CAS 语义)。

**A8 meta 前置判据必须显式声明为 AND。** `status=reported` 是 runner 机械写的旗标,不是证据;若 done-gate 把"meta 通过"当任何既有校验的减免条件,等于给模型侧开了机器可写的绿色通道。设计文本目前是"先读…再跑",应写死"meta 判据只增不减、不通过任何一项即打回",并加负例测试(meta=reported 但验收表缺行仍必须拒)。叠加 #10(engine.js:385 `Object.assign(ctx, out.vars)` 无白名单,已核实)与 C-5 双轨互认期,建议**先做 ctx 白名单(C-2),再切指纹语义(阶段 3)**,顺序不能反。

### C. 与 experience.md 教训的冲突/呼应

**C5【新发现,报告遗漏,建议并入 A 类】** `shared/engine/queue.js` 的 claim(:333-348)/requeue(:271-293)/resume(:232-257)三份清理黑名单**都没清 `pre_engine_wait_heartbeat_at`**,而 ceo-worker 在无 enginePid 时**优先**用它判活(ceo-worker.js:301-303):retry 后新 attempt 若进入 pre-engine 等锁阶段,会继承旧 attempt 的陈旧 pre-engine 心跳、在 handle() 写入新心跳前的窗口内被判死——是「running 进展派生字段在状态迁移边界清理不全」(experience.md:100-103)的第 N 处同族复发,直接证明 #7/#14"字段白名单投影"的必要性;半天可修(三处补删 + 回归)。

## 二、对设计的修正建议汇总(可直接改稿)

1. **指针化分层**:prompt 层指针化(buildEnvelope),**判定层保全文**——queue entry 与 ctx 保留完整 goal/acceptance/bounds;task.md 是信封的去重载体,不是 gate/锁/路由的替代输入。
2. **回退语义改写**:阶段 2 后 task.md 缺失/指纹不符 = 硬失败 + `handoff.fallback` 事件;"降级拼接"只在 entry 仍含全文的阶段有效。
3. **目录键改 queueId**,meta 增 attempts[];状态机补 `reported/rejected→claimed` 再入迁移;meta 写入加 attempt 号 CAS。
4. **report 按 `report-<node>-<attempt>.md` 命名**,只在成功 result 上写;meta 存指针数组。
5. **指纹定义写死**为"目标+验收两节规范化哈希",steer 只进 steer.md;task.md 模板删"steer 追加区";现网 steer 拼 goal 打爆协议门的问题单独 A 类先修。
6. **预展开判定按执行链**(含 harness planner)而非外层 kind;#17 信封缓存键加 runnerId/kind。
7. **meta 判据显式 AND**,加"meta 通过但验收缺行仍拒"负例;C-2(ctx 白名单)排在指纹语义切换之前。
8. **A 类清单调整**:A-1 改"验证+加载隔离";A-3 修法保留但测试口径与节流、progress_task 归属按 B2 修正;A-4 改整键透传;新增"三份黑名单补清 pre_engine_wait_heartbeat_at"(C5)为 A 类第 11 项。
