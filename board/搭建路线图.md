## 阶段 1 · 两条"燃料线"(并行,自包含,不依赖引擎)
> 用户排第一的两项。它们是 agent 的"记忆"和"技能",不依赖编排引擎,可立即开工、并行。

### 1A 知识库点火(§15)
- 补什么:把 `ingest.py`/`query.py` 的 embedding(+图谱抽取)指向**另一台电脑的本地模型端点**(OpenAI 兼容)→ 跑 `knowledge/build.sh` 建 `kb.sqlite`(关系+FTS5+向量+图谱)→ `query.py` 冒烟。**mini 不装 Ollama。**
- 前置:决策 B(端点+模型+维度)。wiki/ 原文 + corpus 已就位。
- 谁做:总管改 ingest/query 接端点(沙箱内);跑 build 由 Codex/机器侧(需能连到那台电脑)。
- 注:embedding 模型+维度**一次定死**,改模型=重建向量索引;那台电脑需在 ingest/query 时可达(否则本机留小兜底 embedder 或走云 embedding)。
- **可信验收**:`python3 knowledge/query.py "玉兔6 的四层组织是什么"` 返回**带出处(documents.path)**的正确答案;增量重建只处理变更文件。
- 解锁:agent 有长期结构化记忆 + GraphRAG;后续监管复盘、用户纠正(corrections 软删除)有处可落。

### 1B 能力库:只补 3 个全局件(41 项目技能不拷)
> 用户定调:41 个旧 skills 是**具体项目用的**,不做基础步骤——**到真正做那个项目时按需拉**。全局只需 3 件:
- **① 全局能力模块索引**:让任何 agent 能查"有哪些模块/能力"(§11 L2 用到才查)。已大体在工作区(`registry.json` + module-discovery),保持/精炼即可。
- **② 环境配置管理**:跨机路径 / .env / 端口 / 密钥位置的统一管理(归 `shared/config/`)。为新机 yutu6 建好。
- **③ 补齐指令稿(instruction-expansion-router · 重建)**:把人随口的指令补全成完整任务。**只绑前门总管 agent = Claude Code CLI**(人↔系统边界;web 端与元宵都对接它、效果完全一致);内部 agent 走精确 task/result 信封,不挂这个扩展。**新机重建,不从旧机拷。**
- 前置:无需大规模拷贝;③ 在新机直接重建。Starlaid 硬排除。
- 谁做:① 总管精炼;② 总管建;③ 总管在新机重建并绑定 Claude Code CLI 总管。
- **可信验收**:任何 agent 能查到模块索引;新机环境配置一处可改;前门 agent 能把一句话指令补成完整 `task.md`,其它 agent 不挂该扩展。
- 解锁:系统有"目录 + 环境 + 前门补全",不被用不上的项目技能拖累。

## 阶段 2 · 编排引擎 MVP(=智能体架构落地 · 最硬核心)
> 把控制台的"spawn 一个 CLI"原料,进化成"按文件协议跑多 agent"的运行时。**可信地基同时织入,不是后补。**
>
> ✅ **MVP 已搭并沙箱自测 PASS(2026-06-18)**:**核心模块 `shared/engine/`**(已从 projects/控制台/ 提升为核心层,与项目解耦)—— 声明式编排(condition/engine)+ 护栏(validate/max_loops/超时)+ human gate + 事件日志(eventlog)+ 任务状态机/attempt(taskstore)+ 信封运行时(cli-runner,机制已用假 CLI 验证)。⏳ 剩:Mac 上用真 claude/codex 验证信封运行时(任务#13)。

补什么(三块一起):
1. **信封运行时**(§3/§4):读 `task.md` → 按 `runners.yaml` 给 agent 绑 runner/模型 → spawn(CLI 子进程,独立 session/system prompt)→ 写 `result.md`。单写主原则、按路径白名单喂上下文(explicit 默认)。
2. **声明式路由执行器**(§18.1):读 `flows/*.yaml`,算 Jinja2 条件边、零 token 选下一节点;先跑通已有的 `review-loop.yaml`(implement→review→done/human)。无边匹配才调一次 planner 节点。
3. **可信地基**(§10 + §18.7,与上面同步落):
   - 任务状态机 + attempt:queued→assigned→running→awaiting_verify→done/failed/paused;重试/换 runner 各算一次 attempt。
   - append-only 事件日志 + cursor(节点 开始/结束/失败/等待)。
   - 验收证据:每个 done 挂测试输出/diff/截图。
   - 护栏:`max_loops` + 墙钟超时;跑前 `validate`(路由可达、引用完整、表达式可解析)。
- 谁做:引擎代码总管在工作区写(建议**扩展控制台 server.js**,沿用 Node 零依赖 + NDJSON);Codex 在机器侧跑/测真 spawn。
- **可信验收**:用 `review-loop.yaml` 跑一个真任务(如"改一段代码+评审"),引擎自动 implement→review→(不过则回 implement,≤max_loops)→done;全过程有事件日志 + 验收证据;dry-run 能拦住坏流程图。
- 解锁:**多 agent 真正按协议接力**——这是蓝图的心脏。

## 阶段 4 · 多智能体自动创建 + CLI 连接(=解锁全部性能)
> 引擎从"跑预定义流程"升级到"按需造 agent + 多模型混用"。

补什么:
- **运行时自动创建**(§11):无预置 agent 就当场造(角色+提示词+模型);5 种协作策略——串行/并行/层级/辩论/流水线。
- **per-agent 模型 + 故障转移**(§18.4):每 agent 独立 provider/模型,混用订阅/API/本地;429/超时自动转下一后端,余量耗尽前预测性切换(LiteLLM/网关可选)。
- **队列两语义 + planner**(§8/§18.1):steering(打断式)/ follow-up(排队式);结构未知时 planner 产子图再交回确定性编排。
- **接更多 CLI/runner**:除 claude/codex,按需接 Gemini/Nous 等;**Peekaboo 已登记为 GUI/桌面控制 runner**(需点击/原生 App/无 API 网页的节点派给它),统一进 `runners.yaml`。
- 谁做:总管写编排逻辑;Codex 机器侧验真多 runner。
- **可信验收**:给一个没有预置流程的任务,引擎能造出 2–3 个 agent 用某策略协作完成,并发失败有兜底;一个 agent 跑 codex、另一个跑 claude/本地,混用成功。
- 解锁:**全部性能**——任意拆分、任意模型、任意策略。

## 阶段 6 · 硬化 + 监管闭环 + 回归(越用越硬/省/可信)
> 有了使用数据(事件日志),开启自我改进回路。持续进行,非一次性。

- **硬化**(§16):质量运营把高重复+确定性序列 → 🦴骨头(脚本)/🦵软骨(参数化 skill)入能力库;可消除的思考 → 💪肌肉记忆(重设计流程)。门槛触发(≥N 次)、可回退、带测试。
- **监管闭环**(§17):反复失败(attempt≥N)/重大漏洞 → meta 复盘(给误区命名)→ 规则/检查项 → 交质量运营硬化成检查脚本或写进 L0/L1。下次事前介入 + 可结合辩论。
- **回归**(§18.7):pass^5(同组任务连跑 5 次全过率)定期回归,验证"文件态记忆"更稳更省。
- **可信验收**:跑出第一条"上报→系统级升级"(某项目硬化下沉成基础路由模块);监管把一次真实反复问题固化成一条检查项;pass^5 报告方差下降。
- 解锁:经验沉淀成结构,token↓ 延迟↓ 确定性↑——**越用越硬、越省、越可信**。
