# 角色编制 · 弹性策略(2026-07-03,拍板 Q1 第一步)

## 0. 老板设计哲学(编制的第一性原理)

**角色 = 专业领域的注意力承接单元。** 设角色不是为了组织图好看,而是为了:

1. **防注意力稀释**:一个角色的提示词/上下文只装一个领域,任务进来时模型的注意力不被无关职责摊薄;
2. **提任务并发**:角色即队列,领域拆开后互不阻塞,可并行消费;
3. **强化知识库领域权重**:角色绑定自己的 read_paths/知识交接文档,领域知识在该角色上下文中权重最高。

**编制原则**:闲置归档;重复任务过多、或某个 skill 复杂到需要独立注意力时自动展开(扩编);扩编由质量运营监督提议。编制是弹性的——角色随任务流量伸缩,不随历史惯性存续。

## 2. 软归档 vs 硬归档(NR11 消费点说明)

**软归档(本步)= 纯文档标记。** grep 确认:`engine-runner.js`(roleMapFromConfig / roleExecMetaFromConfig / routeRunnerForRole)与 `ceo-worker.js`(第 1492 行 route 解析)只读 `runner / execution / label / rollout`,**均不消费 `archived` 字段**。因此:

- 归档角色**路由仍可用**(roleMapFromConfig 照常解析出 runner)——不会破坏任何潜在的隐性依赖;
- 归档的效果 = 进 INDEX 与绩效报告的闲置区,供人和秘书/CEO 分派时避开;
- 依 NR11(无消费点的字段要么有消费者要么明说是文档标记),config 条目内用 `archivedNote` 字段明示这一点。

**硬归档(未实施,待观察期)= 有真实消费者的下线。** 现成杠杆已存在,无需新代码:
- `projects/控制台/server.js` → `disabledQueueAgentIds()`:`agent.json` 的 `status` 以 `disabled|archived|retired` 开头时,该角色被剔出队列 worker 名单;
- `shared/engine/agents.js` → `validateAgent`:对 disabled/archived/retired 状态跳过 role-in-routing 校验(允许从 model-routing 摘除)。

**硬归档触发流程**:软归档满一个观察期(建议 14 天)且期间零任务、零回退链引用 → 质量运营提议硬归档(附证据)→ 老板拍板 → 设置 `agent.json status` 并按需从 model-routing roles 摘除。本步不做。

## 4. 扩编机制(编制的另一半)

**触发条件(满足其一)**:
1. 同类任务 7 天内 ≥5 次重复落在同一现有角色(注意力被稀释的量化信号);
2. 单一 skill 的提示词/知识包膨胀到承接角色 prompt 的约 30% 以上,需要独立注意力;
3. 质量运营绩效巡检发现某领域一次通过率因角色兼职下降。

**流程**:质量运营提议(附事件日志/重复度/通过率证据)→ 老板拍板 → HR 建档(规格卡四要素、hr_manager 查重与边界审核、hr_specialist 注册+smoke)。
**复活已归档角色**:同一流程,但 HR 建档步骤简化为「移除 config 中 archived 字段 + INDEX 移出归档名单」。

## 5. 回滚与风险

- **回滚**:本步全部可逆——删掉 config 三个 archived* 字段 + 还原 INDEX 小节即可,零行为变化。
- **无 env 开关**:软归档不改变任何运行时行为(无代码消费点),保守默认天然成立;硬归档实施时才需要开关/灰度。
- **残余风险**:若有脱离 config 路由、硬编码 `reasoning_architect` 的调用方(grep 未发现),软归档不影响它们;绩效报告目前靠人读 INDEX 归档区,尚无自动出报表的消费者。
