## 一、总体判断

**稳定性:结构性事故链在跑,不是散发 bug。** 系统当前最主要的故障形态是一条可完整复现的放大链:server.js 在事件循环上做 2.5s 级同步操作(spawnSync sqlite3、MB 级同步 tail、定时全量 SHA256)→ watchdog 探活 `/api/health` 超时(357 条重启记录中 202 条 reason=`console-http-failed timeout 2500ms`)→ `launchctl kickstart -k` 整机重启 → 所有 running 进入恢复流程 → 重入队后撞上过粗的 `frontend-public` 写域(3 天 all_blocked 1,402~1,496 起,一个域占近 9 成冲突)→ 锁等待又超过 watchdog 的 12min 无进展阈值(< 引擎自身 15min 节点超时、锁等待合法上限 30min)→ 再次触发整机重启。叠加"判死"没有单一事实源(6 类心跳字段、5 个裁决者、4 套读取优先级、3 套阈值)与异步执行路径根本不刷队列心跳(cli-runner.js:761-774),历史"误杀"不是偶然而是必然。另有两处正在流血的具体事故:codex skill YAML 损坏致 68/126 node.fail 且降级链两跳共享同一损坏目录、文件至今未修;董事会 `compact(base,5000)` 保头截尾把老板任务正文截丢(已实证的正确性事故,worker 靠 brief 暗示侥幸反推出任务)。结论:稳定性问题集中在**基础设施层的判定与调度**,而非业务逻辑层;修掉链条头两环(server 同步阻塞、资源域粗粒度)预计消除大部分表观误杀。

**性能:token 与 IO 双重浪费,均有量化实证。** Token 侧:交接链路是"逐级全文前缀拼接 + 全量 ctx JSON 回放",单条老板任务全链 ≈18 万字节 prompt(粗估 9 万 token),单个 implement 信封 35KB 中 40-56% 是逐字重复(spec_snapshot 把 goal+验收表整体再抄一遍、"秘书补全稿"字样在 review 信封出现 17 次),对模型真正新增信息量不足 15%;而系统内指针式任务(repair 工单)实测信封仅 10-12KB,同系统对照证据已在。IO 侧:单任务稳态 ~10 次文件读写/秒(500ms 轮询 + 2s 心跳 + 5s sweep 三层叠加),事件日志 85% 是 node.output+keepalive 噪声,每个 worker 每次唤醒同步解析 2MB tail,taskstore 全量重写且 vars 双份存储近平方级膨胀,brief.md 已无界追加到 1.9MB 并作为 inputs[0] 传给每个主管任务。failover 每个候选还重建信封+重跑阻塞式 python KB 检索。这些都不需要新架构,靠白名单、指针化、增量 offset、缓存四招可解。

**紧凑度:双份/三份实现已从"债务"演变为"正确性问题"。** 全仓至少存在:cli-runner 同步/异步双轨 ~250 行逐字复制(已发生行为漂移,即心跳缺失)、DEFAULT_ROLE_MAP 双份(21 vs 15 角色)、节点超时三个默认值(600/900/480s)、灰度路由四件套整段复制且 **rolloutBucket hash key 已漂移**(worker 与引擎可能把同一任务哈希进不同桶)、queue 状态迁移三份互不一致的字段黑名单、readJson/writeJsonAtomic ≥5 份(三种原子写变体)、failover.js 硬编码 PROVIDER_RUNNER 与 yaml/config 三处平行维护、optimizer 铁律文本双源、done-gate 与 loop-engineering 收敛校验双实现双执行、6 份 handoff 提案 950 行全部 proposal_only 无一采纳、队列目录 `memory-officer`/`memory_officer` 双命名并存。done-gate.js 1283 行已成正则规则堆并把控制台项目知识硬编码进 shared 层,引擎无法被第二个项目复用。紧凑度的病根是"每次事故加一条正则/一句合同/一份拷贝"的演化路径,需要的是单源化纪律而不是重写。

## 三、【交接文件夹机制设计 v1】

设计原则(吸收 agent-handoff-protocol 案例 + 既有教训):**只指针、不复制状态**(6-19 交接稿腐烂反例);**门禁判据优先结构化**(meta.json 优先于正则);**失败必可观测**(每次指针→内联回退必发事件);**补丁落盘≠运行态生效**(灰度开关必须进热重载指纹,rollout 哈希单实现防桶漂移);**repair 工单是已验证活原型**(指针式信封 10-12KB vs 拼接式 35-47KB,直接推广其模式)。

### 3.1 目录结构(路径约定)

```
projects/<项目>/artifacts/handoffs/
  <from>--<to>/                    # pair 目录,agent id 经 normalizePairId 统一
    <taskId>/                      # 每任务一夹
      task.md                      # 任务稿(发起方写,source of truth)
      report.md                    # 汇报稿(执行方写)
      meta.json                    # 状态机 + 指纹(结构化门禁判据)
      context.md                   # 可选:背景包落盘(秘书→CEO 专用)
      steer.md                     # 可选:运行中追加指令区
```

- pair 命名:`secretary--ceo`、`ceo--supervisor-控制台`、`supervisor-控制台--worker`、`secretary--repair-lead`(后者原地保留 board/repair-tickets 作为该 pair 实现,不迁移)。
- agent id 一律小写连字符(`normalizePairId` 强制,顺手根治 `memory_officer` 双目录问题)。
- 智能体"收件箱"语义:执行方在 `*--<自己>/` 下找任务稿;发起方在 `<自己>--*/` 下收汇报稿。

### 3.2 文件命名与模板

**task.md**(固定分区,顺序即截断优先级——正文永远最前,任何截断先砍附件区):
```
# 任务稿 <taskId>
## 目标(正文)          ← 老板/上级原文,禁止前置背景包
## 边界
## 结构化验收表          ← 完整保留(done-gate 逐字锚点匹配要求,执行方从此复制)
## 输入指针              ← context.md / brief 片段 / decisions.md:行号+80字摘要,只指针
## steer 追加区          ← 运行中引导消息按时间戳追加,不改目标本体
```

**report.md**(执行方写,替代"从 stdout 捞 JSON"的散装汇报):
```
# 汇报稿 <taskId> / <node>-<attempt>
## 结论 JSON 块          ← 与现有结构化输出合同同 schema(receipt/acceptance_table/logic_chain)
## 摘要(≤500字)
## 证据指针清单          ← 工作区相对路径+行号(done-gate.js:116 现有解析规则直接兼容)
```

**meta.json**(结构化状态机,门禁与观测的第一判据):
```json
{ "task_id": "...", "from": "ceo", "to": "supervisor-控制台",
  "status": "created|claimed|reported|accepted|rejected",
  "spec_fingerprint": "<sha256(task.md 正文+验收表)>",
  "created_at": "...", "claimed_at": "...", "reported_at": "...",
  "queue_id": "...", "engine_run": "engine-runs/<taskId>" }
```
状态每次变更同步 emit `handoff.status` 事件(from/to/taskId/status)——失败可观测。

### 3.3 写入/读取时序

1. **发起方**(如 CEO 派单):写 `task.md` → 写 `meta.json(status=created, fingerprint)` → 入队 queue entry,entry 只带 `{ task_doc: "<task.md 相对路径>", spec_fingerprint, 一句话意图 }`,goal 字符串 ≤200 字。
2. **worker claim**:照旧走 queue.claim;engine-runner 读 spec 见 `taskDoc` 字段 → meta.status=claimed。
3. **buildEnvelope**:CLI runner(codex/claude,主链全是)信封 = 一句话意图 + task.md 路径 + fingerprint + 输出合同,由 runner 自己读文件;openai_http 纯文本 runner 由 builder **预展开内联**(复用 fetchKbContext 预取模式,同一文件只内联一次、去重)。
4. **执行方收尾**:写 `report.md` → meta.status=reported;stdout JSON 照旧输出(双写过渡期)。
5. **review 节点**:合同改为"读 `<report.md 路径>` 逐行复核",不再靠 ctx 内联 26KB。
6. **done-gate**:先读 meta.json(fingerprint 一致性、status=reported)作结构化判据,再跑证据指针校验;gate 通过 → meta.status=accepted;打回 → rejected + 事件带原因。
7. **steer**:运行中引导写 `steer.md` 追加区(engine-runner 安全点通知 runner 重读),不再 `ctx.goal +=`,消除 goal 与 fingerprint 漂移。

### 3.4 与现有 queue / engine-runs / done-gate 的关系

### 3.6 env 开关灰度方案

- `YUTU6_HANDOFF_MODE=off|shadow|on`(默认 off)。shadow=照常拼接信封,但同时写 handoff 三件套并发事件——先验证文件产物正确,不影响任何在跑行为。
- `YUTU6_HANDOFF_PAIRS=secretary--ceo,ceo--supervisor-控制台`:按 pair 逐对放开。
- `YUTU6_HANDOFF_ROLLOUT_PERCENT=0..100`:pair 内按 taskId 哈希分桶。**rollout 哈希函数只在 handoff.js 一处实现**,ceo-worker 与 engine-runner 都 require 它(吸取灰度桶漂移教训 memo2 #7)。
- 开关读取时机:每任务入队时判定并**把判定结果写进 spec**(`handoffEnabled: true`),下游只认 spec 字段不再各自读 env——同一任务全链路一致,杜绝"worker 认为开、引擎认为关"。
- 回退:任何环节读 task.md 失败(缺文件/指纹不符)→ 自动降级现状拼接 + emit `handoff.fallback{reason}`,任务不失败。
