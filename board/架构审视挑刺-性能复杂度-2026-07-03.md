## 一、交接文件夹机制:IO 攻击不成立,但清理/一致性攻击成立

**1.1 [不成立] "增加文件 IO"本身不是问题。** 量化:每任务 3-5 个文件 + ~4 次 meta 原子写 + 几条事件,对比稳态轮询 IO 和下述 loop-engineering 单轮 35MB 拷贝是零头。攻击点不在写入量。

**1.2 [确认·严重] 设计 v1 通篇没有 retention 条款,而本系统实证"不在轮转白名单=永不清理"。** 核实:`projects/控制台/server.js:600` `rotateGeneratedArtifacts()` 白名单只有 `['engine-runs','engine-jobs','engine-tasks']`;白名单外的现行反例三个——`artifacts/loop-engineering` 593MB/86 目录、`artifacts/ui-optimize/shots` 424MB、brief.md 1.8MB 无界。新建 `artifacts/handoffs/` 不写清理条款,就是第四个反例。**修正:设计 3.1 节必须补 retention 小节,且不能简单挂 mtime 轮转(见 1.3)。**

**1.3 [确认·正确性洞] 指针悬挂会把"截丢正文"升级成"丢整个正文"。** 核实:engine-runs 轮转策略为 14 天/200 个(server.js:83-84),而 failed 队列现存 247 件、最老 6-19(报告自己承认,已超 14 天)。设计规定 entry goal ≤200 字 + task.md 指针:若 handoffs 纳入同款轮转,老 failed 件 requeue 时 task.md 已删,`handoff.fallback` 降级拼接**拿不到正文可拼**——回退路径在最需要它的场景下是空的。meta.engine_run 与 engine-runs 互指,两棵树轮转节奏不同必悬挂。**修正:handoff 文件生命周期必须与 queue entry 终态绑定(entry 归档时同步归档/删除),回退路径需在 entry 内保留正文压缩副本或明确"指针失效=任务需人工重建"。**

**1.4 [确认] 第四棵树违背紧凑目标,且可复用现有目录。** 核实:每任务文件树现状不是报告说的 3 套而是至少 5 套(queues 1175 文件、engine-runs 1210 文件/192MB、loop-engineering 86 目录、review-loop-fixture、board/repair-tickets 120 件),handoffs 是第 6 套。同时核实:engine-runs 目录名就是队列 cr- id(`artifacts/engine-runs/cr-1783067230375-bcec5a06/` 等),节点目录在其下,`engine-runs/<taskId>/<node>/task.md` 已存在。**修正:task.md/meta.json 直接放 `engine-runs/<taskId>/` 根一级(零新树、轮转现成、engine-runner 已知路径,只需入队时先建目录 + 轮转按 mtime 天然豁免活跃任务);或放 `queues/<agent>/docs/<id>.md` 随 entry 生命周期走。二选一,砍掉 handoffs 新树。**

**1.5 [确认] pair 目录层是装饰。** 发现工作靠 queue entry 的 `task_doc` 路径,没人真扫"收件箱";pair 层只引入 normalizePairId 一套新命名规范——而命名规范漂移正是 memory-officer/memory_officer 病根。**修正:按 taskId 单层,from/to 写进 meta.json 字段。**

**1.6 [确认] 对 board 链路机制退化为"照旧内联+多写文件"。** openai_http 预展开意味着纯文本 runner(board_* 全是)拿不到任何 token 收益,而正文截丢事故恰发生在董事会链路,救命药是 P0 的 compact 修复而非文件夹。**修正:board pair 明确排除出灰度名单,写进 3.6。**

## 二、遗漏的性能热点(两个大的,一个半大的)

**2.1 [遗漏·最大单笔写放大] loop-engineering 快照机制逐轮全量拷贝 changed_files,曾单轮拷入 35MB 事件日志。** 核实:`shared/engine/loop-engineering.js:153-171` `snapshotFiles` 对每个 changed_file `fs.copyFileSync` 同步整拷,`artifacts/loop-engineering/cr-1783000600755-52703e50/round-1/files/` 下实测有 35MB 的 `__engine-events.jsonl` 快照;该目录树共 593MB/86 任务,是 artifacts(1.8GB)第一大户,不在任何轮转名单。报告 #12/#14 讲写放大却只字未提。**修正:快照加单文件体积上限并排除 `*.jsonl`/日志类;loop-engineering 与 ui-optimize 纳入 rotateGeneratedArtifacts 白名单(一行改动)。**

**2.2 [遗漏] ui-optimize/shots 424MB 截图零清理**,第二大户,报告未提。同上修正。

**2.3 [半遗漏] KB 检索阻塞在主路径每节点都跑,不只 failover。** 核实:`shared/engine/cli-runner.js:127` buildEnvelope 每次调 `fetchKbContext`,:55 spawnSync python3(超时 4s)。报告 #17 只提 failover 候选重复检索;实际主路径每节点、每 attempt 都阻塞一次 python+sqlite,而 goal 在任务内跨节点不变。**修正:按 taskId 缓存一次,砍掉 N-1 次调用(与 #17 的缓存合并成一条)。**

**2.4 [表述校准] "#1 定时全量 SHA256"夸大:** 核实 server.js:1091-1102,`safeHashFile` 对 >2MB 文件只哈希 size+mtime,内容读取仅 ≤2MB 文件,public 共 35MB/266 文件——同步读上限约 35MB,方向对、量级需改口。另:"2s 心跳"实测为 WORKER_HEARTBEAT_MS=5000、sweep=5000ms、no-progress poller 500ms(ceo-worker.js:2133-2171),"~10 IO/s"估算偏高,不影响结论。

## 三、过度设计清单(建议砍/降级)

1. **meta.json 五态状态机 + handoff.status 事件 + done-gate 前置判据(3.2/3.4)——最大的一处。** 与 queue entry 现有磁盘状态机(queued/running/done/failed/canceled/paused)逐态平行:created≈queued、claimed≈running、reported≈report.md 存在、accepted/rejected≈done-gate 结论。报告全文主题是"多份实现漂移",v1 却亲手新造第二个状态机。**修正:meta.json 只留 fingerprint + 双向指针,状态从 queue entry + 文件存在性推导;handoff.status 事件砍掉。**
2. **三维灰度开关(MODE×PAIRS×ROLLOUT_PERCENT,3.6)。** 读失败自动回退已兜住风险;PERCENT 分桶是重建刚坑过自己的 rollout 哈希机械(#7)。**修正:留 MODE+PAIRS 两维,砍 PERCENT 及配套 tests/handoff-rollout.test.js。**
3. **steer.md + "安全点通知 runner 重读"(3.3-7)。** 为低频功能新开信令通道,并造成 steer 存储(queue)与消费(文件)双路径——新漂移点。**修正:steer 留在 queue entry,下一节点边界重渲染进 prompt,不新增文件。**
4. **scoreCandidates 的"重排"半边(四-Routerly)。** 跳过有 quota-degrade 状态的 runner 是一条 if;按质量分重排是给 3 天约 200 任务的单机系统建预测路由,数据撑不起。**修正:留"跳过",砍"重排"。**
5. **#11 "done-gate 校验 receipt.changedFiles"称机器可核边界——虚假安全感。** changedFiles 是模型自报字段,防不了越界写;真边界需 git status/fs diff 级核对。**修正:改为 gate 侧跑 `git status --porcelain` 比对允许区,或把文案降级为"自报审计"。**
6. **迁移编排 4 阶段 ~3-4 周(3.8)偏重。** 自动回退在手,shadow 与首 pair 灰度可合并。**修正:压缩为两阶段(阶段 0 即刻;阶段 1=首 pair 直接开+回退观测),并给 shadow 双写数据定删除时限。**

## 四、修正后建议的落点(一句话版)

交接机制保留"指针化+正文优先截断+fingerprint"内核,但:寄生在 engine-runs/<taskId>/ 或 queue docs 下(不建新树)、砍 pair 层与五态 meta、生命周期绑 queue entry、board pair 除名;A 类清单新增两条半天级快赢——loop-engineering/ui-optimize 纳入轮转白名单 + 快照排除 jsonl(立收 1GB 磁盘与最大单笔写放大)、fetchKbContext 按 taskId 缓存。
