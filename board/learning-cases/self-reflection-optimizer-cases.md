## 初始原则 · 证据驱动的自动优化边界
- 来源: ~/.codex/skills/self-reflection-optimizer/SKILL.md
- 场景:老板点名某个模块做自省优化,或自动优化洞察沉淀了新的优秀案例。
- 现象:如果只说“优化一下”,系统容易把争议改动也自动做掉,或把低风险小修拖给主人确认。
- 根因/判断:自省优化必须先把问题、证据、收益、风险和验证拆开,再决定自动执行还是交给主人拍板。
- 改法:将每个缺点分类为 `auto_execute`、`owner_decision` 或 `defer`;低风险高收益项自动改并验证,争议/不确定/影响现有功能项只生成决策项。
- 验证:tests/self-reflection-optimizer.test.js 校验 skill、触发器、学习案例和注册表。
- 可复用原则:任何自动优化都要先有证据和风险分级,再决定执行权归属;“挑刺”不是直接重构的授权。

## 2026-07-01 · 页面智能体 token 预算先观测后压缩
- 来源: projects/控制台/artifacts/self-reflection-optimizer/page-agent-token-architecture-20260701.md
- 场景:对页面智能体交互逻辑与 token 使用效率整体架构做自省优化。
- 现象:页面能展示模型用量,但秘书/CEO 默认背景包没有 token 预算估算;学习案例越成功,默认上下文越可能膨胀。
- 根因/判断:用量面板观测模型消耗,但缺少“哪段默认上下文导致消耗”的入口;直接压缩上下文会影响智能体判断,应先加可观测预算。
- 改法:给 `secretary-tools.js` 增加 `context_budget`,在 `context-text` 顶部输出预算和 top sections;争议性的上下文压缩、UI 展示、任务级 token 归因交给主人拍板。
- 验证:node tests/secretary-context-budget.test.js; node tests/llm-usage-safety.test.js; node --check projects/控制台/secretary-tools.js。
- 可复用原则:多智能体系统引入更多背景/案例前,必须先给默认上下文做预算观测;没有预算观测时,不要自动压缩关键上下文,先把重段落暴露出来。

## 2026-07-01 · 自省触发按案例条目去重
- 来源: projects/控制台/artifacts/self-reflection-optimizer/page-agent-token-trigger-dedupe-20260701.md
- 场景:新增优秀案例后,`self-reflection-trigger.js` 需要继续触发自省优化,同时避免案例库文件其他段落更新导致同一案例重复入队。

## 2026-07-01 · 案例事件写入要有并发模拟
- 来源: projects/控制台/artifacts/self-reflection-optimizer/ui-optimizer-self-reflection-a41a1743-20260701.md
- 场景:董事会复核 `ui-optimizer` 自省优化时指出 `learning_case.appended` 事件写入存在并发风险。
- 现象:仅给事件补 task/queue/root 字段还不够;如果写入端仍是裸 JSONL append,多个优化循环同时收尾时可能产生交错、重复或丢元数据,导致后续审计链路不稳定。
- 根因/判断:案例事件是自动优化触发后续治理的边界记录,它既要有字段 schema,也要有写入原子性和模拟验证;没有并发测试的事件补丁容易静默失效。
- 改法:把 `ui-optimizer` 的 `learning_case.appended` 写入收敛到专用 helper,用 lockdir 包住 append,并补 `tests/ui-optimizer-event-writer.test.js` 模拟多次写入后逐行 JSON 解析。
- 验证:node --check shared/agents/ui-optimizer/append-learning-case-event.js; bash -n shared/agents/ui-optimizer/loop.sh; node tests/ui-optimizer-event-writer.test.js; node tests/learning-cases-policy.test.js。
- 可复用原则:凡是会触发后续 agent/队列治理的事件写入,不能只验证“字段存在”;还要有并发或重复写入模拟,证明 JSONL 行完整、关键元数据保留、锁或幂等机制生效。

## 2026-07-01 · 内部队列目录不得污染 agent 发现

- 场景:审视多智能体交互、职责边界和并发处理时,核查 `artifacts/queues/` 与 `configuredQueueAgents()` 的一致性。
- 现象:`_organize-idempotency` 和嵌套 `queues` 是内部/异常目录,但原先的磁盘队列发现只按安全字符过滤,会把它们当成 queue agent 暴露给 UI 和 worker 扫描。
- 根因/判断:文件式队列天然会把“状态目录”和“角色队列目录”放近,如果没有明确的 agent-dir predicate,内部目录会逐步污染队列视图、active root 扫描和职责边界判断。
- 改法:给 server 与 ceo-worker 增加队列 agent 目录过滤,排除 `_` 前缀和 `queues/bulletin` 内部名;新增 `tests/queue-agent-discovery.test.js` 锁住兼容别名 `memory-officer` 不被误伤。
- 验证:node tests/queue-agent-discovery.test.js; node tests/agents-check.test.js; capability registry JSON validation。
- 可复用原则:多智能体系统从磁盘自动发现 agent 时,必须有明确的“什么不是 agent”规则;过滤内部目录优先于清理历史数据,避免修复过程破坏可追溯记录。

## 2026-07-01 · 协议 gate 升级必须同步 prompt 与测试夹具

## 2026-07-01 · WebUI 抽屉语义要闭合焦点链
- 来源: projects/控制台/artifacts/self-reflection-optimizer/console-webui-self-reflection-d7a11626-fa38955b-20260701.md
- 场景:控制台 `new-api` 网关页面用详情抽屉展示最近调用明细,同时 CEO brief 要求系统检查 WebUI 事件流状态语义、危险动作状态和 a11y name/role/value 同步点。
- 现象:详情抽屉只切换可见/隐藏状态,没有声明 `role=dialog`、`aria-modal`、标题归属,打开时焦点不进入抽屉,关闭时不回到触发项,键盘用户也不能用 Escape 退出。
- 根因/判断:抽屉是模态交互,不能只做视觉覆盖层;打开、关闭、键盘退出和读屏语义必须形成同一条状态链,否则 UI 看起来能用,但键盘和辅助技术状态会断裂。
- 改法:给抽屉补 dialog/modal/label 语义,打开时保存触发焦点并聚焦关闭按钮,关闭时恢复焦点并恢复 `aria-hidden`,补 Escape 关闭;跨到 body inert、焦点陷阱或 URL hash 行为的进一步变更列为主人拍板项。
- 验证:node tests/newapi-a11y.test.js; node --check tests/newapi-a11y.test.js; node --check tests/run.js; node 解析 `projects/控制台/public/newapi.html` 内联脚本;Peekaboo 截图 `projects/控制台/artifacts/self-reflection-optimizer/peekaboo-console-webui-newapi-fa38955b-20260701.png`。
- 可复用原则:任何被视觉设计为模态的 WebUI 抽屉/弹层,都必须同时闭合 role/name/value、焦点进入、焦点恢复和 Escape 退出;会改变全局页面 inert、历史记录或后端语义的加强项必须先降级为 owner_decision。
