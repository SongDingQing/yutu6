# 学习案例库

用途:沉淀自动优化、董事会评审、维修复盘、UI/架构反复问题中的可复用案例,让秘书、CEO、董事会后续决策时先读案例,不断提升判断质量。

## 写入原则
- 每个案例必须有来源路径或事件证据,不能只写观点;自动优化产生的案例还必须保留 `projects/控制台/artifacts/engine-events.jsonl` 中的事件类型或事件筛选线索。
- 每个案例按「来源 -> 场景 -> 现象 -> 根因/判断 -> 改法 -> 验证 -> 可复用原则」记录;`来源`、`验证`、`可复用原则` 三项不得省略。
- `可复用原则` 必须写成能指导后续不同项目/页面/agent 决策的通用判断,不能只复述本次任务结果。
- 只记录可复用认知,不堆流水账;一次性任务结果只放原 artifacts。
- 不写密钥、token、cookie、验证码、私钥。

## 当前案例文件
- `ui-optimization-cases.md`:UI 自动优化和办公室/工作区视觉交互案例。
- `self-reflection-optimizer-cases.md`:自省优化的挑刺、分类执行、主人拍板和案例触发经验。
- `agent-harness-research-cases.md`:agent harness、研究方法、工具循环、质量监管与多智能体协调的可复用案例。

## 使用要求
- 秘书:处理老板新指令前,把相关学习案例作为背景之一,避免重复遗漏;若引用了案例,在转交 CEO 的信封中写 `参考案例: board/learning-cases/<file>.md#<案例标题或原则>`。
- CEO:做项目路线和架构判断时,引用相关案例原则,不要只看当前任务表面;派单 brief 的验收或边界中保留 `参考案例:` 行。
- 董事会:事前评审时检查当前方案是否违反已有案例原则,并在评审输出中写 `参考案例:` 或 `参考原则:`;新的高价值评审结论补成候选案例。
- 自动优化循环:每轮或收尾追加案例摘要后,必须同步写事件日志,事件类型固定为 `learning_case.appended`,以便追踪哪些优化轮次产生了案例。
- 自省优化触发:新增优秀案例后,调用 `projects/控制台/tools/self-reflection-trigger.js --source <案例文件> --module <模块名>` 入队秘书;脚本默认选取最新 `##` 案例条目并按案例条目哈希去重,避免案例文件其他段落更新时把同一案例反复触发。
- 前门链路复查:涉及秘书入口、CEO 路由和主管派发的治理任务,案例或状态证据必须保留 `secretary -> CEO -> supervisor` 的 taskId/queueId 线索。
