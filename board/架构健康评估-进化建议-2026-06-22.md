- 角色:CEO(总指挥)· 监管/复盘
- 性质:**只复盘、不改代码**。给老板看趋势与方向。
- 范围:控制台(console)。Starlaid 全程硬排除。

## 一、一句话结论

**系统不需要"推倒重来"的大重构,但已经到了必须"收敛"的拐点。**
今天暴露的 8 类问题不是 8 个孤立 bug,而是 **4 个架构弱点** 反复发作的不同表象。继续逐个打补丁,只会让弱点被掩盖、补丁互相打架(今天已经出现"补丁 A 引出问题 B")。建议:**不做一次性大重构,做"有方向的渐进式收敛"——把已经被验证做对的那一个模式,定为全系统标准,逐步把旧子系统迁过去,并立三条护栏防止补丁继续堆叠。**

## 三、系统性根源:8 个表象 → 4 个共性弱点

### 根因 A:进程生命周期靠"推断",没有"声明 + 校验"的监督树
**对应表象:2 detached 卡死 / 3 worker 不持续 / 5 孤儿误杀活 engine。**

现状:进程用 `detached:true + unref` 在 `server.js`、`ceo-worker.js`、`engine-runner` 多处各自 spawn;是否"该活/该死"靠 **裸 PID + `kill -0` + `ps` 命令字符串 + 时间戳** 拼出来猜。

为什么必然出问题:这是"猜",不是"知道"。任何边界条件都会翻车——
- env / node 路径缺失 → 子进程静默死,没人知道(问题 2 的真因);
- 心跳更新慢(引擎在长计算)→ 被当成卡死误杀(问题 5);
- 重启交接 / PID 复用 → 把活的当孤儿、把孤儿当活的(问题 5);
- 没有真正的 supervisor,worker 存活靠定时器 + watchdog 两套兜底(问题 3)。

**真正缺的:一个把"期望状态"和"实际状态"分开、持续 reconcile 的监督者(supervision tree)。** 现在的自愈逻辑散在 `server.js` / `ceo-worker.js` / `watchdog-daemon.js` 三个文件、互相重叠,谁都不是权威。

### 根因 B:状态一致性靠分散文件锁 + 优雅释放,缺统一的"租约 = 唯一真相"模型
**对应表象:1 slot 锁泄漏 / 4 CEO 队列不消费(卡 running)。**

现状是**两套并存的锁模型**:
- **旧模型**(engine-slot 锁、queue-claim、worker-pid):裸 PID + 长 TTL(slot 锁 2 小时)+ 优雅释放。acquire 和 spawn 之间一崩溃 → 锁文件留 2 小时;handle 在 finish 前崩 → 队列项永远卡 running;没有中央 reaper,worker 死了就没人清。
- **新模型**(`resource-locks.js`,刚写的):lease(5 分钟)+ ownerPid + 进程启动标记(starttime)+ 心跳续租 + 协调器内 sweep。**这套是对的。**

弱点:正确的模式只用在一个子系统,其余还停在旧模式。**不同子系统不同失效行为**,就是一致性 bug 的温床。

### 根因 C:识别/守卫用层叠正则,无置信度、无作用域确认,默认动作过激
**对应表象:6 董事会误触发 / 7 识别过宽 + Starlaid 误判。**

现状:`keywordProjectId` / `notifyKeywordName` / Starlaid 守卫 / 董事会触发,全是 `if(/正则/.test(s))` 层叠 + 兜底 fallback。例如:
- 项目归类规则里 `queue|worker|队列` 这种**通用词**直接归"控制台",别的项目提一句就误归;
- 董事会触发**只要出现"队列/锁"关键词 + 一个动作词("修/改/fix")** 就开会——常规 bug 修复也触发(问题 6);
- Starlaid 英文动作词没单词边界,把 `buildSecretaryEnvelope` 里的 `build` 当成"主动构建 Starlaid"(问题 7,同一根因修了两次)。

**本质:在用关键词匹配做本该由"分类器 + 置信度 + 作用域校验"做的决策,而且默认动作是激进的(默认开董事会、默认阻断、默认归类)。** 错一次代价高,又没有"不确定就走安全默认 + 留痕"的设计,所以一遇新文本就误判、就得再加一条正则。这是补丁堆叠最严重的一块。

### 根因 D:代码层职责未分层,核心机制 copy-paste,补丁必须多点同步
**对应表象:8 前后端边界乱(+ 放大上面所有问题)。**

`server.js` 2645 行,把 **HTTP 传输 + 队列协议 + 引擎生命周期 + 任务板渲染** 全揉在一起;同一段 spawn 逻辑在 `server.js` 和 `ceo-worker.js` 各写一遍——所以今天 node-bin 修复要**改两处**。`ceo-worker.js` 2897 行同样混。

后果:一个 bug 多处改、容易漏改、难测试、失效域不清。**没有领域层(domain layer),核心机制没被抽成"单一实现"被各处复用,而是复制粘贴。** 这让前面 A/B/C 的每个修复都被迫"多点打补丁",进一步加速堆叠。

### (附)一个贯穿性的坏味道:在"出口"抑制,而非在"入口"消除
飞书刷屏(问题 9)用指纹去重硬扛;董事会误触发先收紧再 config 关停。**很多补丁是在出口做抑制(去重/节流/跳过/关开关),而不是在入口消除错误来源(上游误判 + 重复触发)。** 出口抑制越多,系统越像"混乱机械"——这正是老板说的那个味道。

## 四、评估:要不要全方位重构?

**不要做"停机大重构"。** 理由:① 系统正在高频生产使用(每天几十个真实任务跑通);② `resource-locks.js` 证明团队已经知道"正确模式"长什么样,不是认知缺失,是**没铺开**;③ 大重构本身是最大的不稳定来源,与"越来越稳"目标相悖。

**该做的是"渐进式架构收敛":** 按弱点分级,该架构性解决的别再打补丁,补丁够用的就明确收口、别假装是架构。

| 弱点 | 处置 | 优先级 | 理由 |
|---|---|---|---|
| A 进程生命周期 + B 状态一致性 | **架构性解决(合并做)** | **P0** | 卡死/误杀/不持续/泄漏/不消费 五类问题的共同根。`resource-locks.js` 的 lease 模型已验证,把它定为唯一标准、迁移 engine-slot/queue-claim/worker-pid,并建一个真正的 supervisor 收编三处自愈逻辑。 |
| C 识别/守卫 | **半架构(收敛成一个模块)** | **P1** | 停止到处加正则。统一成单一 classifier:输出 `{projectId, intent, 置信度, 命中作用域}`,默认走安全侧(不确定→不开董事会/不阻断/标"待确认"),全部留痕。董事会触发改为"作用域命中 + 高置信"双条件。 |
| D 代码边界 | **架构性但可渐进** | **P1→P2** | 抽 domain 层:`process-supervisor` / `lock`(已存在,扩容)/ `queue` / `recognition` 各一个单一实现模块,`server.js` 退回薄传输层。禁止 spawn/lock/识别逻辑再被 copy-paste。 |
| 飞书去重指纹 | **补丁够用,明确收口** | P3 | 保留,但记一笔:真正的修复在根因 C(减少上游误触发),去重只是兜底。 |
| 个别正则边界 | **补丁够用,纳入 C 收敛** | P3 | 不再单独加,统一进 classifier。 |

## 五、进化路线(让系统越来越稳,而非补丁堆叠)

原则:**每个阶段的主动作是"收敛 / 替换 / 删除",而不是"新增"。** 每阶段必须先有回归测试再切换,旧路径留开关可回滚。

**阶段 0(本周 · 止血已做,补登记):** 把今天所有 detached/心跳/董事会/飞书补丁的"是一次性补丁还是触及共性根因"逐条登记进新建的「架构债清单」(见第六节)。不改代码,只盘点。— *CEO 已在本报告完成初稿。*

**阶段 1(P0 · 进程与锁统一):**
1. 把 `resource-locks.js` 的 lease 模型(lease + ownerPid + starttime + 心跳 + 中央 sweep)提为**唯一锁标准**,迁移 engine-slot、queue-claim、worker-pid。
2. 建 `process-supervisor`:单一权威,持有"期望 worker/engine 集合 + 实际探测",持续 reconcile;收编 `server.js`/`ceo-worker.js`/`watchdog-daemon.js` 三处自愈为一处。
3. 引擎/worker 必须主动上报"我活着 + 我在第几步",supervisor 不再靠 `ps` 字符串猜。
- 验收:slot 不再泄漏(崩溃即被 reaper 回收)、活 engine 在重启/慢心跳下不被误杀、worker 掉了 5s 内被拉回、队列卡 running 自动恢复。

**阶段 2(P1 · 识别收敛 + 边界分层):**
1. 单一 `recognition` 模块:projectId/intent/Starlaid 守卫/董事会触发全走它,输出带置信度与作用域;默认安全侧 + 留痕。删散落正则。
2. `server.js` 抽出 `queue`/`engine`/`taskboard` domain 模块,退回薄传输层;spawn 逻辑只剩一份。
- 验收:常规 bug 修复不再触发董事会;Starlaid 误判清零且真实 Starlaid 仍阻断;node-bin 类修复只需改一处。

**阶段 3(P2 · 出口抑制反向治理):**
逐条审计现有"去重/节流/跳过/开关关停",每条回答"上游能不能不产生"。能从入口消除的,把出口抑制降级为纯兜底。
- 验收:飞书噪声来自"真实需要告知"的事件,而非"误触发后被压住"的事件。

**衡量"在进化而非堆补丁"的指标(每周 CEO 看一眼):**
- 复发率:同一 fingerprint/同一根因再次提单的次数(目标↓);
- 补丁多点同步数:一个修复要改几处文件(目标→1);
- 出口抑制条数(去重/跳过/关开关)净增量(目标≤0,即只减不增);
- 架构债清单"已收敛/总数"比例(目标↑)。

## 六、防止补丁继续堆叠的三条护栏(机制建议)

老板要的"进化而非补丁",本质是个**流程护栏**问题,给三条:

1. **每个 repair/补丁完成必须自答一题并登记:** "这是一次性补丁,还是触及共性根因?" 若触及共性,写进 `board/架构债清单.md`(新建),挂到 A/B/C/D 四个弱点之一。维修员现有的"问题模式→根因→预防"沉淀机制再加一层"架构债"维度即可,不必新建系统。

2. **架构不变量清单(invariants),新代码对照评审:**
   - 任何持有资源的实体,必须有 `lease + ownerPid + starttime + 心跳`(不准再裸 PID + 长 TTL);
   - 任何分类/触发决策,必须有 `置信度 + 作用域校验 + 安全默认 + 留痕`(不准再裸正则直接动作);
   - 任何核心机制(spawn/lock/识别/队列状态机)只能有**一份实现**,禁止 copy-paste。

3. **单一事实源原则落到代码:** spawn / lock / recognition 各只允许一个模块对外导出;评审时 grep 到第二处实现即打回。

## 七、给老板的话(趋势判断)

- **好消息:** 团队已经"知道对的模式"(`resource-locks.js` 就是范本),且每天能跑通大量真实任务。这不是能力问题,是"正确模式没铺开 + 缺收敛护栏"。
- **风险:** 若继续逐个打补丁(尤其在根因 C 识别守卫上),复发-再补的循环会让 `server.js`/`ceo-worker.js` 越来越像"混乱机械",最终某个补丁会推翻另一个。今天已经有苗头(董事会先收紧后关停、Starlaid 修两次)。
- **建议拍板:** 批准"阶段 1(P0 进程与锁统一)+ 三条护栏"先行。这是投入产出比最高的一步——它一次性消掉 5 类反复发作的稳定性问题,且不需要停机。识别收敛(P1)随后跟上。
- **不建议:** 停机大重构。

> 本报告为复盘性质,未改动任何代码或配置。下一步若批准,由 CEO 拆成项目主管可执行 brief(阶段 1 + 护栏),交项目主管做项目内规划与实现,维修员负责运行态切换与回滚。
