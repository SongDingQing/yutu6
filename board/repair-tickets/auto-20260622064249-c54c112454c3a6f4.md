根因:
- 失败链路是控制台队列自愈/锁清扫误杀,不是业务任务本身失败。事件 2026-06-22T06:42:44Z 启动 supervisor-控制台/3a4ff93e 的 engine pid=83361,但 2026-06-22T06:42:49Z 恢复逻辑按 running 记录/锁记录推导出 enginePid=66070 并判定不存在,随后把 running 文件重新入队;原 engine 轮询发现 running 文件消失后触发 queue-running-missing,被 SIGTERM 终止并释放 slot。
- 代码根因在 projects/控制台/ceo-worker.js 的 enginePidForRunning(): running 记录尚未写入 enginePid 的短竞态窗口里,它从锁文件回退读取 lease.pid;而 slot/runner 锁初始 pid/ownerPid 是 ceo-worker 进程 pid,不是 engine-runner pid。该 worker pid 被当成 engine pid 后,恢复逻辑误判 engine 缺失并释放锁。
- 反复性: engine-events.jsonl 中同类 queue.recovered reason=running 项记录的 enginePid ... 已不存在 共查到 5 次; 本次 queue-running-missing terminate 1 次,属于系统性竞态/架构一致性问题。

处理:
- 最小修改 projects/控制台/ceo-worker.js: enginePidForRunning() 改用 Runtime.enginePidFromRecord(entry/lease),只接受明确 enginePid/engine_pid 或命令确认为 engine-runner.js 的 legacy pid,不再把 worker pid 当 engine pid。
- 同文件 sweepStaleRunning() 增加保护: fresh running 项若尚无 enginePid 但 heartbeat 未过期,先 emit queue.running.keepalive reason=waiting for engine pid 并继续等待,不立即 requeue/释放锁。
- 增加 tests/stale-running-heartbeat.test.js 回归: freshNoEnginePid 模拟刚 claim、slot 只有 worker pid、尚未附着 enginePid 的竞态,断言仍保持 running、不复制到 queued、锁不被释放。
