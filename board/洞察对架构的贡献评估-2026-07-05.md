# 洞察员对玉兔6架构的贡献评估

_评估日期:2026-07-05 · 评估人:秘书/维修部主管_

## 一句话结论

洞察员的价值主要是**持续的外部对照系（防闭门造车 + 灵感流）**,广度很大;但"架构提升"不是洞察直接完成的,而是走**「洞察发现 → 架构审视收敛 → 实现落地」**这条漏斗。目前漏斗末端(真正改到架构代码)的转化是**少数几项(有实锤)**,多数洞察停在"设计层借鉴/待排期"。诚实说:洞察带来的是**方向校准和可执行落点清单**,不是一次性大幅重写。

## 二、发现层(广度)——洞察员产出了什么

| 指标 | 数据 |
|---|---|
| 自动洞察批次 | 36+ 批(2026-06-24 ~ 06-30,每 4 小时轮换主题) |
| 扫描开源案例 | ~100+ 个(multi-agent 编排 / 队列引擎 / LLM 网关 / GUI grounding / 像素素材 / 前端设计 等 7 大主题轮转) |
| 沉淀「已借鉴库」清单 | 40+ 个(`board/insights/borrowed-libs.md`,每个标注借鉴点 + 迁移边界 + 是否 watch 上游) |
| 去重与冷热分区 | `seen-repos.json` 防重复推荐 + `references/` 冷区归档,渐进披露只读热区 |

**特点**:每个案例都核了许可证、标了"只借设计/机制、不引运行时"的迁移边界,几乎不制造待办堆积——这是它克制、可信的地方。

## 三、转化层——洞察如何变成架构落点

2026-07-03 架构审视报告(`board/架构审视报告-2026-07-03.md` 第四章)把散落的洞察**收敛成 7 个具体落点**,每个都写明"借鉴点 → 玉兔6 落点 → 排期"。这是洞察产生架构价值的**关键一跳**——把"别人怎么做"翻译成"我们改哪行代码":

| 洞察案例 | 借鉴点 | 玉兔6 落点 |
|---|---|---|
| cua(#8) | 特权执行要有机器可核边界 | 特权写路径白名单 + done-gate 校验 |
| handoff-protocol(#7) | 交接=文件夹+指针+机器骨架 | 交接机制(指针化 prompt 层) |
| gpt-load(#5) | 失败黑名单+冷却自愈;映射单源 | runner 字段删硬编码 + quota retry_after |
| opik/ThinkWatch(#2/#4) | 质量分回写路由;模型+工具统一账本 | runner.quality / runner.call 事件 |
| Routerly(#1) | 路由带健康/用量信号 | scoreCandidates 纯函数评分 |
| RouterEval(#3) | 路由变更需离线回归口径 | routing-replay 只读对照脚本 |
| OpenInference(#6) | 统一 trace 字段贯穿 task→node→call | 事件带 trace_id/span_id |

## 四、落地层(实锤)——真正进了代码的

| 落点 | 状态 | 代码位置 | 提升 |
|---|---|---|---|
| 交接机制(#7 handoff) | ✅ 已落地(shadow 观察期) | `shared/engine/handoff.js` | prompt 层指针化,交接信封体积大幅下降,减 token |
| 特权写边界(#8 cua) | ✅ 已落地(shadow 观察期) | `shared/engine/write-audit.js` | 特权角色改动落在允许区外即拦,堵最大单点风险 |
| 统一 trace 字段(#6) | 🟡 部分落地 | `projects/控制台/llm-usage.js` | LLM 用量带 trace 关联,尚未贯穿 task→node→call 全链 |

## 五、待采纳(设计已备、代码未动)

以下洞察落点**架构审视已给出可执行设计,但尚未进代码**——这正是"洞察 ≠ 自动提升"的证据:

- **路由健康评分 scoreCandidates(#1 Routerly)**:代码中无 `scoreCandidates`,当前路由仍靠 failover 硬序,未兑现"predictive"。
- **路由离线回归 routing-replay(#3)**:无 `tools/routing-replay.js`,改路由 yaml 仍缺前后对照口径。
- **runner.call / runner.quality 统一账本(#2/#4)**:事件流里无这两条 emit,质量分回写路由、模型+工具统一账本尚未建地基。

## 六、客观评价:洞察对架构的提升「有多少」

- **不是**:一次性大幅重写架构 / 洞察直接改代码。
- **是**:① 持续外部对照,避免玉兔6 在编排/队列/网关/可观测这些成熟领域重复造轮子或走弯路;② 一次架构审视把洞察**收敛成 7 个带排期的可执行落点**,把"灵感"变"工单";③ 其中 **~2.5/7 已进代码**(交接、特权写边界、部分 trace),其余待排期。
- **瓶颈在漏斗末端**:发现多、收敛清晰,但**代码落地转化率约 1/3**。要放大洞察的架构价值,关键不是让洞察员扫更多案例,而是把第四章那 7 个落点的**剩余 4.5 项按排期推完**(尤其路由评分 + 回归口径 + 统一账本,它们互相咬合)。

## 七、建议

1. 给洞察落点建一张"采纳看板":7 个落点各标 已落地/进行中/待排期,每张设"两周内采纳或归档"硬时限(架构审视第四章横切项已定此规则,落一下)。
2. 优先推 **runner.call/quality 事件**(一次改动补 #2/#4/#6 三案例地基,纯增量事件、低风险),它是路由评分(#1)和回归口径(#3)的数据前提。
3. 洞察员保持现节奏即可(广度已够),重心从"发现"移到"落地漏斗"。
