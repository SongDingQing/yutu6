# 玉兔6 任务可靠执行 · 研究(对标别人的多智能体)

> 老板点名研究:怎么让任务尽可能成功执行、别让 CEO 老是分配错误;多参考别人的多智能体设计,看有什么不同。2026-06-20。

## 老板的痛点
任务失败太多(归属判死 + node_failed),CEO 分配/归属判断老出错。

## 一、我们设计的脆点(根因)
CEO 用 **LLM"判断"项目归属**(inferred classification)+ **判不准就 fail-dead(杀任务)**。这是最脆的组合:
- ① LLM 判断是**非确定**的、会错;
- ② 不确定就**杀任务**(硬失败)而不是退而求其次。
→ 结果:同一句话有时过、有时被判死;失败还连锁触发维修。

## 二、别人怎么做(研究结论)

### 编排模式(2026 主流 5 种)
fan-out / pipeline / debate / **supervisor** / swarm。我们用的是 **supervisor(orchestrator-worker)**:CEO→主管→员工——这个选型**没问题、很主流**(LangGraph 叫 supervisor、crewAI 叫 hierarchical、AutoGen 叫 group-chat-manager,都是它)。

### 路由方式——这才是关键差异
- **LangGraph**(2026 生产部署最多):用 **DAG 图**,路由是**确定性的"边/函数"**——条件边是代码判断,不是自由 LLM 发挥。
- **OpenAI Swarm**:用**显式 handoff**——一个 agent **显式地"交给"某个命名 agent**(返回 Agent 对象),不是"判断+猜该给谁"。
- **规则路由 vs LLM 路由**(研究对比):规则确定、亚毫秒、可调试但僵;LLM 灵活懂语义但**会误分类 + 有延迟**。

### 别人怎么避免路由错误/任务失败
1. **确定性优先**:显式声明 / 规则先行,**LLM 只兜底**;
2. **fail-soft + 重试**:路由不确定→给默认/转人工,**绝不硬失败**;transient 失败**自动重试**;cheap-first、出错再升级(model cascade)。

## 三、关键差异:我们 vs 别人(一句话)
> **别人把"路由"做成确定性的(图的边 / 显式交接 / 规则),LLM 只负责"干活"、不负责"判断该路由到哪";而我们让 LLM 既判断路由、又干活,判断错就死。**

这就是我们失败率高、CEO"分配错误"的**根本**——不是 CEO 笨,是我们把一个**该用确定性规则的事(路由)**交给了**非确定的 LLM 判断**,还配了**硬失败**。

## 四、改进建议(让任务尽可能成功)
1. **归属路由"确定性优先"**:**显式 `projectId` 永远最优先**(直接采用、绕过 LLM 判断)→ 其次关键词规则 → **最后才** LLM 兜底。(现在的 bug 正是显式 projectId 被 Starlaid 检测推翻了。)
2. **fail-soft,绝不判死**:路由真判不准 → 落到默认"控制台"项目 或 needs-human 待你指认,**绝不进 `failed/`**。(= 已派的 `a4d6f1e9`)
3. **node_failed 自动重试**:transient 失败(meowa 超时、CLI 抖动)**自动重试 N 次**再放弃,别一次失败就开维修。
4. **CEO 结构化规划 + 校验重试**:CEO 出**结构化计划**,校验不通过 → **重试规划**,而不是杀任务。
5. **显式交接契约**:借 Swarm 的 handoff 思路——每个交接**显式声明"交给谁/期望返回什么/失败怎么办"**,少靠自由判断。

## 五、方向(对标后的取舍)
- **"我们设计简单"不是坏事**——简单 = 稳、好维护。supervisor 模式选对了,**不要为了花哨引入 LangGraph/重框架**(违背零依赖)。
- **要改的就一环**:把**路由**从"LLM 判断 + 硬失败"升级成"**确定性优先 + fail-soft + 自动重试**"。这是把别人验证过的可靠性模式,用我们的**零依赖方式**实现。
- 改完这一环,任务成功率会显著上来,CEO 也不会再"分配错误"——因为路由不再靠它瞎猜。
