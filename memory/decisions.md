# 决策记忆 · 定了什么 + 为什么

> 精炼版长期决策(运营级待决/流水账在 `board/decisions.md`)。更新于 2026-06-20。

- **老板入口默认秘书→CEO**(2026-07-01):非维修任务先由 secretary 收口、读 board 背景、补 task 信封,再转 CEO 决策路线和分派;维修/救火/重启/权限/孤儿进程/内存止血等走 repair-lead/repair;手动直派仅作显式调试覆盖。
- **自动优化必须沉淀案例**(2026-07-01):UI/架构自动优化不能只产出一次性改动,每轮/收尾要把可复用案例写入 `board/learning-cases/`,并让秘书、CEO、董事会读取这些案例来提升后续架构判断。
- **网关边界**:codex+gpt、claude code+opus 保留 **CLI 订阅**,不进 new-api;其余模型(minimax/deepseek/智谱…)都走 new-api 网关。原因:订阅按量更划算 + CLI 能力强。
- **codex 沙箱**:codex runner 固定 `--sandbox danger-full-access`。原因:workspace-write 在常驻 worker 环境反复 Operation not permitted。
- **维修员 = codex 特权**:`codex-privileged` 担任,失败自动接单。原因:绕 sandbox 限制、能救火、老板不想手动转交。
- **洞察员 = 分析非待办**:产出借鉴分析(insights.md),不堆待办卡。原因:老板要"分析值得借鉴什么",不要点一堆卡。
- **办公室默认视图**:等距像素办公室为默认,工位/链路图可切。
- **记忆系统**:完整版(memory 分层 + 记忆官 + 主动提炼 + 检索注入 + 定期整合)。决定于 2026-06-20。
- **会话交接**:旧会话满了写交接说明(board/交接-*.md),新会话继承秘书角色。
- **GLM-5.2 定位**(2026-06-20):去掉「智谱设计师」这个窄角色;GLM-5.2(走 new-api,老板充了会员、token 多、编程强)**保留为通用模型 runner**,后续承接编程/通用任务,可作 worker_code 的补充以省 codex/claude 额度。
- **董事会评议机制**(2026-06-21):重要架构任务必须先由 DeepSeek(new-api)、GLM-5.2(zhipu-glm)、GPT-5.5(codex)、Opus-4.8(claude) 四董事挑刺评议,最多 3 轮;默认安全直接执行,唯一阻断是第 3 轮后 Opus 仍判误判风险,此时生成需主人点击的决策卡。
- **重启韧性双向门**(2026-06-23 监管复盘定):watchdog/console 重启既要防"杀活引擎"(已有),也要防"重启不上"——起新进程前必须确认旧进程端口已释放(轮询 free,超时 SIGKILL 升级)再 `bind`;"重启失败"(EADDRINUSE)与"重启太频繁"(节流)分两条处理路径,前者连续 N 次失败升级告警主人,不无脑 cooldown 空转。原因:2026-06-23 端口 41218 占用致重启自我级联、控制台夜间宕机 ~3 小时。
- **失败必可观测 + 工单闭环 SLA**(2026-06-23 监管复盘定):engine-runner 任何非 0 退出前必落带 `exit_code`/stderr 尾部的 `node.fail` 与 run 目录证据,禁止"退出码 1;未写明原因"黑盒;`dedupe-cooldown` 静默压制重复工单达阈值要升级人工/根因(压制≠解决);每日复盘统计 `status: todo` 悬挂工单与最老悬挂时长,超时升级主人;`infra_restart` 类只重入队不开重复工单。原因:2026-06-23 `worker_code/oldjump` exit-1 零产物、6 张工单全悬挂、25 次冷却压制掩盖严重性。
