## 故障模式 / 术语(监管复盘)
- **重启即清扫,清扫即杀活**(2026-06-22):误区命名。把 console/watchdog 重启或 `cleanup-runtime` 当成无副作用操作,实际对在途 engine 是一次批量 SIGTERM kill,衍生「无进展判死」误判与维修工单刷屏。
- **infra_restart(基础设施重启失败类)**(2026-06-22):拟新增的失败分类。落在重启窗口内、reason 为 SIGTERM/运行清扫的 node 失败应归此类,只重入队、不计 maxRetry、不开维修工单,区别于真正的 `node_failed` 业务失败。
- **节点无进展判死(480s)**:engine 在 480s 内无 stdout 进展即被超时回收。四类已知误判:① 心跳新鲜仍判死——待与 `engine_heartbeat_at` 续约联合判定;② **`node.output` 进展只写 EventLog、不刷 running `progress_at`**,看门狗看不到活跃输出而误杀(2026-06-22 已在 `cli-runner.js` 同步 `Queue.touchProgress` 修复);③ **写 running 的状态边界没清运行态派生字段**(`node_event_at/progress_event/progress_node/progress_task/enginePid/cancel_requested` 等),下一轮继承旧时间戳刚启动就被快速再杀——已知三条边界都需守卫:requeue/resume 恢复路径(2026-06-22 已修)+ `claim()` queued→running 认领路径(2026-06-22 后续发现并补齐,`tests/queue.test.js` 三路径回归);**并在消费侧加兜底**——watchdog 只信任 `progress_task` 与当前 taskId 匹配的进度,不匹配则退回 `started_at` 重新计时(2026-06-22 已在 `ceo-worker.js` 加,`tests/stale-running-heartbeat.test.js`),清理侧+消费侧双保险;④ **固定 480s 门槛不尊重任务声明的 `nodeTimeoutSec`**,长运行/沉默型任务(如 governance 复盘 `nodeTimeoutSec=1500`)被短门槛误杀(2026-06-22 已在 `ceo-worker.js` 改有效门槛取 `max(全局 no-progress, 任务 nodeTimeoutSec/timeoutSec)`)。目标:收敛成"进展事件 + 心跳 + 输出刷新 + 任务级超时契约"单一数据源 watchdog,并把 running 恢复字段白名单化。
- **队列命名空间污染(审计/缓存当任务)**(2026-06-23):误区命名。把幂等审计、缓存、索引、worker 元数据 JSON 直接写进 `artifacts/queues/<agent>/` 根目录,被按 `queues/<agent>/*.json` 识别任务的文件队列扫描器误吞成伪 queueAgent(如 `_organize-idempotency`)、`queueId=null`、`task=…-undefined` 空目标 review-loop,衍生伪 worker 空跑、资源锁冲突、重复自动提单;done gate 打回只是末端症状。预防:运行队列目录只存任务信封,辅助数据移出 `queues/` 命名空间(或用非 `.json`/显式排除)并加测试。详见 experience.md 同条。
- **沉淀≠落地(规则只进 memory 未落代码即复发)**(2026-06-24):误区命名。把复盘产出的防复发规则写进 memory 就当成"已闭环",实际它只是建议、没有 owner 把它变成代码/测试/检查门。2026-06-23 的 R1/R3/R4 全部仅入 memory,2026-06-24 核查均未落代码,同族故障按量级复发(console-http timeout 风暴 31+101)、部分恶化(all_blocked 595→1961)。是 [[补丁已落盘 ≠ 运行态已生效]] 的上游同族:前者运行态没生效,本条连代码都还没改。预防:每条规则绑 owner 工单 + 次日核查落地,落地率纳入复盘指标。
