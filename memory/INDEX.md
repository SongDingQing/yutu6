# 玉兔6 · 记忆系统 INDEX

> 玉兔6 的长期记忆库。和 `board/`(运营状态:方向/进展/工单)不同,`memory/` 是**提炼后的长期记忆**——面向"回忆和复用",越精炼越好。
> 建立:2026-06-20 · 维护:记忆官(memory_officer)+ 秘书

## 分层(五类记忆)

| 文件 | 记什么 | 谁写 |
|---|---|---|
| `preferences.md` | **老板画像**:语言/风格/红线/成本观/审美/协作习惯 | 秘书 + 记忆官 |
| `decisions.md` | **决策记忆**:定了什么、为什么这么定(精炼版,详账在 board/decisions.md) | 记忆官 |
| `experience.md` | **经验库**:成功模式 + 失败教训(现象→根因→避免) | 记忆官 + 监管 |
| `entities.md` | **实体记忆**:项目/角色/能力/机器/外部案例的稳定事实 | 记忆官 |
| `sessions/` | **会话摘要**:每次会话提炼"这次发生了什么、记住什么" | 记忆官 |

## 记忆规则(防止越长越乱)

1. **提炼不流水账**:只记"以后还用得上"的;一次性细节不进记忆。
2. **可执行 / 可检索**:每条尽量"条件 → 结论/做法",便于检索注入。
3. **去重 + 纠正**:新记忆和旧的冲突时,**更新**而非堆叠;过时的标注或删除。
4. **对事不对人**:经验/教训改流程,不追责。
5. **红线**:密钥/token/身份信息**绝不写入**记忆。

## 工作方式(完整记忆系统)

- **主动提炼**:每个任务完成 / 每次会话结束,记忆官提炼要点写入对应分层。
- **检索注入**:智能体干活前,按相关性把相关记忆注入上下文(升级现在固定的 board 背景包)。
- **定期整合**:定时任务跑记忆官做 consolidate——合并重复、纠正过时、精简索引。

## 与 knowledge/ 的分工

- `memory/` 放提炼后的长期记忆,用于快速注入:维修经验进 `experience.md`,项目 ↔ 技术/方案映射进 `entities.md`。
- `knowledge/` 放原文 RAG 检索库与 ingest/query 管道;维修完成结果不直接写 `knowledge/` 或 `kb.sqlite`。
- 当维修复盘或项目技术事实积累到需要原文检索时,再按主人指令把 wiki/board/corpus 的原文统一整理后由 `knowledge/ingest.py` 抽取。
