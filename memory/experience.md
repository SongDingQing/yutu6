## 失败教训(已踩过的坑)

- **客户端固定信任的证书链在服务器重装时被静默换掉 → 所有客户端网络请求静默失败,且无人察觉近 10 天**(2026-07-04 维修复盘,已修):元宵 App 用 network_security_config 固定信任内置 CA(res/raw/yuanxiao_ca.pem,旧机 5/4 签发);6/25 嫦娥服务器重装时 deploy 脚本因服务器上没有 ca.crt/ca.key 走了"自签兜底"分支,新证书不是该 CA 签的 → App(1.25/1.26)所有 HTTPS 静默失败,推送升级/聊天/任务全断,直到老板报"没收到升级提醒"才发现(服务器日志里手机零请求是关键证据)。是「declared-but-not-honored」在证书层的同族:客户端"声明"了信任锚,部署侧没有消费者保证链条延续。
  - 根因:证书信任是跨端契约(App 内置 CA ↔ 服务器证书),但部署脚本的 CA 签发路径依赖服务器上存在 CA 文件,重装后无人补;客户端 TLS 失败又是静默的(catch 后不上报),没有任何一侧能观测到断链。
  - 做法(已修):① 新签 CA v2 + IP-SAN 服务器证书装上服务器,**ca.crt/ca.key 一并落在 /opt/yuanxiao/certs/**(deploy 脚本检测到 CA 即自动用它续签,链条自愈);② App 内置双信任锚(旧+新 CA 并存,一个 pem 多证书);③ 本地 CA 私钥备份在 ops/config/certs-v2(不入 git)。
  - 预防:凡"客户端固定信任"的证书/密钥/指纹,部署脚本必须把签发材料当作部署必备品校验(缺失即硬失败,不许静默降级自签);客户端 TLS 失败应有可观测出口(至少界面上显示连接态,而非静默);服务器日志"某设备长期零请求"应纳入巡检信号。

- **孤儿清理误杀活引擎**:把"清理孤儿进程治内存"写得太激进,误杀了正在跑的 engine(stale engine lease → SIGTERM)。
  - 根因:活引擎没有续约机制,被当成 stale。
  - 做法:引擎加**心跳续约**,只杀真正失联的;清理策略宁可保守。秘书下指令别用"激进清理"这类措辞。

- **codex worker 启动 Operation not permitted**:`worker_code` 用 `--sandbox workspace-write` 在常驻 worker/launchd 环境里反复初始化失败(PATH aliases / app-server),阻塞所有 codex 实现任务 9 次。
  - 根因:workspace-write sandbox 与 launchd/TCC/子进程环境组合不稳定。
  - 做法:codex runner 固定用 `--sandbox danger-full-access`;系统性启动问题别靠重启,要改调用模式。

- **跨队列优先级丢失**:priority 0 的任务路由到下游队列后变回 50。
  - 根因:路由时没传递优先级。
  - 做法:在**正确的下游队列**里重新插队(jump);或修路由让其携带优先级。

- **任务板"待办/排队"语义混淆**:已启用、排队中(queued)的任务被显示在"待办/备选"区,老板误以为"点了没反应"。
  - 根因:待办区数据源混入了 queued;queued 标签写成"待办"。
  - 做法:queued = "排队中",归进行中/单列;待办区只放可启用候选。

- **沙箱删不了队列文件 → 取消/合并任务造成重复**(2026-06-20):秘书在沙箱对 `queues/` 能创建 + **rename(=能改优先级 setPriority / 重排 / steer)**,但**不能 unlink(删除)**(EPERM)。⚠️**2026-06-22 实测纠正**:我之前误以为 rename 也 EPERM、好几次跟老板说「改不了优先级、要 CEO 带外」,是过度保守的误判——**改优先级 / 重排 / 改名我自己能做**;只有删除 / 取消 / 合并删旧任务(unlink)才真做不了。
  - 根因:沙箱挂载「可创建、不可删除」限制。
  - 做法:取消/合并**已入队**任务需**本机执行**(已给秘书加本机通道);合并前先确认能取消旧任务,否则别派合并;webUI 的「取消排队」按钮走本机进程、可用。

- **硬性回归测试别用「僵硬结果断言」,要验证「真做的逻辑链」**(2026-06-22 老板纠正):我设计硬化测试时举例「合并任务 → 断言任务数必须减少」,老板指出硬伤——任务本就无可合并项、或队列只剩一个任务时,「必须减少」会误判甚至卡死。
  - 根因:把「证明真的在做」错当成「对结果数量的硬断言」;结果可能合理地不变(无可压缩 / 只剩一个)。
  - 做法:回归测试要求执行者**提交逻辑链 + 最简可验证证据**证明「真的在做、不是假的」(例:合并任务给出「逐一检查了哪些、哪些可并/不可并及理由」,无可并项就证明查过且无相似项);假完成拿不出真实逻辑链就过不了。**验证的是过程真实性,不是僵硬的结果数。**

- **项目归属守卫误杀说明文本或代码标识符**(2026-06-20 维修复盘):控制台任务中只是复述 "Starlaid 排除/红线" 或引用 `buildSecretaryEnvelope()` 这类函数名,却被判成主动操作 Starlaid,导致 CEO 停止派单。
  - 根因:`project-guard.js` 曾把诊断全文、红线说明和代码标识符一起扫,英文 action 词缺少单词边界,`build`/`test`/`run` 容易被误命中。
  - 做法:项目归属判断优先结构化 `projectId`;诊断/红线/非主动操作上下文降权或排除;英文动作词必须用单词边界,并给真实主动 Starlaid 操作保留阻断回归测试。

- **"清理卡死任务"只移任务文件、没杀进程也没释放 slot 锁 → slot 锁泄漏、死锁**(2026-06-21):秘书让老板 `mv` 卡死任务文件到 failed,以为清掉了,但 engine slot 锁(`engine-slots/slot-0.json`)和占用进程都没动,slot 永久泄漏、谁都拿不到 → 全队列死锁。
  - 根因:把"清理卡死任务"等同于"移任务文件",忽略了**任务文件、slot 锁、占用进程是三个独立状态**,要一起处理。
  - 做法:清卡死任务必须三件一起——杀进程 + 释放/清 slot 锁 + 移任务文件,缺一不可;给老板任何清理命令前,先核实**全部副作用**,别只动一处。

- **detached engine 进程卡死后,重启 console 杀不掉它**(2026-06-21):为防"重启自杀"把 engine 改成 detached(独立于 console 进程树),结果它卡死后,重启 console 杀不掉这个独立进程,它继续占 slot、续约心跳逃过判死;秘书反复让老板重启都没用。
  - 根因:没把"历史改动:engine 改 detached"和"重启杀不掉独立进程"联系起来,反复用重启这个无效手段。
  - 做法:遇到"重启没生效",先查是不是有**独立于 console 的 detached 进程**在占用/续约;detached 进程要有独立清理通道(看门狗/特权清理),不能只靠重启 console。

- **反复治标(重启)、挖根本太晚:worker 跑完就停 + 队列不消费**(2026-06-21):任务每跑完一个 worker 就停、不拉下一个,且 CEO 队列从不被消费;秘书一直当成"修复要重启才生效"反复让老板手动重启,把人惹烦,根本(worker 非常驻持续循环 / CEO worker 没消费队列)挖得太晚。
  - 根因:被"修了要重启生效"的表象带走,同一现象(跑完就停)反复出现却没停下来问"为什么每次都停"。
  - 做法:**同一现象反复出现 = 强信号,立刻停治标、挖根本**(进程生命周期、worker 主循环是否常驻、队列是否真被消费),别用同样的治标手段(重启)循环消耗老板;系统级反复问题优先让 codex/维修员直接改根因,而不是秘书反复手动绕。

- **重启/清扫批量杀活引擎,衍生工单刷屏 + 无进展误判(2026-06-22 监管复盘)**:24h 内一波 SIGTERM 集群(13:42–13:45Z,6 个 task.failed)+ 多次「480s 无节点进展判死」+ enginePid 失效重入队,全部紧跟 watchdog 5 次重启(console-not-running / hotfix reload / worker-heartbeat-stale),并自动开出 4 张维修工单(全部悬挂 todo 未结案)。
  - 根因:把"重启控制台 / 清运行时(cleanup-runtime)"当成无副作用操作,实则对在途引擎是一次批量 kill;失败分类一律走 node_failed → 耗尽重试 → 开工单;"无进展判死"未与心跳续约联合判定,心跳还新鲜也照杀。是「孤儿清理误杀活引擎」「detached 卡死重启杀不掉」「反复治标重启」的同族新化身。
  - 误区命名:**`重启即清扫,清扫即杀活`**。
  - 做法(可验证):① 重启前先 drain/quiesce 在途引擎,只 sweep 心跳确已失联的,心跳新鲜不清(烟测:重启时心跳新鲜的 engine 不收 SIGTERM);② 落在重启窗口内的 SIGTERM 标 `infra_restart`,重新入队但不递增 maxRetry、不开维修工单(验收:repair.ticket.created=0);③ 480s 无进展须与 `engine_heartbeat_at` 联合判定,心跳新鲜则延长宽限,只有心跳也 stale 才判死;④ 维修工单要有终态:infra_restart 类源任务重试成功后自动 closed,监管/质量每日 sweep 超 24h 的 todo 工单归类;⑤ heartbeat-stale 触发的重启是治标,优先让 codex 修 worker 续约根因而非反复重启。

- **直派文本启发式把"红线排除"误当"角色排除" → 派错队列 + 无节点进展超时**(2026-06-22 维修复盘):任务文本同时含「目标角色(前端设计师实现)+ 项目红线排除(Starlaid 排除)+ 其他部门/角色名(人力资源)」时,`engine-runner.js` 的 `directQueueForGoal` 误判后被直派到 `hr_manager`,而该角色明确不做代码/UI,任务空跑、3 次重试只有 node.start、最终被 480s 无节点进展判死 SIGTERM。
  - 根因:`hasNegatedDirectRoute` 的角色否定正则缺**语义域隔离**——after 正则把"角色名后 16 字内出现『排除』"一律视为该角色被排除,于是把相邻的 `Starlaid 排除` 红线语境错算成 `前端设计师排除`,frontend_designer 分支被否掉后落到 HR 关键词。是「项目归属守卫误杀说明文本或代码标识符」的同族(文本启发式吃进红线/排除语境就误判)。
  - 做法:进入角色否定匹配前,先**剥离 Starlaid/星桥 + 排除/不涉及/无关/跳过/不处理 等项目红线语境**,再做角色否定;为「精确失败文本→目标角色」和「真实角色排除→null」各加正反向回归(`tests/project-routing.test.js`)。预防/自动化:直派判断改为**结构化 `assignee`/`directAgent` 优先、文本启发式仅兜底**;把红线/项目语境剥离抽成**独立工具函数**复用到其他 direct agent 选择,避免同类误伤。

- **队列清扫器把 owner-worker pid 当 engine pid → 新任务被误判 engine 缺失而 requeue/杀 engine**(2026-06-22 维修复盘):长驻 worker 刚 claim 任务、enginePid 尚未写入 running/slot 的短竞态窗口里,清扫/恢复逻辑从锁文件回退读 `lease.pid`(实为 ceo-worker 进程 pid,非 engine-runner pid),把它当 engine pid 判定"engine 不存在",重新入队 running 文件;原 engine 轮询发现 running 消失触发 `queue-running-missing` 被 SIGTERM。engine-events 同类 `queue.recovered reason=running enginePid…不存在` 已 5 次,属系统性竞态,是「孤儿清理误杀活引擎」「重启即清扫,清扫即杀活」的同族(运行态/锁态 pid 语义不一致)。
